VPNs & Proxies

Encrypted tunnels and anonymous relays are the backbone of safe remote access and attacker anonymity. Today you build both.

THEORY

VPN (Virtual Private Network): A VPN creates an encrypted tunnel between a client and a server. All traffic from the client is routed through the tunnel, encrypted by the VPN protocol, and decrypted at the VPN server before being sent to its final destination. To the outside observer, the traffic appears to come from the VPN server, not the client. VPNs use protocols like OpenVPN (SSL/TLS-based, UDP/TCP port 1194), WireGuard (modern, kernel-level, UDP), IPsec (IKEv1/IKEv2, UDP ports 500/4500), and L2TP (often combined with IPsec).

Proxy types:

# Forward Proxy — client configures proxy address; proxy fetches on behalf # Reverse Proxy — sits in front of servers, handles TLS, load balancing, caching # SOCKS5 — generic proxy that tunnels any TCP/UDP traffic (no application awareness) # HTTP(S) Proxy — understands HTTP protocol, can cache, filter, modify requests # Transparent Proxy — intercepts traffic without client configuration # SSH SOCKS5 proxy — one command, instant encrypted tunnel: ssh -D 1080 -C -N user@your-server.com # -D 1080 = SOCKS5 proxy on localhost:1080 # -C = compression, -N = no remote command (just tunnel)

Forward vs Reverse Proxy:

# Forward proxy (client-side): Client → Forward Proxy → Internet → Target Server # The target server sees the proxy's IP, not the client's. # Reverse proxy (server-side): Client → Internet → Reverse Proxy → Web Server # The client never directly contacts the web server. # Nginx as reverse proxy example: server { listen 443 ssl; location / { proxy_pass http://127.0.0.1:3000; } }
Important distinction: A VPN protects your traffic between you and the VPN server — everything is encrypted. A proxy only forwards specific application traffic (or at SOCKS5, any TCP/UDP) but does NOT automatically encrypt it. A proxy without encryption is just an address mask, not a security tool. For operational security, use a VPN for full traffic protection or SSH tunnel for encrypted proxying.
// attacker vs defender mindset
⚔ Attacker
  • Use SSH dynamic port forwarding (SOCKS5) or a VPN to route your attack traffic through intermediate hosts — makes attribution harder and bypasses egress filtering.
  • Set up a reverse proxy (nginx) to hide your C2 server's real IP — Cloudflare or a VPS acts as the visible frontend; your real server is obfuscated.
  • Use Tor (SOCKS5 on 127.0.0.1:9050) for anonymous scanning and recon — proxychains nmap -sT target.com routes through Tor.
  • Chain proxies — traffic flows through multiple hops (Tor → VPN → VPS) to create layers of separation between you and the target.
🛡 Defender
  • Require VPN access for all administrative access to internal systems. No direct SSH/RDP exposure — enforce VPN termination before internal access.
  • Use a reverse proxy (Nginx, HAProxy, Cloudflare) in front of web applications to handle TLS termination, WAF filtering, and hide backend server IPs.
  • Block outbound proxying — use firewall rules to prevent connections to known VPN/proxy ports (1194, 1080, 1723). Monitor for unauthorized tunnel creation.
  • Log and alert on Tor exit node connections — regularly updated Tor exit node lists enable detection of traffic from anonymous sources.
// real named incident
2019 — NORDVPN DATACENTER BREACH — PRIVATE KEY EXPOSURE
Compromised datacenter exposed NordVPN's private TLS keys
In 2018, one of NordVPN's rented servers in Finland was compromised when an attacker gained access to the datacenter's management infrastructure. The attacker obtained the private key for NordVPN's TLS certificate but could not decrypt any VPN traffic because NordVPN correctly implemented perfect forward secrecy (PFS) using ephemeral Diffie-Hellman key exchanges. PFS ensures that even if the long-term private key is stolen, past sessions cannot be decrypted. The incident highlighted a critical lesson: VPN providers must not only secure their tunnels (protocol) but also their infrastructure (server hardening, access control, key management). NordVPN responded by migrating all servers to RAM-only operation with no persistent storage.
HANDS-ON LAB

Goal: set up an SSH SOCKS5 proxy, configure an OpenVPN server, and trace traffic routing differences.

  1. SSH into a remote VPS (or local VM) and create a SOCKS5 tunnel: ssh -D 1080 -C -N user@remote-host.
  2. Configure your browser to use SOCKS5 proxy at 127.0.0.1:1080. Verify your public IP changes by visiting whatismyip.com.
  3. Install OpenVPN: sudo apt install openvpn easy-rsa -y. Generate server certificates and client config using easy-rsa.
  4. Start the OpenVPN server: sudo systemctl start openvpn@server. Connect from a client using the .ovpn profile.
  5. Trace the traffic path with traceroute: compare direct connection, SOCKS5 proxied, and VPN-routed routes.
  6. Set up a simple reverse proxy with Nginx: proxy_pass http://localhost:3000 and test accessing the backend through it.
Deliverable: A working SOCKS5 proxy and a working OpenVPN connection, with traceroute output showing the different routing paths for each configuration.

INTERVIEW READY
Q: "What's the difference between a VPN and a proxy?"
A: "A VPN creates an encrypted tunnel for all your device's traffic at the operating system level — every application, every protocol. A proxy only handles traffic from a single application configured to use it, and many proxies don't encrypt traffic. VPNs are for full privacy and security. Proxies are for specific traffic routing needs like web scraping, geo-spoofing, or bypassing content filters."
Q: "Explain forward vs reverse proxy."
A: "A forward proxy sits between clients and the internet — clients configure it explicitly, and it fetches resources on their behalf. Used for anonymity, content filtering, and access control. A reverse proxy sits between the internet and servers — clients don't know it exists. It handles TLS termination, load balancing, caching, and WAF filtering. Nginx and Cloudflare are commonly used as reverse proxies."
Q: "What is perfect forward secrecy and why does it matter for VPNs?"
A: "Perfect forward secrecy (PFS) ensures that each VPN session uses a unique, ephemeral session key derived through Diffie-Hellman key exchange. If a VPN provider's long-term private key is later compromised, past sessions cannot be decrypted because the session keys were temporary and not stored. This was critical in the NordVPN breach — the attacker got the private key but couldn't decrypt any recorded traffic because NordVPN used PFS."
