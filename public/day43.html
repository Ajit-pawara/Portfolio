Active Directory Enumeration with BloodHound

BloodHound maps Active Directory attack paths that attackers will exploit.

THEORY

BloodHound architecture: Collector (SharpHound.exe for Windows, BloodHound.py for Linux) enumerates AD objects and relationships. Graph database (Neo4j) stores nodes (users, groups, computers, GPOs) and edges (relationships like MemberOf, AdminTo, CanRDP). Visualization in BloodHound GUI shows attack paths.

# Collect data (run on domain-joined Windows) SharpHound.exe -c All --zipfilename bloodhound # Or from Linux via BloodHound.py bloodhound-python -d domain.com -u user -p pass -ns 10.10.10.1 -c all

Key pre-built queries:

  • Find Domain Admins (DA) -- who has rights
  • Shortest paths to DA -- attack paths matter
  • Kerberosable users -- Kerberoastable accounts
  • AS-REP roastable users -- no pre-auth required
  • All sessions from high-value targets -- pass-the-hash gold
Key insight: BloodHound doesn't exploit -- it maps. Every edge represents a possible attack. Removing edges (ACL hardening, group cleanup, GPO changes) removes attack paths. Measure by tracking 'Shortest Paths to DA' before and after.
// attacker vs defender mindset
ATTACKER
  • Start with built-in queries (pre-built on left panel in BloodHound). 'Shortest Paths from Owned to Domain Admins' reveals lateral movement routes.
  • Analyze edges: MemberOf (direct group), AdminTo (local admin), HasSession (admin logged into computer), CanRDP (RDP access), GenericAll (full ACL control -- can reset password). GenericAll on a user = can overwrite their password without knowing current one.
  • Kerberoastable users have ServicePrincipalName set. Attackers request TGS, crack offline. AS-REP roastable users have 'Do not require Kerberos preauth' flag, attacker requests TGT, cracks offline.
  • GPO abuse: GPOs with 'Enforce' or 'Link' on OU = manage computers/users. GPO to add local admin, deploy startup script, or modify registry.
DEFENDER
  • BloodHound is your #1 AD security tool. Run monthly, compare path counts.
  • Harden: remove Domain Admin from non-DA machines, restrict local admin, tiered administration model (Tier 0/1/2).
  • Honeypot: add fake 'high-value' user with weird name that requests TGTs on access. Monitoring team alerted on any interaction.
  • No regular user should have GenericAll/GenericWrite on any privileged user or computer. Audit ACLs monthly.
// real named incident
2021 -- COLONIAL PIPELINE -- RANSOMWARE
DarkSide used AD enumeration tools including BloodHound-like techniques.
While DarkSide specifically used Cobalt Strike (not BloodHound), their methodology matched: single VPN credential -> enumerate AD -> find privileged path -> deploy ransomware across domain. Colonial Pipeline paid $4.4M ransom, pipeline shut 6 days, fuel panic across US East Coast. DarkSide affiliate dissolved after US DOJ recovered $2.3M. Lesson: one non-privileged account should not lead to Domain Admin in a single BloodHound edge.
HANDS-ON LAB

Goal: Deploy BloodHound collector on AD environment (authorized). Use SharpHound.exe w...

  1. Deploy BloodHound collector on AD environment (authorized). Use SharpHound.exe with -c All.
  2. Start Neo4j database: sudo neo4j console. BloodHound uses default neo4j/neo4j. Browser opens localhost:7474.
  3. Launch BloodHound: bloodhound --no-sandbox. Auth against Neo4j.
  4. Drag+drop the ZIP file from SharpHound into BloodHound GUI.
  5. Run pre-built query: 'Find all Domain Admins' (top left queries panel).
  6. Run 'Shortest Paths to Domain Admins' -- right-click start node 'Me', see every path.

INTERVIEW READY
Q: "What is BloodHound?"
A: "Graph-based AD analysis tool. SharpHound collects data. Neo4j stores it. BloodHound visualizes relationships as nodes+edges with attack path queries. Maps path from any user to Domain Admin."
Q: "Common attack paths in AD?"
A: "1) User->HasSession->Computer->AdminTo->Admin->MemberOf->DA. 2) Service account with SPN (Kerberoast). 3) User with GenericAll on another user (reset password). 4) GPO that can deploy machine. Analyze edges, remove unnecessary ones."
Q: "How often to run BloodHound?"
A: "Monthly baseline. After any major AD change (new admin, group membership change, GPO update). Always before and after AD hardening project to measure improvement."
