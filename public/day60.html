Windows Event Logs Deep Dive

Event IDs 4624, 4688, 4648, 1102 — these numbers are a SOC analyst's second language. Today you understand every critical Windows Event Log and what attackers try to wipe.

THEORY

Event ID 4624 — Logon: This is the most common Security event, generated every time a user or process authenticates. Key fields: LogonType (2=interactive, 3=network, 4=batch, 5=service, 7=unlock, 8=networkCleartext, 9=newCredentials, 10=remoteInteractive, 11=cachedInteractive). LogonType 3 with a blank TargetUserSid indicates a computer account connection. LogonType 2 from an unexpected IP signals lateral movement. LogonType 9 combined with a network logon is the classic overpass-the-hash indicator — the attacker is using a harvested hash to authenticate as a different user.

Event ID 4688 — Process Creation: Every process launch generates this event when audit policy is enabled. Critical fields: CommandLine (the full command — this reveals everything), ParentProcessId (who spawned it), CreatorProcessId. Suspicious patterns: cmd.exe or powershell.exe spawned by Office (macro attack), wscript.exe or cscript.exe from Temp folder, rundll32.exe with no DLL path, regsvr32.exe with a URL (Squiblydoo attack), certutil.exe used to download files. Sysmon Event ID 1 provides richer data than the default 4688.

Event ID 1102 — Audit Log Cleared: When the Security log is cleared, this event is generated — and it's the last entry before the log goes blank. Attackers who forget about Event ID 1102 leave a permanent record of their log tampering. Combined with Service Control Manager events (7040, 7045) showing a service stopping the EventLog service, this forms a strong "attacker wiping evidence" detection scenario.

Key detection rule: Any Event ID 1102 triggered outside of a scheduled maintenance window should be considered a high-severity incident requiring immediate escalation to Tier 2.
// attacker vs defender mindset
⚔ Attacker
  • Disable Windows Event Log audit policies before running tools — auditpol /clear /category:* prevents event generation entirely.
  • Use alternate logon types that generate fewer events — LogonType 3 (network) for pass-the-hash, LogonType 9 (newCredentials) for token theft.
  • Clear specific event IDs rather than the entire Security log — wevtutil cl Security /rt:True followed by selective re-creation.
  • Name malicious services and processes to closely resemble legitimate Windows services — svchost.exe variants, lsass_helper.dll, etc.
⛑ Defender
  • Forward all Windows Event Logs to a SIEM using Winlogbeat — never rely on local log storage alone; attackers clear local logs first.
  • Enable detailed logging via GPO: Audit Process Creation (include command line), Audit Logon (success + failure), Audit Account Management.
  • Deploy Sysmon across all Windows endpoints — Event IDs 1, 3, 8, 10, 11, 12, 13, 15 provide process, network, and registry telemetry far beyond default logging.
  • Alert specifically on Event ID 1102 (log cleared) with immediate escalation — if any Security log is cleared outside patching, assume breach.
// real named incident
2017 — NOTPETYA — EVENT LOG TAMPERING
NotPetya's wiper component executed 'wevtutil cl' commands to clear Windows Event Logs across infected systems
The NotPetya malware (June 2017) caused over $10 billion in damages globally. Its 'mischa' component executed wevtutil cl against the Security, System, Application, and Setup logs on every infected machine, clearing all entries. Forensic analysts determined that 80% of infected systems had no Security log entries for the period immediately preceding ransomware activation. Systems with Sysmon deployed retained evidence of the wevtutil execution — Sysmon Event ID 1 showed the command line: wevtutil.exe cl Security. This case proved that default Windows Event Log clearing is destructive, and that defense-in-depth logging (Sysmon + Security log) provides critical redundancy.

INTERVIEW READY
Q: "What Windows Event IDs do you monitor for lateral movement detection?"
A: "Event ID 4624 with LogonType 3 (network logon) from an unusual source — a workstation connecting to a server over the network indicates lateral movement. Event ID 4648 (explicit credentials) — a process logged on using alternate credentials, which is the pass-the-hash or overpass-the-hash pattern. Event ID 4688 showing remote execution tools (PsExec, WMI, WinRM) from unexpected parent processes. Event ID 5140 (share accessed) — an admin share accessed from a non-admin workstation."
Q: "How does Sysmon extend Windows Event Logging beyond the default configuration?"
A: "Sysmon provides 29 event IDs that default logging doesn't cover. Key ones: Event ID 1 (process creation with full command line including hash), Event ID 3 (network connection mapped to the process), Event ID 7 (module load), Event ID 8 (CreateRemoteThread — the core API for process injection), Event ID 10 (process access — who opened a handle to LSASS), Event ID 11 (file creation), Event ID 12/13/14 (registry modifications), Event ID 15 (file stream — alternate data stream detection)."
Q: "Can an attacker completely avoid generating Windows Event Logs?"
A: "Not completely. Event ID 1102 (log cleared) always fires when the Security log is cleared. Attackers can disable the Event Log service, but that generates service control manager events (7040, 7045). They can toggle audit policies with auditpol /clear, which requires admin privileges. The most sophisticated approach is to selectively delete individual event records, but this requires knowing exactly which events to delete. Best defense: ship logs to a remote SIEM in real-time so local tampering doesn't destroy the evidence."
