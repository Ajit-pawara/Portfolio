Broken Authentication & Access Control

Authentication proves who you are. Access control proves what you can do. When either breaks, attackers become administrators. Today you test both.

THEORY

Broken Authentication covers any flaw in how an application verifies identity: weak password policies, credential stuffing (automated login attempts with breached credentials), session fixation, JWT manipulation, and MFA fatigue attacks. OWASP consistently ranks broken authentication in the Top 10 because a single authentication bypass grants the attacker full account access.

JWT structure and the "none" algorithm attack:

# A JWT has three base64url-encoded parts separated by dots: header.payload.signature # Legitimate token: eyJhbGciOiJSUzI1NiIsInR5cCI6IkpXVCJ9. eyJ1c2VyIjoiYWRtaW4iLCJyb2xlIjoidXNlciJ9. signature_here # Algorithm confusion attack — attacker changes alg to "none": # Server using a vulnerable JWT library will accept empty signature: eyJhbGciOiJub25lIiwidHlwIjoiSldUIn0. eyJ1c2VyIjoiYWRtaW4iLCJyb2xlIjoiYWRtaW4ifQ.

Broken Access Control (IDOR, PrivEsc, RBAC bypass): Access control enforces what an authenticated user can do. Common failures: Insecure Direct Object References (IDOR) — changing /profile/123 to /profile/456 to see another user's data; privilege escalation — a regular user accessing admin endpoints; missing function-level access controls — the admin panel has no authorization check, just a different URL.

Key insight: Authentication and access control are separate concerns. A user can be perfectly authenticated (valid session, strong MFA) but still exploit broken access control (accessing data they shouldn't). Many security assessments focus heavily on authentication while neglecting authorization — a pattern attackers actively exploit.
// attacker vs defender mindset
⚔ Attacker
  • Test for JWT algorithm confusion: decode the token, change alg to none or swap RS256 to HS256 using the public key as the HMAC secret, and re-encode.
  • Use credential stuffing with breached credentials — hydra -l admin -P rockyou.txt target.com/login. Automated login attempts at scale.
  • Check for IDOR by incrementing/decrementing numeric IDs in URL paths or API requests. Test /api/user/2 when logged in as user 1.
  • Try forced browsing — access /admin, /dashboard, or /api/admin/users directly with a low-privilege session.
🛡 Defender
  • Always use the latest JWT libraries with explicit algorithm whitelisting — never rely on the token's alg header to select the verification algorithm.
  • Implement rate limiting on login endpoints — 5 failed attempts = 15-minute lockout. This stops credential stuffing and brute force.
  • Use server-side session management with random, unpredictable session IDs — regenerate ID after login, enforce expiry, implement absolute and idle timeouts.
  • Enforce authorization checks on every API endpoint and page load — never rely on hiding UI elements. Check the user's role/permissions server-side.
// real named incident
2018 — JWT ALGORITHM CONFUSION — COUNTLESS APPLICATIONS VULNERABLE
JWT libraries defaulted to accepting 'none' algorithm, allowing universal token forgery
In 2018, security researchers disclosed a widespread flaw in JSON Web Token implementations: many server-side JWT libraries would accept a token with the algorithm header set to "none" and an empty signature — effectively trusting any payload the attacker sent. Applications using these libraries would verify the token as valid regardless of who created it. The fix required libraries to whitelist accepted algorithms rather than trusting the token's header. This vulnerability affected thousands of applications using JWT for authentication, including major SaaS platforms and enterprise systems.

INTERVIEW READY
Q: "What is the JWT algorithm confusion attack?"
A: "The attacker modifies the JWT header to change the algorithm from RS256 (asymmetric, requires private key to sign) to HS256 (symmetric, uses a shared secret) and signs the token using the server's public key as the HMAC secret. Since the public key is often known or obtainable, the attacker can forge valid tokens. Another variant sets alg to 'none' where the library accepts an empty signature. The fix is whitelisting accepted algorithms server-side."
Q: "What's the difference between authentication and authorization?"
A: "Authentication verifies who you are — your identity. Authorization determines what you're allowed to do — your permissions. You can be authenticated as a regular user but not authorized to access admin endpoints. Many breaches happen when authentication is strong but authorization checks are missing. Every API endpoint must independently verify both."
Q: "What is IDOR and how do you test for it?"
A: "Insecure Direct Object Reference is when an application exposes a direct reference to an internal object (like a database ID) in a URL or API parameter without proper authorization checks. I test by creating two accounts, noting the object IDs for each, then trying to access account B's resources while logged in as account A. If the server returns account B's data, there's an IDOR vulnerability."
