OWASP Top 10 & Web App Security Fundamentals
Equifax / Apache Struts CVE-2017-5638 (2017)
// three mental-model buckets
Bad Input Handling
SQLi, XSS — attacker input isn't validated, so it gets executed. #1 on every OWASP list.
Bad Access Decisions
Broken auth, IDOR, CSRF — the system trusts the request without proving identity or permission.
Bad Hygiene
Vulnerable components, security misconfig, insufficient logging — known problems left unfixed.
// the real incident

Equifax 2017 (CVE-2017-5638): Apache Struts RCE via a malformed Content-Type header. Equifax knew about the patch — it was 2 months old when attackers hit. 147M records. The root cause isn't the exploit — it's OWASP #06 (Vulnerable Components). The patch existed. The process to apply it didn't.

// interview answer
Q: Walk me through the OWASP Top 10 in one sentence per category.
Bad Input (A01-A03): Injection, broken auth, sensitive exposure. Bad Access (A04-A05, A07): IDOR, misconfiguration. Bad Hygiene (A06, A08-A10): vulnerable components, logging failures, SSRF. Group them into three buckets and you'll never blank in an interview.

OWASP Top 10 Overview
OWASP Top 10 2021 — Insecure Design
// 2021 changes
Insecure Design (A04)
NEW category. Bugs are implementation errors; design flaws are architectural. You can't patch bad architecture.
SSRF (A10)
NEW entry. Server-Side Request Forgery — the server makes requests to internal resources based on attacker-supplied URLs.
Logging Failures (A09)
Not knowing you're breached is a vulnerability. If you can't detect it, you can't contain it.
// the real shift

In 2021 OWASP moved from "how you code" to "how you think about security." Insecure Design means a secure development lifecycle (SDL) with threat modeling at the requirements stage — not bolted on after coding. Most breaches start with a design assumption, not a coding mistake.

SQL Injection Deep Dive
Heartland Payment Systems (2008)
// three injection types
UNION-Based
' UNION SELECT username,password FROM users -- Attacker appends their own SELECT to the original query. Most visible type.
Error-Based
Force the database to spill data in error messages. MySQL extractvalue() reveals data via SQL error output.
Blind SQLi
No data returned — but true/false responses leak one bit at a time. ' OR 1=1 -- vs ' OR 1=2 --
// the defense
SELECT * FROM users WHERE email = ?
Parameterized query — the ? is a placeholder. User input is data, not executable. PREVENTS SQLi entirely.
cursor.execute("SELECT * FROM users WHERE email = %s", (email,))
Python psycopg2 example — database driver handles escaping. Never use f-strings for SQL.
// interview answer
Q: How does a parameterized query prevent SQL injection?
The SQL statement structure is compiled first with placeholders. User input is sent separately as data — the database engine cannot interpret it as SQL code even if it contains quote characters or UNION keywords. The query structure and the data never mix.
// real incident

Heartland Payment Systems (2008): Albert Gonzalez used SQL injection through web forms to plant SQL queries that extracted 130M credit card numbers over months. Heartland processed 100M transactions/month — the breach wasn't detected by internal monitoring, only by Visa's fraud analytics. $145M in damages.

Cross-Site Scripting (XSS)
British Airways / Magecart (2018)
// three XSS types
Stored XSS
Payload saved on server (e.g., comment field). Every visitor gets infected. Most dangerous.
Reflected XSS
Payload in URL or request. Victim must click a crafted link. Common in search parameters.
DOM-Based XSS
Payload never reaches server — JavaScript reads from URL hash and writes to DOM. Hardest to detect.
// the defense
Content-Security-Policy: default-src 'self'
CSP header — blocks inline scripts and external script sources. Kills most XSS vectors dead.
<script>alert(document.cookie)</script>
Classic XSS test payload. If this executes, any script can — not just alerts.
// real incident + interview answer
Q: Walk through the British Airways Magecart attack. What was the root cause?
Magecart injected JavaScript via a third-party script vulnerability on BA's payment page. The script read form fields (name, card number, CVV) before HTTPS submission and exfiltrated them to an attacker-controlled domain. 380K transactions compromised. Root cause: no Subresource Integrity (SRI) on third-party scripts and no CSP restricting script-src. Every script loaded had the same privilege as BA's own code.

CSRF & Session Attacks
Facebook CSRF Token Flaw (2013)
// core concepts
CSRF
Attacker tricks your browser into making an authenticated request you didn't intend. Cookie is sent automatically — the server thinks it's you.
Session Hijacking
Stealing a session cookie (via XSS, network sniffing, or leaked token). Attacker now IS you without needing your password.
Anti-CSRF Token
Random token embedded in every form. Server validates it. Attacker's fake form can't know the token.
// defenses
Set-Cookie: sessionid=abc; SameSite=Strict
SameSite=Strict — cookie won't send on cross-site requests. Blocks CSRF at the browser level.
<input type="hidden" name="csrf_token" value="random">
Anti-CSRF token in forms — server checks it matches the session. Attacker's forged request doesn't have this value.
// real incident

Facebook CSRF (2013): attackers crafted image tags that sent POST requests to Facebook's "like" and "post" endpoints while victims were logged in. Since cookies were sent automatically and Facebook lacked anti-CSRF tokens on those endpoints, the forged requests succeeded. Users posted attacker-controlled content without ever clicking a button.

Broken Authentication & Access Control
JWT Algorithm Confusion (2018)
// authentication vs authorization
Authentication
"Who are you?" — login, password, MFA, session token. Proving identity.
Authorization
"What can you do?" — RBAC roles, permissions, access levels. Happens AFTER auth.
// attack techniques
Credential Stuffing
Using leaked username/password pairs from one breach to log into another service. Users reuse passwords.
MFA Fatigue
Spam push notifications until the victim accepts one to make it stop. Uber breached this way in 2022.
JWT "none" Algorithm
Change JWT header "alg":"HS256" to "alg":"none". Some libraries accept and skip signature verification. Forge any token.
// interview answer
Q: What is JWT algorithm confusion and how do you prevent it?
The server reads the token's algorithm header and validates accordingly. If you send {"alg":"none"}, some libraries skip signature verification entirely — you can forge any user's token. Prevention: whitelist allowed algorithms at the application level, never trust the token's header. Use asymmetric (RS256) over symmetric (HS256) and validate the key type matches.

Security Misconfiguration
Verizon S3 Bucket Exposure (2017)
// what to check
Default Credentials
admin:admin, root:toor. First thing every scanner checks. Shodan finds these in production daily.
Open S3 Buckets
Public-read on a bucket means the entire internet can list and download every object. No auth required.
Directory Listing
Server exposes all files in /uploads/. Attackers browse for config files, backups, stored uploads.
Verbose Errors
Stack traces with SQL queries, file paths, and internal IPs. Debug mode in production = free recon.
// real incident

Verizon S3 (2017): an internal S3 bucket was left publicly accessible — no authentication required to list or download objects. 14M customer records including names, PINs, account details, and call logs. The bucket was discovered by UpGuard researchers doing routine internet scanning. No sophisticated exploit — just an HTTP request to the bucket URL. This is what "security misconfiguration" looks like.

// command
aws s3 ls s3://bucket-name --no-sign-request
List an S3 bucket without credentials. If this returns data, the bucket is publicly accessible — a finding in any pentest.

File Upload & SSRF Vulnerabilities
Capital One SSRF + S3 Exfiltration (2019)
// file upload bypasses
Extension Bypass
Block .php but accept .php5, .phtml, .PHP (case). Blacklists always miss variations.
MIME Bypass
Server checks Content-Type: image/jpeg. Attacker uploads PHP with image/jpeg header. MIME is just a header — trivially spoofed.
Magic Byte Bypass
Server checks file signature (GIF89a). Attacker prepends GIF header to PHP payload. File command fails, but PHP executes.
// SSRF explained
What it is
Application fetches a URL you provide. Instead of an external URL, you supply http://169.254.169.254/latest/meta-data/ (AWS metadata endpoint).
What it gets
IAM role credentials, internal service info, cloud provider secrets. The server makes the request — firewalls trust the server.
// real incident

Capital One 2019: Paige Thompson exploited a misconfigured AWS WAF that allowed SSRF to the EC2 metadata service (169.254.169.254). The metadata service returned IAM role credentials, which had read access to S3 buckets containing 100M+ credit applications. One SSRF request + one over-privileged IAM role = the largest cloud data breach at the time.

Firewalls, IDS, IPS Explained
Palo Alto PAN-OS Zero-Day CVE-2021-3064 (2021)
// three devices, one goal
Firewall
Rules-based gatekeeper. Blocks/allows by port, IP, protocol. Stateless (packet-by-packet) or stateful (tracks connections).
IDS (Snort/Suricata)
Watches traffic, matches against signatures, ALERTS. Does NOT block — you review logs later.
IPS (inline)
Same as IDS but ACTIVE — drops malicious packets in real-time before they reach the target. Higher risk of false positives blocking legit traffic.
// iptables basics
iptables -A INPUT -p tcp --dport 22 -s 192.168.1.0/24 -j ACCEPT
Allow SSH only from your subnet. Default deny everything else.
iptables -A INPUT -p tcp --dport 80 -j DROP
Drop all HTTP traffic — no web server exposed. Silent reject, no response sent.
iptables -L -v -n
List all rules with packet/byte counters — see what's being matched and how often.
// real incident

PAN-OS CVE-2021-3064: a buffer overflow in Palo Alto's management interface allowed unauthenticated RCE over the network. The irony: the firewall itself needed to be patched. If the management interface was exposed to the internet (a common misconfiguration), attackers could bypass every rule the firewall enforced by owning the firewall itself.

VPNs & Proxies
NordVPN Datacenter Breach (2019)
// vpn vs proxy
VPN
Encrypted tunnel for ALL traffic at OSI L3. Entire device's traffic routed through remote network. More secure, more overhead.
Proxy
Application-level (L7). Only traffic for that app/protocol. SOCKS5 proxies can handle any TCP traffic but don't encrypt by default.
Reverse Proxy
Sits in front of servers — hides backend topology, terminates TLS, load balances. Nginx, HAProxy.
// openvpn setup
openvpn --config client.ovpn
Connect to an OpenVPN server using client config. All traffic routed through encrypted tunnel.
ssh -D 1080 user@remote
SOCKS5 proxy over SSH. Configure browser to use localhost:1080 — traffic tunnels through SSH.
curl --proxy socks5://127.0.0.1:1080 ifconfig.me
Test your proxy — should show remote server's IP, not yours.
// real incident

NordVPN 2019: a Finnish datacenter hosting a NordVPN server was compromised — unauthorized access to the server room allowed attackers to obtain TLS private keys. NordVPN's PFS (Perfect Forward Secrecy) meant past sessions couldn't be decrypted even with the keys. The takeaway: VPN providers can be compromised at the infrastructure level. PFS protects historical traffic. Always verify a VPN supports it.

// Quick Self-Quiz — Can You Answer These Without Looking?

Cover the answers. Say them out loud. If you can't answer one fluently, revisit that day before continuing.

Q01
What three mental-model buckets organize the OWASP Top 10? Give an example from each.
Bad Input Handling (SQLi, XSS), Bad Access Decisions (broken auth, IDOR, CSRF), Bad Hygiene (vulnerable components, misconfiguration, logging failures).
Q02
How does a parameterized query prevent SQL injection at the database engine level?
SQL structure is compiled with placeholders first, then user input is sent as data — the engine cannot interpret it as SQL code even if it contains quotes or UNION keywords.
Q03
What is the difference between stored, reflected, and DOM-based XSS?
Stored = payload saved on server (comment field). Reflected = payload in URL/request param. DOM-based = payload in URL hash, never reaches server — manipulated client-side by JS.
Q04
How did the Magecart attack on British Airways steal payment details without breaking HTTPS?
Injected JS via third-party script read form fields before HTTPS submission and exfiltrated to attacker domain. No SRI on scripts plus no CSP meant injected code had full page access.
Q05
What is a CSRF attack and what two defenses prevent it?
Forge authenticated request using victim's cookies sent automatically. Defenses: anti-CSRF tokens in forms AND SameSite=Strict/Strict cookie attribute.
Q06
Walk through the JWT algorithm confusion attack. What header value do you change and why does it work?
Change "alg":"HS256" to "alg":"none". Some libraries skip signature verification when alg=none. Prevention: whitelist allowed algorithms server-side instead of trusting the token header.
Q07
How did the Capital One SSRF attack work? What two misconfigurations allowed it?
SSRF to 169.254.169.254 (AWS metadata) returned IAM credentials. Two misconfigurations: WAF allowed SSRF, and the IAM role had read access to 100M+ S3 records it didn't need.
Q08
What does iptables -P INPUT DROP do? Why is it a better baseline than allowing everything?
Sets default policy to DROP — any traffic without an explicit ACCEPT rule is silently dropped. Forces you to whitelist specific ports instead of blacklisting — far more secure.
Q09
What is the difference between a VPN and a proxy? When would you use each?
VPN tunnels ALL traffic at OSI L3 — entire device encrypted. Proxy handles single app at L7. Use VPN for full privacy (public Wi-Fi). Use proxy for one-off routing (access geo-blocked content in browser only).
Q10
Why did Perfect Forward Secrecy protect NordVPN customers even after the datacenter breach exposed private keys?
PFS generates unique session keys per connection using ephemeral Diffie-Hellman. Even with the server's long-term private key, an attacker cannot decrypt past recorded sessions — each session's key is independently generated and discarded.
