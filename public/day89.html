Key insight: The final report is the deliverable. The compromise is just a step along the way. If you get root access in 30 minutes but spend the remaining 90 minutes writing a clean report with screenshots, CVSS scores, and remediation steps, you've done a professional job. If you spend 2 hours compromising the machine and leave 10 minutes for a rushed report, the client (and your interviewer) will not be impressed. Document as you go — take a screenshot after every successful command.

Sample pentest report structure for reference: Professional reports include: Executive Summary (non-technical, what was found, risk level), Scope & Methodology (IPs, tools, approach), Risk Summary (table with finding name, CVSS, severity, status), Detailed Findings (each vulnerability: description, proof, CVSS vector, remediation), Recon Data (full Nmap output, discovered URLs, enumerated users), Exploit Chain (step-by-step with timestamped commands), and Remediation Appendix (prioritized fix list).

// execution mindset
⚔ Pentester Mindset
  • Work your plan, not the tools. Start with recon (what's here?), then threat model (what's the easiest path?), then exploit (one door at a time). Every command should have a purpose.
  • Take a screenshot after every successful command. Terminal output, tool run, shell access, escalation step, flag file. Screenshots with visible timestamps are the gold standard for proof.
  • If a tool or exploit doesn't work after 3 attempts, move on. There are always multiple paths to root on any target. vsftpd didn't work? Try UnrealIRCd. Not there? Try Samba. Not there? Try the web app.
  • Simulate the "exfiltration" step by finding and reading a sensitive file (shadow, database config, customer data CSV). Don't actually transfer it off the network if this is a shared environment.
🛡 Client Mindset
  • The client wants to know: "Could an attacker do what you just did, and what do I need to fix to stop them?" Every finding should answer both questions.
  • They appreciate responsible disclosure language. Instead of "this is vulnerable," write "the vsftpd service version 2.3.4 contains a known backdoor (CVE-2011-2523) that allows unauthenticated remote attackers to execute arbitrary commands on the affected system."
  • They value risk-based prioritization. Not every finding is equally urgent. CVSS 10.0 remote code execution on an internet-facing server is different from CVSS 4.3 information disclosure on an internal-only staging box.
  • They need actionable remediation. "Patch vsftpd to version 2.3.5+" is actionable. "Improve security" is not. Every finding must have a specific, verifiable fix instruction.

PROFESSIONAL PENTEST REPORT STRUCTURE — INDUSTRY STANDARD
Professional pentest reports follow a standard structure used by Rapid7, CrowdStrike, Deloitte, and boutique firms
The penetration testing industry has converged on a standard report structure over the past decade. PTES (Penetration Testing Execution Standard) and OWASP's testing guide both define report sections that nearly all commercial firms follow. Key components: Executive Summary (one page max, no technical jargon), Technical Summary (findings overview with risk matrix), Methodology (tools, approach, scope), Detailed Findings (each with description, proof, CVSS 3.1 vector, screenshot, remediation), Exploit Chains (multi-step attack narratives), and Appendices (raw tool outputs, full Nmap results, discovered credentials). Your capstone report should mirror this structure exactly — when an interviewer sees your report matches the professional format, they know you've done your research.
HANDS-ON LAB

Goal: Execute the full pentest plan from Day 88 against Metasploitable 2. Achieve initial access, escalate to root, document every step, and produce the final report.

  1. Start your timer (2 hours recommended). Power on Metasploitable 2. Run Phase 1 recon: nmap -sV -sC -p- [target] and Gobuster directory scan. Document all open ports and services.
  2. Identify at least 3 potential vulnerabilities from the Nmap output. Prioritize: vsftpd 2.3.4 backdoor (CVE-2011-2523), UnrealIRCd backdoor (CVE-2010-2075), or Tomcat manager default credentials.
  3. Exploit one service to get initial access. For vsftpd: msfconsole -q -x "use exploit/unix/ftp/vsftpd_234_backdoor; set RHOSTS [target]; run". Screenshot the shell.
  4. Escalate privileges. On Metasploitable 2, the vsftpd backdoor gives root directly. If you used a different vector, run sudo -l, check SUID binaries, or run linpeas to find escalation paths.
  5. Simulate data exfiltration: find and cat /etc/shadow, /etc/passwd, and any flag files. Take screenshots with timestamps. Do NOT transfer files off the VM.
  6. Write the complete report into your Day 88 template. Add all screenshots, CVSS scores, exploit chain narrative, and remediation recommendations. Export as PDF.
Deliverable: Complete Pentest Report PDF (5-8 pages) with Executive Summary, Scope, Risk Summary, 3+ detailed findings with screenshots and CVSS scores, exploit chain walkthrough, and remediation appendix. This is the crown jewel of your entire 90-day portfolio.

Q: "Tell me about a penetration test you performed from start to finish."
A: "As part of my 90-day cybersecurity portfolio, I conducted a full penetration test against a deliberately vulnerable Linux target (Metasploitable 2). I started by defining scope and rules of engagement, then ran Nmap full port scan and service enumeration. I identified vsftpd 2.3.4 which has a known backdoor (CVE-2011-2523). I exploited it using Metasploit and gained a root shell immediately. I documented every step with timestamped screenshots, extracted password hashes as a proof of impact, and wrote a comprehensive 6-page report with executive summary, CVSS-scored findings, exploit chain narrative, and remediation recommendations."
Q: "How do you ensure the accuracy of findings in a pentest report?"
A: "I verify every finding multiple ways. For example, if a scanner reports vsftpd 2.3.4, I confirm by connecting to the port and reading the banner directly. I test the exploit in a controlled way before adding it to the report. I calculate CVSS scores using the official calculator, not guesswork. I include screenshots of each step with visible timestamps as proof. Before finalizing, I review the report for false positives — can I reproduce this finding consistently? If not, I mark it as a potential finding requiring further investigation."
Q: "What would you include in an executive summary for a penetration test report?"
A: "The executive summary is written for non-technical leadership. It includes: the engagement dates and scope (what was tested), the overall risk rating (Critical/High/Medium/Low based on highest severity finding confirmed), a brief summary of key findings in business language — 'An attacker could gain full control of the web server and access customer data,' the number of findings by severity (e.g., 2 Critical, 3 High, 1 Medium), and a high-level remediation priority statement. It does NOT include technical jargon, CVSS vectors, or exploit details — that goes in the technical sections."
