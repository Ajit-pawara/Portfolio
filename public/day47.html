Password Cracking Hashcat & John the Ripper

Passwords are the most common authentication mechanism and the most common failure point. Today you crack hashes using GPUs, rules, and wordlists.

THEORY

Why password cracking matters: In every penetration test, you'll extract password hashes — from SQL injection on web apps, from dumped SAM hives on Windows, from /etc/shadow on Linux, or from network captures. These hashes are the keys to the kingdom. Cracking them reveals the plaintext passwords users chose, which often get reused across services. The difference between a useful pentest and a forgettable one is often whether the team cracked the hashes and demonstrated the business impact of weak passwords.

Hash modes (Hashcat) you must know:

Mode 0 — MD5 (legacy web apps, CMS systems) Mode 1000 — NTLM (Windows password hashes) Mode 3200 — bcrypt $2*$ (modern web apps, Blowfish-based) Mode 1800 — sha512crypt (Linux /etc/shadow) Mode 22000 — WPA-PBKDF2-PMKID (Wi-Fi WPA2 handshake) Mode 13100 — Kerberos 5 TGS-REP (AD service tickets) Mode 17200 — PKZIP (encrypted ZIP files)

The cracking workflow: (1) Identify the hash type using hashid or hash-identifier. (2) Select the correct Hashcat mode. (3) Start with a dictionary attack against rockyou.txt — this cracks 30-50% of hashes. (4) Apply rule-based attacks — best64.rule appends common suffixes, d3ad0ne.rule tries leet-speak substitutions. (5) For remaining hashes, brute-force with mask attacks — ?u?l?l?l?d?d?d?d targets the most common password structures. The key insight: you don't need to crack every hash. You need to crack enough to demonstrate the pattern.

Key insight: Hashcat uses your GPU. A single modern GPU (RTX 3080) cracks 100 billion MD5 hashes per second — that's every 8-character lowercase password in under 3 seconds. The bottleneck is almost never the hash; it's the wordlist quality and rule selection. Invest in wordlist curation, not hardware.
// attacker vs defender mindset
⚔ Attacker
  • Always identify the hash type first — feeding an NTLM hash into bcrypt mode wastes hours. Use hashid or paste the hash into an LLM for instant identification.
  • Start with rockyou.txt + best64.rule — this combination cracks most hashes under 8 characters before you even consider brute-force or mask attacks.
  • For WPA2 handshakes, use hc22000 mode + rockyou.txt. Packet captures store the handshake — hcxpcapngtool converts them to Hashcat format.
  • Loop-mode cracking: crack what you can, use the cracked passwords as seeds to crack related accounts (password reuse), then escalate laterally.
🛡 Defender
  • Use slow hashing algorithms — bcrypt, Argon2, scrypt, or PBKDF2 — which make GPU cracking 10,000x slower than MD5 or SHA-1. Password complexity doesn't matter if the hash cracks in milliseconds.
  • Always use salts with password hashes — the LinkedIn breach (167M unsalted SHA-1 hashes) was 90% cracked in days. Salting forces attackers to crack each hash individually.
  • Enforce minimum password length (12+ characters) over complexity rules — "correct-horse-battery-staple" takes centuries to brute-force regardless of hash speed.
  • Implement account lockout and MFA — even if an attacker cracks a hash, they still can't log in without the second factor.
// real named incident
2012 — LINKEDIN BREACH — 167 MILLION PASSWORDS
Security researchers used Hashcat with GPU clusters to crack 90%+ of unsalted SHA-1 hashes within days
In 2012, LinkedIn suffered a data breach exposing 167 million email addresses and password hashes. The passwords were hashed with SHA-1 without salts — a catastrophic design failure. When the data was finally released publicly in 2016, security researchers deployed Hashcat on GPU clusters and cracked over 90% of the hashes within days. The most common password was "123456" (753,000 users). The breach became the definitive case study for why unsalted fast hashes are unacceptable. LinkedIn settled a class-action lawsuit for $1.25M. The incident directly influenced NIST SP 800-63 guidelines that now explicitly require salted, slow password hashing.
HANDS-ON LAB

Goal: crack real password hashes of multiple types using both Hashcat (GPU) and John the Ripper (CPU), and apply rule-based attacks.

  1. Create a lab directory: mkdir ~/cracking-lab && cd ~/cracking-lab. Download rockyou.txt: sudo apt install wordlist-rockyou or find it at /usr/share/wordlists/.
  2. Generate test hashes: echo -n "password123" | md5sum | cut -d' ' -f1 > md5.txt. mkpasswd -m sha-512 "Password123!" > sha512.txt.
  3. Use hashid: hashid -m md5.txt — it should return mode 0. Crack it: hashcat -m 0 md5.txt /usr/share/wordlists/rockyou.txt.gz --show.
  4. Create a custom rule file (custom.rule) with: $1 $2 $3 (append 1,2,3). Run: hashcat -m 0 md5.txt rockyou.txt -r custom.rule.
  5. Download a sample NTLM hash file from a CTF or use: echo "Administrator:500:aad3b435b51404eeaad3b435b51404ee:31d6cfe0d16ae931b73c59d7e0c089c0:::" > ntlm.txt. Crack with mode 1000.
  6. Document: hash type identification method, cracking command, time to crack, and the rule/mask used. Compare Hashcat vs John for the same workload.
Deliverable: A password cracking lab report showing at least 3 different hash types cracked with Hashcat or John the Ripper. Include the crack times, rules used, and the identified passwords.
# Step 1 — Identify hash type hashid -m '$2y$10$N9qo8uLOickgx2ZMRZoMyeIjZAgcfl7p92ldGxad68LJZdL17lhWy' bcrypt $2y$ | Mode: 3200 # Step 2 — Dictionary attack (MD5, mode 0) hashcat -m 0 -a 0 md5_hashes.txt /usr/share/wordlists/rockyou.txt.gz # Step 3 — Rule-based attack (append digits) echo '$1 $2 $3 $4' > append.rule hashcat -m 0 -a 0 md5_hashes.txt /usr/share/wordlists/rockyou.txt.gz -r append.rule # Step 4 — Crack NTLM (Windows, mode 1000) hashcat -m 1000 -a 0 ntlm_hashes.txt rockyou.txt # Step 5 — Mask attack (8-char lowercase) hashcat -m 0 -a 3 md5_hashes.txt ?l?l?l?l?l?l?l?l # Step 6 — John the Ripper alternative john --format=raw-md5 --wordlist=rockyou.txt md5_hashes.txt john --show md5_hashes.txt
INTERVIEW READY
Q: "Walk me through your password cracking methodology when you encounter a set of hashes."
A: "First I identify the hash type using hashid or pattern analysis. Then I select the correct cracking mode and start with a dictionary attack using rockyou.txt — this cracks common passwords instantly. For remaining hashes, I apply rule-based attacks starting with best64.rule, then move to mask attacks targeting common patterns like Capital+lower+digits. Finally I use brute-force for short hashes under 8 characters. I track crack rates to decide when to stop — if rockyou.txt + best64 doesn't crack anything after 10 minutes, I switch to a different approach or wordlist."
Q: "What's the difference between a dictionary attack, a rule-based attack, and a mask attack?"
A: "A dictionary attack tries each password from a wordlist exactly as written. A rule-based attack applies transformations to each wordlist entry — appending digits, substituting letters (a>@, e>3), capitalizing — which cracks passwords that are wordlist variations. A mask attack generates passwords matching a pattern, like ?u?l?l?l?d?d?d?d (uppercase + 3 lowercase + 4 digits). Dictionary attacks are fastest (most constrained), rule attacks cover common mutations, and mask attacks cover structural patterns. I use them in that order."
Q: "Why is bcrypt better than SHA-1 for password storage?"
A: "SHA-1 was designed for speed — it computes billions of hashes per second on a GPU, making dictionary attacks trivial. Bcrypt is designed to be slow and computationally expensive — it includes a work factor parameter (cost) that controls iteration count. A cost factor of 10 means 2^10 = 1024 iterations, but each iteration is Blowfish-based and GPU-unfriendly. Even on a high-end GPU, bcrypt cracks at around 5,000 hashes per second versus 10 billion for SHA-1. Bcrypt also includes a built-in salt, eliminating rainbow table attacks."
AI TOOLS
Hash identifier assistant
Paste an unknown hash into an LLM — it can identify the format and suggest the correct Hashcat mode faster than hashid for complex multi-algorithm formats.
Rule generation engine
Describe the target password policy (e.g., "Must include uppercase, digit, and special char, 8-12 chars") and have an LLM generate optimal Hashcat rules and masks.
Wordlist curator
Ask which wordlists are most effective for a specific scenario — "best wordlist for cracking NTLM hashes from a corporate Domain Controller" — and where to find them.
Performance cheat sheet
Ask for a comparative cracking speed table: "How many hashes per second for mode 0, mode 1000, mode 3200 on an RTX 4080?" — plan your cracking strategy.
