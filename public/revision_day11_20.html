Python for Security II — Sockets & Automation
Mirai Botnet 2016 · Dyn DNS Attack
// core concepts
Socket
Programming interface for network connections. s = socket.socket() → s.connect((ip,port)) → s.send() → s.recv(). Like a phone handset in code.
SOCK_STREAM (TCP)
Reliable, ordered, connection-based. Use for web, SSH, anything needing guaranteed delivery.
SOCK_DGRAM (UDP)
Fast, connectionless, fire-and-forget. Use for DNS queries, speed-critical recon.
Threading
threading.Thread makes concurrent connections. A for loop + threads scans 65K ports in seconds — this is how Mirai scanned the entire internet.
// automation pattern

Once you open one socket in Python, a for loop scans 65K ports. Threading makes it concurrent. This is exactly how network scanners, brute-forcers, and botnets work — the same 10 lines of socket code, looped intelligently. Always use s.settimeout() to avoid hanging on closed ports.

// real incident

Mirai Botnet (2016): scanned the entire IPv4 internet for open Telnet (port 23) using concurrent socket connections, then tried 61 default username/password combos. Absorbed routers, cameras, DVRs into the botnet. Result: Oct 2016 Dyn DDoS attack took Twitter, Netflix, Reddit offline. Lesson: socket automation at scale is the foundation of both legitimate security tools and the most damaging internet attacks.

// interview answer
Q: How would you use Python sockets in a real security context?
"I've built a banner grabber that connects to open ports and reads service version strings — feeding directly into CVE lookups — and a threaded port scanner using threading for speed. This is the same technique the Mirai botnet used in 2016 to scan the internet for open Telnet ports. Understanding the mechanism makes it straightforward to detect: one source IP rapidly connecting to many destinations on the same port is the scanner signature you'd watch for in a SIEM."

Windows Internals — Registry, Processes & Event Logs
SolarWinds SUNBURST 2020
// registry hives
HKLM
Settings for the entire computer — most important hive for security.
HKCU
Settings for the logged-in user. Malware running as standard user hides here.
Run Key
HKLM\...\CurrentVersion\Run — THE persistence key. Malware writes here to survive reboot.
Services
HKLM\SYSTEM\CurrentControlSet\Services — malware installs as a service for deep persistence.
// core processes — spot the fakes
lsass.exe
Handles authentication. Stores password hashes in memory. Parent = wininit.exe. Primary target for Mimikatz.
svchost.exe
Service Host. Multiple instances normal. Suspicious if not from System32.
explorer.exe
File manager. One per logged-in user. Malware injects here for evasion.
csrss.exe
Critical system process. If killed, Windows crashes. Must run from System32.
// event log IDs to memorize
4624
Successful logon. Baseline all activity. Watch unusual logon types.
4625
Failed logon. Spikes = brute force. First alert to set up.
4688
Process creation. Disabled by default. Catches living-off-the-land attacks.
1102
Audit log cleared. Attacker cleanup. This event fires when they try to clear.
4104
PowerShell script block logging. See exact commands run. Essential for detection.
// real incident

SolarWinds SUNBURST (2020): attackers compromised the SolarWinds build pipeline, inserted a backdoor in a trusted update. 18,000 organizations installed it — US Treasury, DOJ, DHS, Fortune 500s. SUNBURST stored C2 config in the Windows Registry to avoid suspicious file writes. Stayed dormant up to 2 weeks, mimicked legitimate API calls. Undetected for 9 months. Lesson: Event IDs 4688 and 4104 properly collected would have significantly raised detection chances.

// interview answer
Q: What Windows Event IDs do you consider most important for security monitoring?
"4625 failed logons — spike = brute force. 4688 process creation — disabled by default but catches most living-off-the-land attacks. 4104 PowerShell logging — you get the actual commands. 1102 audit log cleared — self-referential: even if attackers clear logs, the clearing itself fires this ID. SolarWinds went 9 months undetected partly because these logs weren't being collected across affected organizations."

Lesson: misconfigured virtualization/container isolation is a fast-growing cloud attack vector. The attacker didn't break in — they added their own VM alongside yours.

// four core tools
Nmap
Port scanning, service/OS detection, NSE scripts. nmap -sV -sC -O 192.168.56.102 — all-in-one recon. --script vuln flags known CVEs.
Wireshark
Packet capture & analysis. Filters: http.request, tcp.port==21, ftp.request.command, tcp.flags.syn==1.

Exploitation framework. msfconsole → search/use/exploit. vsftpd 2.3.4 backdoor gives root shell in 5 commands.
Burp Suite
Intercepting proxy. Browser → 127.0.0.1:8080. Intercept, modify, replay HTTP/S requests.
// nmap switches to know cold
nmap -sV [ip]
Service version detection.
nmap -sS [ip]
SYN stealth scan — fastest, least logged, needs root.
nmap -A [ip]
Aggressive — service, OS, traceroute, scripts combined.
nmap --script vuln [ip]
Vulnerability detection scripts — quick CVE check.
nmap -p- [ip]
Scan all 65,535 ports — slow but thorough.
// real incident

Bangladesh Bank SWIFT Heist (2016): Lazarus Group installed Wireshark inside the bank's network, captured unencrypted SWIFT traffic for weeks, learned the exact protocol format and auth flow, then crafted fraudulent transfers stealing $81M. Wireshark didn't hack anything — it just listened. Lesson: unencrypted internal traffic + network access = full protocol knowledge without ever touching a server.

// interview answer
Q: Walk me through how you'd approach a penetration test at a high level.
"Start with Nmap — service version scan to identify what's running, vulnerability scripts to flag CVEs. Anything interesting gets Burp Suite if web-facing, or Metasploit if a network service with known exploit. Wireshark runs throughout to capture traffic — the Bangladesh Bank heist is the clearest example of why: Lazarus Group used passive Wireshark for weeks and learned an entire banking protocol just by listening to unencrypted traffic."

Cryptography I — Hashing vs Encryption
LinkedIn 2012 · SHA-1 Collision 2017
// hashing vs encryption
Hashing (One-Way)
Irreversible, deterministic, fixed-size. echo -n "hello" | sha256sum. For: integrity, password storage.
Encryption (Two-Way)
Reversible with a key. openssl enc -aes-256-cbc -in plain.txt -out enc.txt. For: confidentiality.
// hash algorithms
MD5 — 128-bit
Broken. Collisions since 2004. Never use for security.
SHA-1 — 160-bit
Deprecated. SHAttered (2017) collision attack. Git migrating away.
SHA-256 — 256-bit
Current standard. TLS certs, blockchain, password hashing (with salt).
bcrypt/scrypt/Argon2
Password hashing. Deliberately slow + memory-hard. Argon2 won 2015 PHC.
// real incident

LinkedIn (2012): 167M unsalted SHA-1 hashes leaked. 90% cracked within days using rainbow tables. Salting alone makes cracking 1000x slower per user. Lesson: never hash passwords with a fast algorithm, and never without salt.

// interview answer
Q: Why is bcrypt preferred over SHA-256 for password storage?
SHA-256 is nanoseconds-fast — GPUs compute billions/sec. bcrypt has a configurable cost factor making each hash take ~250ms. Defense-in-depth = slow + salted + memory-hard (Argon2).

Cryptography II — Symmetric vs Asymmetric
Heartbleed CVE-2014-0160
// comparison
Symmetric (AES)
One key encrypts AND decrypts. Fast, hardware-accelerated. Problem: key sharing. Bulk data encryption.
Asymmetric (RSA/ECC)
Public encrypts, private decrypts. Slow (100-1000x slower). Solves key exchange. Handshakes & signatures.
TLS Handshake
ClientHello → ServerHello + Cert → Key Exchange (ECDHE) → Session key → AES encrypted channel.
Perfect Forward Secrecy
ECDHE = unique session key per connection. Past sessions safe even if private key compromised later. RSA key exchange lacks PFS.
// real incident

Heartbleed (CVE-2014-0160): buffer over-read in OpenSSL Heartbeat leaked up to 64KB server memory per request. Attackers repeatedly read memory until they extracted private RSA keys. 500K+ servers affected. Lesson: private keys are only as safe as the software managing them.

// interview answer
Q: Why does TLS use both asymmetric and symmetric cryptography?
Asymmetric (RSA/ECDHE) solves key exchange — securely negotiates a shared secret. Symmetric (AES) provides fast bulk encryption. Hybrid = security + performance.

Cryptography III — SSL/TLS, Certificates & PKI
DigiNotar CA Breach 2011
// x.509 certificate anatomy
Subject
Who the cert belongs to (CN/SAN). Browser checks this matches the URL.
Public Key
The asymmetric key — anyone can encrypt data only the private key holder can decrypt.
Issuer
The CA that signed this cert. Browser checks if the CA is in its trusted root store.
Validity
Not Before / Not After. Expired certs rejected by modern browsers.
Signature
CA's cryptographic signature proving the certificate hasn't been tampered with.
// certificate chain hierarchy
Root CA
Self-signed, pre-installed in browsers (~150 globally). Compromised root = all certs untrusted.
Intermediate CA
Signed by root CA. Issues actual leaf certs. Limits blast radius if breached.
Server / Leaf Cert
Your browser sees this. Contains domain name, public key, and signature chain to trusted root.
// real incident

DigiNotar CA Breach (2011): attackers compromised a Dutch CA and issued 531 fraudulent certs — including *.google.com. Iranian users MITM'd for months. DigiNotar bankrupted, delisted from all major browsers. Lesson: PKI trust model is only as strong as the weakest CA — any one of ~150 trusted roots can sign a cert for any domain.

// interview answer
Q: What happens when you visit an HTTPS site with an expired certificate?
Browser checks the validity period (Not Before / Not After). If expired, cert is rejected and a "Not Secure" warning appears — even though encryption may be sound. User must manually bypass the warning. Certificate expiration is a common cause of production outages.

Authentication vs Authorization
Capital One 2019 · IDOR
// the fundamental difference
Authentication (AuthN)
"Who are you?" Passwords, MFA, biometrics, certificates, SSO.
Authorization (AuthZ)
"What can you do?" Permissions, roles, ACLs, policies. Checked AFTER auth.
// common auth failures
IDOR
Change /profile?id=123 to ?id=124 and see another user's data. AuthN fine — AuthZ never checked.
Session Hijacking
Attacker steals session cookie (XSS/sniffing). AuthN already done — cookie is the proof.
Broken Access Control
OWASP #1. Checks "logged in?" but not "allowed to do this?"
// real incident

Capital One (2019): former AWS employee exploited a WAF misconfigured to allow SSRF to EC2 metadata service. The WAF's IAM role had S3 read access to 100M+ records. AuthN was fine (WAF properly authenticated). AuthZ was broken (role had far more permissions than needed). Lesson: least privilege — a WAF should only forward traffic, not read customer data.

// interview answer
Q: What's the difference between horizontal and vertical privilege escalation?
Horizontal: accessing another user's data at same privilege level (IDOR — user A reads user B's email by changing an ID). Vertical: gaining higher privileges than intended (standard user becomes admin via JWT role manipulation). Both authorization failures; vertical is usually more dangerous.

Password Security & Hashing Attacks
RockYou 2009 — 32M Plaintext Passwords
// attack types
Dictionary Attack
Hash every word in a wordlist (rockyou.txt = 14M). Most common method — 90% of real hashes fall to this first pass.
Rule-Based Attack
Transform words — capitalize, append numbers, Leet substitution. "password" → "P@ssw0rd1!" Covers real-world patterns.
Rainbow Table Attack
Pre-computed hash→plaintext lookup. Instant for unsalted hashes. Defeated entirely by salting.
Mask / Brute Force
Define character pattern or try all combos. Guaranteed but exponentially slow for long passwords.
// cracking speeds — algorithm matters
MD5 — ~60B/sec
rockyou.txt cracks in milliseconds. Instant on modern GPU.
SHA-256 — ~10B/sec
Still very fast. Unsalted common passwords fall in seconds.
bcrypt (cost 12) — ~25K/sec
2.4M× slower than MD5. rockyou.txt takes ~9 min. The slowness IS the security.
Argon2id — memory-hard
Requires large memory per attempt. GPUs can't parallelize. Current best practice.
// real incident

RockYou (2009): SQL injection dumped 32 million accounts with passwords in plaintext — no hashing at all. The cleaned, sorted list became rockyou.txt — the wordlist shipped with every Kali Linux install. Top 10 passwords: 123456, 12345, 123456789, password, iloveyou, princess. Lesson: every Hashcat tutorial, every CTF challenge, every cracking lab uses this dataset — produced by the privacy failure of 32 million real people.

// interview answer
Q: Walk me through how you'd crack a password hash recovered from a compromised database.
"First identify the algorithm by hash format (MD5=32 hex, SHA-1=40, bcrypt=$2y$). Use hash-identifier if unsure. Then dictionary attack with rockyou.txt — 90% of MD5/SHA-1 real-world hashes crack on the first pass. If that fails, apply best64.rule for transformations, then mask attack. bcrypt cracks at ~25K/sec vs MD5's 60B/sec — that's exactly why bcrypt is correct for password storage."

Q01
What two socket types does Python support, and when would you use each in security work?
SOCK_STREAM (TCP) — reliable, ordered, connection-based. Used for web, SSH, port scanning. SOCK_DGRAM (UDP) — fast, connectionless. Used for DNS queries, speed-critical scanning.
Q02
What Event IDs cover successful logon, failed logon, process creation, and log clearing?
4624 (success), 4625 (failure), 4688 (process creation), 1102 (log cleared). 4104 = PowerShell script block logging.
Q03
What's the difference between a Type 1 and Type 2 hypervisor? Which one does your home lab use?
Type 1 (bare metal) runs directly on hardware — ESXi, Hyper-V, Proxmox. Type 2 runs on top of your OS — VirtualBox, VMware Workstation. Home lab uses Type 2.
Q04
What Nmap switches would you use for service version detection, OS detection, and an aggressive full scan?
-sV (service version), -O (OS detection), -A (aggressive — combines -sV, -O, traceroute, scripts). --script vuln for vulnerability scan.
Q05
What was the real vulnerability in Capital One — authentication or authorization? Why?
Authorization. The WAF was properly authenticated to AWS (AuthN fine). But its IAM role had S3 read access to 100M+ records — far more permission than needed (AuthZ broken).
Q06
Why is bcrypt better than SHA-256 for password storage? Use specific numbers.
SHA-256 = ~10B hashes/sec on a modern GPU. bcrypt (cost 12) = ~25K/sec — 2.4 million times slower. The speed difference is the security: what takes milliseconds for SHA-256 takes minutes for bcrypt.
Q07
What was the origin of the rockyou.txt wordlist shipped with Kali Linux?
The 2009 RockYou breach. Attackers used SQL injection to dump 32M accounts with passwords stored in plaintext — no hashing at all. The cleaned list became the standard dictionary for password cracking.
Q08
How did Mirai's socket automation technique work, and what was the real-world impact?
Mirai used threaded socket connections to scan the entire IPv4 internet for open Telnet (port 23). Default creds let it absorb IoT devices into a botnet. Result: Oct 2016 Dyn DDoS took Twitter/Netflix offline.
Q09
How did SolarWinds SUNBURST use the Windows Registry for stealth?
SUNBURST stored its configuration data and C2 server addresses in the Windows Registry instead of writing suspicious files to disk. It also mimicked legitimate SolarWinds API calls and stayed dormant up to 2 weeks.
Q10
What did Heartbleed leak from server memory, and what was the worst-case impact?
Up to 64KB of arbitrary server memory per request including private RSA keys. Worst case: attacker decrypts past TLS traffic (without PFS) and can impersonate the server going forward.
