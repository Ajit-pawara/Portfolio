Malware Analysis Static Basics

Before you run a suspicious file, you analyze it statically. Today you inspect PE headers, strings, hashes, and imports to determine malicious intent without execution.

THEORY

Static Analysis vs Dynamic Analysis: Static analysis examines a file without executing it. You inspect the binary's structure, metadata, embedded strings, imported functions, and compiled artifacts. It's safe (the malware never runs) and fast (seconds to minutes). Dynamic analysis executes the sample in an isolated sandbox and observes behavior — but it requires containment and takes longer. The rule: always start with static analysis. If you can determine the file is malware through static means, you avoid the risk of execution entirely.

The PE File Format: Most Windows malware ships as PE (Portable Executable) files — .exe, .dll, .sys. The PE structure contains:

DOS Header
Starts with "MZ" (4D 5A). Contains e_lfanew offset pointing to the PE header.
PE Header
Signature "PE\0\0" followed by File Header (machine type, sections, timestamp) and Optional Header.
Section Table
Lists .text (code), .data (global vars), .rdata (read-only), .rsrc (resources). Section names and entropy flags are key indicators.
Import Table
Lists DLLs and functions the binary imports. VirtualAlloc, WriteProcessMemory, CreateRemoteThread = process injection.

Key static analysis signals:

# 1. File Hash — check against VirusTotal / known malware databases sha256sum sample.exe d4e5f6a7b8c9... sample.exe # 2. Strings analysis — extract embedded text strings -n 8 sample.exe | head -30 http://malicious-c2.com/beacon C:\Windows\System32\cmd.exe Mozilla/5.0 (Windows NT 10.0; Win64; x64) HKEY_LOCAL_MACHINE\SOFTWARE\Microsoft\Windows\CurrentVersion\Run # 3. Import analysis — what APIs does it call? # Suspicious imports: VirtualAlloc, WriteProcessMemory, CreateRemoteThread, # CryptDecrypt, URLDownloadToFile, WinExec, RegSetValueEx

Entropy Analysis: Entropy measures randomness in a file section. Normal code has entropy around 5.0-6.5. Packed or encrypted sections have entropy >7.0 (near maximum 8.0). High entropy in .text section means the code is packed/obfuscated — a strong indicator of malware. Tools like Detect It Easy (DIE) can identify the specific packer (UPX, Themida, VMProtect) from signatures.

Key insight: Static analysis alone can classify 70-80% of malware samples. If the imports include process injection APIs, the strings contain C2 URLs, and the entropy is suspicious, you have enough evidence to block the file without ever running it. This is the first filter in every malware analysis pipeline.
// attacker vs defender mindset
ATTACKER
  • Pack the malware with custom packers (not UPX — AV signatures detect UPX) to increase entropy and evade static signature detection entirely.
  • Obfuscate strings at rest — encrypt them with XOR or RC4 and decrypt only at runtime. Static string analysis can't find encrypted strings.
  • Use API hashing — instead of importing VirtualAlloc by name, compute a hash and resolve the API at runtime by walking the PEB. The import table looks clean.
  • Patch the PE timestamp to a false value — set it to a date when the compiler was legitimate (e.g., 2019) to evade time-based heuristics.
DEFENDER
  • Always compute the SHA256 hash first and check VirusTotal — if the hash is known, you have an immediate classification without deeper analysis.
  • Run strings with different minimum lengths — short strings (4 chars) catch commands, longer strings (15+) catch URLs and registry paths.
  • Check the import table against known malicious API patterns: process injection (VirtualAlloc, WriteProcessMemory), persistence (RegSetValueEx), C2 (InternetOpen, WinHttpOpen).
  • Use Detect It Easy to identify packers — an unknown packer with high entropy is suspicious. Flag anything >7.0 entropy for dynamic analysis.
// real named incident
2010 — STUXNET — 5 COUNTRIES, 1,000+ CENTRIFUGES DESTROYED
Static analysis first identified Stuxnet as a sophisticated cyberweapon through PE header and certificate analysis
In June 2010, VirusTotal received a sample from Belarus that raised immediate flags during static analysis. The PE file contained two stolen digital certificates (JMicron and RealTek), multiple zero-day exploits (including the LNK vulnerability CVE-2010-2568 and the Print Spooler vulnerability CVE-2010-2729), and Siemens SCADA-specific strings referencing Step 7 software. The import table showed unusual API combinations for a standard driver — including functions to manipulate PLC (Programmable Logic Controller) frequency converters. The PE timestamp was set to January 2010, but the compiled code referenced dates after the certificate expiration, a clear sign of timestamp forgery. These static indicators — stolen certs, zero-day exploits, SCADA-specific strings — told researchers this was not ordinary malware before anyone had seen it run.
HANDS-ON LAB

Goal: perform static analysis on a safe test file using command-line tools and Detect It Easy to identify suspicious PE characteristics.

  1. Create a harmless PE file for analysis: cp /bin/ls /tmp/analysis/benign.exe — this is a legitimate Linux binary. For Windows PE analysis, download a known-safe PE (e.g., notepad.exe from a Windows VM) or use a sample from github.com/ytisf/theZoo (password-protected malware zoo for educational use).
  2. Compute file hashes: sha256sum /tmp/analysis/sample.exe > sample.hash; md5sum /tmp/analysis/sample.exe >> sample.hash.
  3. Extract strings: strings /tmp/analysis/sample.exe | sort | uniq > sample_strings.txt. Inspect for URLs, file paths, registry keys, and API calls.
  4. Analyze PE headers using readpe (from pev package): readpe /tmp/analysis/sample.exe | head -50 to see DOS header, PE header, sections, and imports.
  5. Install and run Detect It Easy (DIE): diec /tmp/analysis/sample.exe to detect packers, entropy, and compiler info.
  6. Write a one-page analysis report documenting: hash, suspicious strings identified, imported APIs flagged, entropy score, and your verdict (malicious/benign/suspicious).
Deliverable: a static analysis report (one page) with the hash, strings output snippet, import analysis, entropy assessment, and final verdict. This is the exact format used in professional malware analysis workflows.

INTERVIEW READY
Q: "Walk me through how you'd statically analyze a suspicious EXE file."
A: "First, compute the SHA256 hash and check VirusTotal for known classifications. Second, extract strings with a minimum length of 8 — look for URLs, IP addresses, file paths, registry keys, and API names. Strings like 'CreateRemoteThread' or 'http://' in a calculator app are red flags. Third, analyze the PE headers — check the import table for suspicious API combinations: VirtualAlloc + WriteProcessMemory + CreateRemoteThread is a classic process injection pattern. Fourth, check entropy per section — .text entropy above 7.0 suggests packing. Use Detect It Easy to identify the packer. Finally, check the compile timestamp — if it claims to be from 2019 but the file was discovered last week, it's likely forged. Document everything in a one-page report."
Q: "What strings would make you immediately suspicious in a static analysis?"
A: "C2 URLs or IP addresses that resolve to known malicious hosts, registry run keys (Run, RunOnce) that suggest persistence, PowerShell or cmd.exe invocation for command execution, CreateRemoteThread for process injection, VirtualAlloc for memory allocation in another process, InternetOpen/URLDownloadToFile for download capabilities, and any reference to disabling security products (net stop, sc stop, Set-MpPreference). Also suspicious: hardcoded encryption keys, mutex names (often unique per malware family), and file paths in temporary directories."
Q: "What is entropy and how is it used in malware analysis?"
A: "Entropy measures the randomness of data on a scale of 0 to 8. Normal compiled code has entropy between 5.0 and 6.5. Packed or encrypted binaries have entropy above 7.0 because compressed or encrypted data appears more random. A .text section with entropy of 7.8 strongly indicates the code has been packed to evade signature detection. We use tools like Detect It Easy to identify which packer was used (UPX, Themida, VMProtect). High entropy alone isn't proof of malware — some legitimate software uses packing — but combined with other indicators like suspicious imports and unknown strings, it's a strong signal."
AI TOOLS
String analyzer
Paste extracted strings into an LLM and ask it to classify each: C2 indicators, persistence mechanisms, anti-analysis techniques, encryption routines.
API import explainer
Paste the import table (DLL + function names) and ask the LLM to explain what the binary is capable of based on its API combination.
YARA rule writer
Describe the malware's static characteristics (sections, strings, imports) and get a candidate YARA rule to detect similar samples.
VirusTotal automation
Use the VirusTotal API (free) to automate hash lookups — write a Python script that submits hashes and returns detection ratios.
