DAY 49 OF 90 — PHASE 3: WEB & NETWORK EXPLOITATION

Post-Exploitation & Pivoting Basics

You're inside the first system. The real target is behind another firewall. Today you use Meterpreter, SSH tunneling, and SOCKS proxying to move laterally.

THEORY

Post-exploitation defined: You have a shell. Now what? Post-exploitation is everything that happens after initial access — privilege escalation, lateral movement, persistence, data exfiltration, and covering tracks. The most critical skill is pivoting: using the compromised host as a relay point to reach networks that were previously inaccessible. If the web server is the only system exposed to the internet, and the database server is on an internal 10.x.x.x network, the only way to reach the database is through the web server.

Three tunneling techniques:

# SSH Local Port Forward — expose remote service on your local port ssh -L 8080:10.0.0.5:80 user@compromised-host # Now http://localhost:8080 reaches 10.0.0.5:80 through the tunnel # SSH Dynamic Port Forward — SOCKS proxy through the target ssh -D 9050 user@compromised-host # Configure proxychains to use localhost:9050 proxychains nmap -sT 10.0.0.0/24 # Meterpreter pivot — autoroute adds routes through the compromised host meterpreter > run autoroute -s 10.0.0.0/24 meterpreter > run auxiliary/server/socks_proxy

The lateral movement mindset: Every compromised host is a stepping stone. You're not looking for a single straight path to the domain controller — you're mapping the network graph, collecting credentials at each hop, and using the accumulated access to reach deeper. On Windows, use Mimikatz to dump LSASS credentials from memory. On Linux, check .ssh/authorized_keys, ~/.bash_history for passwords, and /etc/shadow. Each new credential set expands your reachable network. The goal is to reach the "crown jewel" — domain controller, database server, source code repository, or backup server — while maintaining access to every system along the path.

Key insight: Pivoting is not about exploiting new vulnerabilities. It's about using the trust relationships and network access you already have. Once you're inside the perimeter, the internal network typically has minimal segmentation and monitoring. Most breaches that make headlines involved an attacker sitting on a single low-privilege box for weeks, collecting credentials, and slowly mapping the internal network before striking.
// attacker vs defender mindset
⚔ Attacker
  • After gaining initial access, immediately run local enumeration (users, network interfaces, routing tables, listening ports) to understand where you are in the network topology.
  • Dump credentials from memory first — Mimikatz on Windows, /etc/shadow and bash history on Linux. Credential reuse is the #1 lateral movement enabler.
  • Set up multiple egress paths — if SSH outbound is blocked, try HTTP/HTTPS tunnels (Chisel, Ligolo-ng) or DNS tunneling (dnscat2). Always have a backup C2 channel.
  • Use proxychains with the SOCKS proxy to run tools (nmap, rdesktop, sqlcmd) through the compromised host as if you were on the internal network.
🛡 Defender
  • Implement network micro-segmentation — each tier (web, app, db) should be in separate VLANs with firewall rules that only allow specific traffic between specific ports.
  • Use jump boxes / bastion hosts for administrative access — no direct SSH/RDP to internal servers. Log and audit all jump box sessions.
  • Deploy EDR on all endpoints with lateral movement detection — unexpected RDP, SMB, or WinRM connections between hosts are the primary signal of an active breach.
  • Rotate local administrator passwords frequently using LAPS (Windows) or similar tools — this limits credential reuse after LSASS dump.
// real named incident
2021 — COLONIAL PIPELINE — DARKSIDE RANSOMWARE
Attackers used compromised VPN credentials to pivot from the IT network to operational technology (OT) systems
The DarkSide ransomware group gained initial access through a compromised VPN account that was no longer in active use but still had access to the corporate IT network. From there, they pivoted laterally across the IT network for weeks, collecting credentials and mapping internal systems. The critical pivot was from IT to OT — the pipeline's operational technology network that controlled the actual fuel flow. The attackers moved through a poorly segmented network where the IT and OT environments shared authentication infrastructure. When they deployed ransomware on the OT systems, Colonial Pipeline was forced to shut down the entire pipeline — 5,500 miles of fuel transport serving the US East Coast — for six days. The company paid a $4.4M ransom. The FBI later recovered approximately $2.3M of the payment. The root cause wasn't a sophisticated zero-day; it was a single compromised VPN credential and insufficient network segmentation between IT and OT.
HANDS-ON LAB

Goal: establish a foothold on a compromised host, set up a SOCKS proxy, and use it to scan and access an internal network segment.

  1. Set up the lab: deploy two Docker containers — one "public" (exposed on port 22) and one "internal" (only reachable from the first). Use Docker networking with a custom bridge.
  2. SSH into the public container from your Kali machine. Run ip route and ip addr to discover the internal network subnet (e.g., 172.17.0.0/16).
  3. Set up SSH dynamic port forwarding: ssh -D 9050 user@public-container -N. Test it: curl --socks5 localhost:9050 http://internal-container.
  4. Configure proxychains: edit /etc/proxychains4.conf to add socks5 127.0.0.1 9050. Run: proxychains nmap -sT -Pn 172.17.0.0/24.
  5. Use Meterpreter (Metasploit) if available: generate a payload with msfvenom, get a reverse shell, then use the autoroute module and socks_proxy auxiliary module.
  6. Document every route, port forward, and discovery command. The deliverable is a network map showing the target topology and the pivot path.
Deliverable: A network topology map showing the relationship between the public host, internal hosts, and the pivot path. Include the commands used for SSH tunneling and proxychains scanning.
# 1 — SSH dynamic port forwarding (SOCKS proxy) ssh -D 9050 user@public-container -N # -D = dynamic port forward (SOCKS), -N = no remote command # 2 — Configure proxychains echo 'socks5 127.0.0.1 9050' >> /etc/proxychains4.conf # 3 — Scan internal network through proxy proxychains nmap -sT -Pn 172.17.0.0/24 |S-chain|-<>-127.0.0.1:9050-<>-<>-172.17.0.2:22-open # 4 — SSH local port forward (expose internal service locally) ssh -L 8080:172.17.0.2:80 user@public-container -N # Access internal web app: http://localhost:8080 # 5 — Meterpreter route + socks proxy meterpreter > run autoroute -s 172.17.0.0/24 meterpreter > run auxiliary/server/socks_proxy # Then proxychains to route through MSF # 6 — Verify tunnel proxychains curl http://172.17.0.2/ <html>Internal Web App</html>
INTERVIEW READY
Q: "Explain the difference between SSH local port forwarding, remote port forwarding, and dynamic port forwarding."
A: "Local port forwarding (-L) forwards traffic from your local machine through the SSH connection to a destination reachable from the remote host. Remote port forwarding (-R) does the reverse — forwards traffic from the remote host back to a destination reachable from your local machine. Dynamic port forwarding (-D) creates a SOCKS proxy on your local machine, allowing any application configured to use the proxy to route traffic through the remote host to any destination."
Q: "What is a pivot and when would you need one in a penetration test?"
A: "A pivot is using a compromised host as a relay to reach otherwise inaccessible network segments. I need a pivot when the target has network segmentation — for example, the web server is in a DMZ accessible from the internet, but the database server is on an internal 10.x.x.x network with no direct internet route. I compromise the web server first, then use it as a pivot to reach the database server. Without pivoting, I'd never reach the database."
Q: "How do you maintain access while pivoting? What happens if the pivot host goes down?"
A: "I always establish redundant access. Multiple C2 callbacks through different egress channels (HTTP, DNS, ICMP) from the pivot host. For the downstream targets I reach through pivoting, I deploy separate persistence mechanisms so I don't lose access if the pivot fails. I also document the full access chain so if one link breaks, I can rebuild it quickly. In real engagements, I never rely on a single pivot point — I always try to establish at least two independent paths into the target environment."

Proxychains config assistant
Ask which proxychains mode (dynamic, strict, random) to use for your specific scenario — some tools need strict, others work with dynamic.
Meterpreter post-exploitation planner
Describe your post-exploitation goals and the LLM generates a Meterpreter module execution plan with the correct commands in order.
C2 egress strategist
Ask which C2 framework and egress technique works best for your scenario — HTTP(S), DNS, ICMP, or domain fronting — based on the target's filtering.
