Malware Analysis Dynamic / Sandbox

Static analysis tells you what a file looks like. Dynamic analysis shows you what it actually does. Today you run malware in an isolated sandbox and observe behavior.

THEORY

Dynamic Analysis Fundamentals: Dynamic analysis executes a suspicious sample in a controlled, isolated environment and observes its runtime behavior. This reveals what static analysis cannot: decrypted strings, network connections, file modifications, registry changes, process injection, and anti-analysis techniques. The key requirement is complete isolation — the sandbox must have no network access to production systems and must be restored to a clean state after each analysis.

What sandboxes monitor:

Process Creation
Track every new process and its parent. Droppers often spawn cmd.exe, powershell.exe, or wscript.exe.
File System
Monitor file creation, modification, deletion. Look for files written to %TEMP%, %APPDATA%, or SYSTEM32.
Registry
Observe Run keys, service creation, COM object registration. Persistence via HKCU\Software\Microsoft\Windows\CurrentVersion\Run.
Network
Capture DNS queries, HTTP requests, TCP connections. C2 beaconing patterns, data exfiltration, and download payloads.

Sandbox evasion techniques malware uses: Professional malware actively detects sandbox environments. Common evasions: checking for small display resolution (1024x768 is a default VM setting), checking CPU core count (sandboxes typically have 1-2 cores), checking for analysis tools (procmon.exe, wireshark.exe, tcpview.exe in the process list), checking for virtual machine artifacts (MAC addresses starting with 00:0C:29 for VMware, 00:15:5D for Hyper-V), and using long sleep delays (5-30 minutes) before executing malicious behavior — sandboxes typically run for 5 minutes.

# Common sandbox detection techniques (in pseudocode) if screen_resolution == "1024x768": exit() # VM default resolution if num_cpus < 2: exit() # Sandbox has single CPU if vm_mac_prefix == "00:0C:29": exit() # VMware MAC address if sleep(300) == instant: exit() # Sleep patched by sandbox # What a real C2 beacon looks like in sandbox network logs GET /beacon/win10-01 HTTP/1.1 Host: 185.130.5.193 User-Agent: Mozilla/5.0 (Windows NT 10.0; Win64; x64) Cookie: session=base64_encoded_computer_fingerprint
Why this matters: Dynamic analysis reveals the actual behavior that static analysis can only guess at. A file with no suspicious strings may decrypt them at runtime. A binary with no import table may resolve APIs dynamically. The sandbox captures all of this — making it the definitive analysis step before classification.
// attacker vs defender mindset
ATTACKER
  • Implement sandbox detection checks before executing the payload — if any VM artifact is detected, sleep benignly or crash to avoid revealing true behavior.
  • Use sleep delays of 10-30 minutes before executing malicious behavior — most automated sandboxes only run for 5 minutes and will miss the C2 beacon.
  • Check for common analysis tools running: process names like procmon.exe, procExplorer.exe, wireshark.exe, and tcpview.exe indicate a sandbox/analyst environment.
  • Encrypt C2 traffic with custom protocols instead of plain HTTP — sandboxes can capture HTTP but not custom TCP encrypted streams without deep packet inspection.
DEFENDER
  • Use a sandbox that patches common evasion techniques — tools like Cuckoo 3 and CAPE can patch sleep calls, set fake resolutions, and hide analysis process names from the sample.
  • Run samples for at least 10 minutes with network simulation (INetSim) that emulates real services — DNS, HTTP, HTTPS — so the sample believes it has outbound connectivity.
  • Compare sandbox network captures against known C2 threat intelligence feeds — a connection to a known malicious IP confirms the sample is malware.
  • Always restore the sandbox VM to a clean snapshot after each analysis — residual artifacts from one sample can interfere with the next analysis.
// real named incident
2014 — DRIDEX BANKING TROJAN — $100M+ LOSSES
European banking security teams used Cuckoo sandbox to analyze Dridex samples, identifying C2 patterns across 300+ bank compromises
The Dridex banking trojan (evolved from GameOver Zeus) infected over 300 financial institutions globally. European banking CERTs deployed Cuckoo sandbox clusters to analyze thousands of Dridex samples daily. Dynamic analysis revealed the malware's full behavior chain: it injected into browser processes (chrome.exe, firefox.exe) via process hollowing to capture online banking credentials, created scheduled tasks for persistence, and used a peer-to-peer C2 infrastructure with encrypted communication. Sandbox network capture identified specific C2 patterns — beacon intervals of exactly 60 seconds, HTTP POST to /gate.php with RC4-encrypted body, and User-Agent strings matching specific Mozilla versions. These C2 signatures were shared via MISP across the banking sector, enabling proactive blocklisting before banks were targeted.
HANDS-ON LAB

Goal: submit a safe test file to an online sandbox (ANY.RUN) and a local sandbox, then analyze the behavioral report.

  1. Open any.run in your browser (free tier available — register with email). This is a cloud-based interactive sandbox that streams the VM desktop to your browser.
  2. Download a known-safe test file for analysis — use EICAR test file (echo 'X5O!P%@AP[4\PZX54(P^)7CC)7}$EICAR-STANDARD-ANTIVIRUS-TEST-FILE!$H+H*' > test.txt).
  3. Upload the file to ANY.RUN, select Windows 10 as the environment, and click "Run." Watch the VM desktop in real-time as the sample executes.
  4. After execution, examine the report: Process Tree (which processes were spawned?), Network tab (any DNS/HTTP requests?), File System (files created/modified?), Registry (keys accessed?).
  5. For local sandbox testing, set up a simple monitoring environment using Procmon on Windows: run Procmon with a filter for the test file's process, capture all operations for 60 seconds.
  6. Create a summary report: what did the sample do? Did it create files? Connect to network addresses? Modify the registry? The EICAR test should be detected and blocked by Windows Defender — note that behavior.
Deliverable: a screenshot of the ANY.RUN analysis report showing process tree, network connections, and file system changes for your test sample. This proves you can execute and interpret dynamic analysis results.

INTERVIEW READY
Q: "Walk me through how you'd dynamically analyze a suspicious executable."
A: "After static analysis, I upload the sample to an interactive sandbox like ANY.RUN or run it in an isolated VM with monitoring tools. I observe: process creation — does it spawn child processes like cmd.exe or powershell.exe? File system — does it write files to %TEMP% or %APPDATA%? Registry — does it add Run keys for persistence? Network — does it make DNS queries or HTTP requests to known or suspicious IPs? I let the sample run for at least 5-10 minutes. After execution, I analyze the behavioral report — the process tree, network captures, file modifications, and registry changes tell the complete story of what the malware does."
Q: "How do malware authors evade automated sandbox analysis?"
A: "Common techniques include: checking for VM artifacts like MAC address prefixes (00:0C:29 for VMware), small screen resolution (1024x768), or low CPU core count. Long sleep delays of 10-30 minutes bypass short sandbox runtimes. Checking for analysis tools in the process list (procmon.exe, wireshark.exe). Using sleep skipping detection — if sleep(300) returns instantly instead of waiting, the sandbox has patched the sleep function. Some malware checks for user mouse movements or keyboard input, which don't exist in automated sandboxes."
Q: "What's the difference between Cuckoo and ANY.RUN for dynamic analysis?"
A: "Cuckoo is an open-source sandbox that you host yourself — it uses a custom agent running on the guest VM and communicates via HTTP. It's free but requires significant setup (VMs, networking, dependencies). ANY.RUN is a commercial cloud sandbox with a free tier — it provides real-time desktop streaming, interactive analysis, and detailed reports without setup. Cuckoo gives you more control and configuration but requires infrastructure. ANY.RUN is easier to start with for quick analysis. For a professional setting, many organizations use both: Cuckoo for bulk automated analysis and ANY.RUN for deep manual analysis."

CAPE sandbox (open-source)
Cuckoo-based open-source sandbox with enhanced VM evasion detection and config dump extraction. GitHub: kevoreilly/CAPEv2.
Dynamic report interpreter
Paste a sandbox JSON report into an LLM and ask it to summarize the malware's behavior, extract IoCs, and suggest detection rules.
INetSim network simulator
Open-source service simulator for sandbox environments. Emulates DNS, HTTP, HTTPS, SMTP, and other services so malware thinks it has internet access.
