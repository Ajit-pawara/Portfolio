Intro to SIEM — Splunk / ELK Basics

Security Information and Event Management (SIEM) aggregates logs from every corner of your infrastructure and makes them searchable at scale. Today you learn the architecture of Splunk and the ELK stack, write SPL and KQL queries, and understand how SIEM drove the SolarWinds investigation.

THEORY

What is SIEM? SIEM (Security Information and Event Management) is a centralized platform that collects, normalizes, and analyzes log data from across an enterprise — servers, firewalls, endpoints, cloud services, applications. It correlates events from disparate sources to detect patterns that no single log source would reveal.

Splunk architecture:

# Core Splunk components: Forwarder → collects logs from sources (agents on endpoints, syslog, API) Indexer → ingests, parses, and stores data in indexed buckets Search Head → provides the search interface (web UI / REST API) across indexers # Splunk Search Processing Language (SPL): source="/var/log/auth.log" | stats count by src_ip | where count > 10 index=windows EventCode=4625 | top 10 Account_Name # SPL pipeline: search → filter → transform → aggregate → output

ELK stack architecture:

# Elasticsearch + Logstash + Kibana (ELK): Logstash → ingests, transforms, and ships logs (input → filter → output) Elasticsearch → distributed search and analytics engine (stores & indexes logs) Kibana → visualization and dashboard layer (charts, maps, alerts) # Common ingestion pipeline: Beats (Filebeat/Winlogbeat) → Logstash → Elasticsearch → Kibana # KQL (Kibana Query Language) example: event.dataset: "system.auth" and event.outcome: "failure" NOT source.ip: 10.0.0.0/8 and event.action: "user_login"

SIEM data sources: Windows Event Logs (Security, System, Application), Linux syslog (auth.log, messages), firewall logs (iptables, pfSense), web server access logs (Apache, Nginx), cloud trail logs (AWS CloudTrail, Azure Monitor), endpoint detection (EDR telemetry).

Key insight: SIEMs don't stop attacks — they detect them. The value comes from the queries and correlation rules you write. A misconfigured SIEM with no tuned detections is an expensive log storage system. Splunk is the industry leader but expensive; the ELK stack is open-source and widely adopted in startups and mid-market.
// analyst vs engineer mindset
⚔ Analyst
  • Write time-bounded searches: earliest=-7d latest=now — always scope your query to avoid scanning the entire index.
  • Chain SPL commands with the pipe operator: search index=* | where status=failed | stats count by user — each pipe filters or transforms.
  • Use top and rare to find outliers: top src_ip shows frequent sources; rare src_ip shows the one-hit wonders that might be scanning.
  • Bookmarked dashboards for common incident types (brute force, malware beacon, phishing) save hours during an active investigation.
🛡 Engineer
  • Deploy forwarders with inputs.conf — configure which log files to monitor, what sourcetype to assign, and any filtering at the source.
  • Set index retention policies — hot (SSD, 7d), warm (HDD, 30d), cold (archive, 1y), frozen (deleted). Storage is the #1 cost in Splunk.
  • Use Logstash grok filters to parse unstructured logs into structured fields — a poorly parsed log is almost useless for correlation.
  • Deploy Filebeat on all Linux endpoints for a zero-agent-cost ELK pipeline. Centralised logging must never rely on agents staying healthy — build redundancy.
// real named incident
2020 — SOLARWINDS ORION SUPPLY-CHAIN ATTACK — 18,000 ORGANIZATIONS
FireEye's Splunk team ran 'search index=* ImagePath=*solws*' and found the backdoor in minutes
In December 2020, attackers (widely attributed to APT29 / Cozy Bear) compromised SolarWinds' build pipeline and inserted a backdoor called SUNBURST into Orion software updates. The backdoor was distributed to 18,000 organizations. FireEye's internal IR team used Splunk to search across their entire log corpus with a single SPL query: search index=* ImagePath=*solws* — this identified every host running the compromised Orion agent. The breadth of the search — across all indexes, on every endpoint — highlighted the power of a well-maintained SIEM. Without centralised logging, detecting the scale of the compromise would have taken weeks instead of hours. FireEye then pivoted to DNS logs and found beaconing to attacker C2 infrastructure, enabling them to identify and notify affected customers before the backdoor was publicly disclosed.
HANDS-ON LAB

Goal: deploy the ELK stack locally, ingest sample syslog data, and write KQL queries to detect failed authentications.

  1. Run ELK via Docker: docker run -d -p 5601:5601 -p 9200:9200 -p 5044:5044 --name elk sebp/elk. Wait ~2 minutes for all services to start.
  2. Access Kibana at http://localhost:5601. Create the default index pattern (logstash-*) under Stack Management → Index Patterns.
  3. Generate sample syslog data: logger -n localhost -P 5044 "Failed password for root from 192.168.1.100 port 22 ssh2". Repeat with varied IPs and usernames.
  4. In Kibana Discover, write KQL queries: message: "Failed password" and message: "Failed password" and NOT source.ip: 10.0.0.*.
  5. Create a dashboard: add a bar chart of failed logins by source IP over the last 15 minutes. Save as "Failed Auth Monitor."
  6. Delete the index (DELETE /logstash-* via Dev Tools) and confirm the data disappears — SIEM data lifecycle management.
Deliverable: A Kibana screenshot showing your "Failed Auth Monitor" dashboard with at least three distinct source IPs. Document each KQL query and what it filters for. This is the core analyst workflow — build a query, visualize it, save it as a reusable dashboard.

INTERVIEW READY
Q: "What's the difference between Splunk and the ELK stack?"
A: "Splunk is a commercial SIEM with a powerful search language (SPL), enterprise features like RBAC and data replication built in, and higher per-GB licensing costs. ELK is open-source — Elasticsearch for storage and search, Logstash for ingestion and parsing, Kibana for dashboards. ELK requires more hands-on configuration but is free at scale. Splunk's strength is its ecosystem and ease of use; ELK's strength is its flexibility and cost."
Q: "How do you build a SIEM detection rule for brute-force SSH attempts?"
A: "I'd search for Failed password events grouped by source IP and destination IP over a 10-minute window. In SPL: index=linux source="/var/log/auth.log" "Failed password" | bucket _time span=5m | stats dc(source_ip) as unique_ips by _time, dest_ip | where unique_ips > 5. The threshold depends on the baseline — for a web server, 5 unique IPs in 5 minutes might be normal. For an internal jump box, 2 is suspicious. I'd also exclude known admin subnets."
Q: "What is a SIEM correlation rule and give an example?"
A: "A correlation rule combines events from different sources to identify a multi-stage attack. Example: correlate a Windows Event ID 4625 (failed logon) followed shortly by 4624 (successful logon) from the same IP — this indicates a successful brute force. The rule would fire an alert when both events occur from the same source IP to the same target within 60 seconds. This is more meaningful than either event alone."
AI TOOLS
SPL query generator
Describe your detection need in plain English — "show me failed logins grouped by user over the last hour" — and get a ready-to-run SPL query with time range and stats commands.
Grok pattern helper
Paste a raw log line into an LLM and ask it to generate the Logstash grok filter pattern. For unstructured logs (firewalls, custom apps), this saves 30 minutes of manual regex construction.
ELK deployment debugger
Describe your ELK error — "Kibana says 'no alive nodes'," "Logstash won't start," "Elasticsearch cluster health is yellow" — and get targeted diagnostic commands and config fixes.
KQL translator
Ask an LLM to translate between SPL and KQL: "Give me the KQL equivalent of index=* | stats count by sourcetype" — useful when switching between Splunk and Kibana environments.

ENTREPRENEUR ANGLE

Every mid-sized business that handles PCI-DSS, HIPAA, or SOC 2 data needs a SIEM — but most can't afford a dedicated Splunk engineer at $150k+/year. The opportunity: "ELK-as-a-Service" for small businesses. Deploy a managed ELK stack for $500-1,000/month that includes pre-built dashboards for compliance requirements (failed logins, admin access, firewall changes).

Most MSPs (Managed Service Providers) charge $50-100/endpoint/month for SIEM. With ELK, your cost is nearly zero after the initial setup. If you onboard five small clinics that each need HIPAA log monitoring at $800/month, that's $48k/year in recurring revenue with minimal infrastructure cost. The differentiation: you send them a weekly compliance summary generated from their Kibana dashboards.

TEACH IT BACK

Teach someone in 90 seconds: "What's the difference between a SIEM forwarder, indexer, and search head?" Draw the data flow: log source → forwarder (agent) → indexer (storage + parsing) → search head (query interface). Then explain how SPL pipes work like shell pipes — each stage passes the results to the next. If you can draw the ELK pipeline (Beats → Logstash → Elasticsearch → Kibana) on a whiteboard and explain what each component does, you've internalised today's material.
