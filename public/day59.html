Writing Detection Queries

A good SIEM is only as good as its queries. Today you write Sigma rules, Splunk SPL, and KQL queries to detect real attack patterns.

THEORY

Sigma — the universal detection rule format: Sigma is a generic signature format for log events, analogous to YARA for files. A Sigma rule describes a detection pattern, and a converter transforms it into SIEM-specific queries — Splunk SPL, Elasticsearch KQL, QRadar AQL, Azure Sentinel KQL, or plain SQL. This means you write one detection rule and deploy it across any SIEM backend. Sigma rules are YAML files with fields: title, id, status, description, logsource (product, service, category), detection (keywords, selections, condition), and falsepositives.

Example Sigma rule — detecting mimikatz usage:

title: Mimikatz Command Line Detection logsource: product: windows category: process_creation detection: selection: CommandLine|contains: - 'sekurlsa::logonpasswords' - 'privilege::debug' condition: selection falsepositives: - Legitimate admin tool usage (rare) level: high

Writing effective Splunk SPL: Splunk Search Processing Language (SPL) chains commands with pipes. Key SPL commands: search, stats, dedup, eval, rex, table, where, timechart. Example: index=main sourcetype=WinEventLog:Security EventCode=4625 | stats count by Account_Name, Source_Network_Address | where count > 10 | sort - count — finds accounts with more than 10 failed logon attempts.

// attacker vs defender mindset
⚔ Attacker
  • Study public Sigma rules to identify gaps in detection coverage — no rule covers every variant of a technique.
  • Test detection rules before deployment — run your attack tools against environments with known rules and measure what triggers.
  • Use living-off-the-land techniques that don't match common detection patterns — certutil.exe for downloads instead of dedicated tools.
  • Chain multiple low-severity events that individually don't meet Sigma thresholds but collectively achieve the objective.
⛑ Defender
  • Layer detection rules — write one broad rule for coverage and one strict rule for precision. Broad captures everything, strict confirms real incidents.
  • Test every rule against known-good traffic and known-bad traffic before deployment — use a PCAP corpus of both benign and malicious traffic.
  • Version-control Sigma rules in a Git repository with peer review — rule changes introduce bypasses.
  • Map each rule to a MITRE ATT&CK technique ID — enables threat-informed defense and gap analysis.

NATO COOPERATIVE CYBER DEFENCE CENTRE — SIGMA STANDARD
NATO CCDCOE uses Sigma as the standard detection rule format across heterogeneous SIEM deployments
The NATO Cooperative Cyber Defence Centre of Excellence (CCDCOE) in Tallinn, Estonia, adopted Sigma as its standard detection rule format to enable consistent threat detection across its 32 member nations' diverse SIEM environments. A single Sigma rule authored at the CCDCOE is converted to Splunk SPL, Elasticsearch KQL, QRadar AQL, ArcSight EDL, and Azure Sentinel queries — ensuring every member state detects the same threat regardless of their chosen SIEM platform. This cross-platform portability is why Sigma has become the de facto standard for detection engineering in both military and civilian SOCs worldwide.
HANDS-ON LAB

Goal: write a Sigma rule for a real attack pattern and convert it to Splunk/Elasticsearch query formats.

  1. Create a Sigma rule file named detect_schtasks_persistence.yml. Use the Sigma template: title, id, logsource, detection, falsepositives, level.
  2. Write a detection for suspicious scheduled task creation: detect command lines containing 'schtasks' with 'Create', 'RU SYSTEM', and base64-encoded arguments.
  3. Install the Sigma converter: pip install sigmatools. Convert your rule to Splunk SPL: sigmac -t splunk detect_schtasks_persistence.yml.
  4. Convert the same rule to Elasticsearch KQL: sigmac -t es-qs detect_schtasks_persistence.yml.
  5. Write a second Sigma rule for detecting mimikatz — match on CommandLine containing 'sekurlsa::logonpasswords' or 'privilege::debug'.
  6. Test both rules by creating a test EVTX file and verifying the converted queries would match.
Deliverable: two Sigma rule YAML files, their Splunk SPL output, their Elasticsearch KQL output, and a brief explanation of how each rule detects the targeted technique.

INTERVIEW READY
Q: "What is Sigma and why is it important for detection engineering?"
A: "Sigma is an open-source, generic signature format for log events. You write a detection rule once in Sigma YAML format and convert it to any SIEM query language — Splunk SPL, Elasticsearch KQL, QRadar AQL, Azure Sentinel, or plain SQL. Its importance is cross-platform portability: a SOC team can write one rule and deploy it across heterogeneous environments. It also enables threat intelligence sharing — the community shares Sigma rules the way the antivirus industry shares YARA rules."
Q: "How do you write an effective detection rule without causing alert fatigue?"
A: "Start with a specific, narrow rule that covers the known technique with minimal false positives. Example: detecting mimikatz — match on 'sekurlsa::logonpasswords' in the command line plus 'mimikatz' in the process name. Test against known benign traffic. If false positives occur, add exclusions — filter out specific admin accounts or known admin workstations. Then create a broader, lower-severity rule that catches variants the strict rule misses. Layer strict and broad rules."
Q: "How do you convert a Sigma rule into a working Splunk query?"
A: "Use the sigmac converter: sigmac -t splunk myrule.yml. The converter maps the Sigma logsource to the Splunk source type, translates the detection keywords into search terms joined by OR/AND, and outputs a valid SPL query. For example, a Sigma rule with CommandLine|contains: 'mimikatz' becomes CommandLine="*mimikatz*" in Splunk SPL. You can also hand-convert by understanding the mappings."
