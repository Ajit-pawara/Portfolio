Cloud Security Basics AWS/Azure Shared Model

The cloud is someone else's computer, and you're responsible for what runs on it.

THEORY

The shared responsibility model: In traditional on-premises security, you own everything — the building, the servers, the network, the data. In the cloud, security is split between you and the provider. AWS and Azure secure the infrastructure (hypervisors, hardware, network fabric). You secure everything inside your accounts: IAM users and roles, network ACLs, data encryption, OS patching on EC2, S3 bucket policies, and application-layer security. The line shifts based on service type — IaaS gives you more control (and more responsibility), while SaaS offloads most of it to the provider.

The three most dangerous cloud mistakes: 1) Public S3 buckets — over 200,000 buckets have leaked data since 2018, including 1.2 billion records from a single misconfigured AWS S3 bucket in 2019 (Attunity). 2) Overly permissive IAM roles — assigning AdministratorAccess to an EC2 instance means if that box is compromised via SSRF (like Capital One), the attacker gets full cloud admin. 3) Hardcoded cloud credentials — developers committing AWS keys to public GitHub repos. Automated scanners find these within minutes.

The AWS IAM policy language: Every permission in AWS is expressed as a JSON policy document with Effect, Action, Resource, and optional Condition.

{'{'} "Version": "2012-10-17", "Statement": [ {'{'} "Effect": "Allow", "Action": "s3:GetObject", "Resource": "arn:aws:s3:::prod-app-data/*", "Condition": {'{'}"IpAddress": {'{'}"aws:SourceIp": "10.0.0.0/8"{'}'}{'}'} {'}'} ] {'}'}
Key insight: The shared responsibility model means "the cloud provider is responsible FOR the cloud, but you are responsible FOR what is IN the cloud." Misconfigurations are the #1 cause of cloud breaches — not provider vulnerabilities.
// attacker vs defender mindset
X Attacker
  • Scan for open S3 buckets using bucket-stream or Grayhat Warfare — search for s3.amazonaws.com/[bucket-name] patterns.
  • Enumerate IAM roles from compromised EC2 metadata: curl http://169.254.169.254/latest/meta-data/iam/security-credentials/.
  • Search GitHub for hardcoded cloud keys using truffleHog or git-secrets.
  • Exploit overly permissive security groups — if port 22 or 3389 is open to 0.0.0.0/0, run a masscan against AWS IP ranges.
Y Defender
  • Enable S3 Block Public Access at the account level — a single checkbox that prevents any bucket from becoming public.
  • Use IAM roles instead of long-term access keys for EC2, Lambda, and ECS — roles auto-rotate credentials.
  • Configure VPC endpoints and instance metadata service v2 (IMDSv2) to block the SSRF-to-metadata attack.
  • Enable AWS CloudTrail with alerts on ConsoleLogin, CreateUser, PutBucketPolicy, and AuthorizeSecurityGroupIngress.
// real named incident
2019 — CAPITAL ONE BREACH — 100 MILLION RECORDS
Paige Thompson exploited SSRF through a misconfigured WAF to access the EC2 metadata service
Paige Thompson, a former AWS employee, identified a Server-Side Request Forgery (SSRF) vulnerability in Capital One's web application firewall configuration. The WAF was forwarding requests to the EC2 metadata service (169.254.169.254) instead of blocking them. Thompson used curl commands against the metadata endpoint to retrieve temporary IAM credentials for a role attached to the EC2 instance. That role had s3:ListBucket and s3:GetObject permissions on all S3 buckets in the account. Over 100 million credit applications dating back to 2005 were exfiltrated — including names, addresses, credit scores, and Social Security numbers. Capital One was fined $80 million by the OCC and $190 million in class-action settlements. The root cause: a single SSRF filter bypass combined with an over-privileged IAM role.
HANDS-ON LAB

Goal: configure AWS IAM users with least-privilege policies, create and secure S3 buckets, and understand the shared responsibility model through hands-on practice.

  1. Create a free AWS account and log into the IAM console. Create a new IAM user with programmatic access only and attach the 'IAMReadOnlyAccess' managed policy.
  2. Configure the AWS CLI: aws configure — enter your Access Key ID and Secret Access Key. Test with aws s3 ls and aws iam list-users.
  3. Create an S3 bucket: aws s3 mb s3://yourname-security-lab-001. Upload a test file: echo 'test' > test.txt && aws s3 cp test.txt s3://yourname-security-lab-001/.
  4. Make the bucket public by attaching a bucket policy that allows s3:GetObject for Principal '*'. Verify public access in a browser incognito window.
  5. Use aws s3api get-bucket-policy --bucket yourname-security-lab-001 to retrieve the policy. Then enable Block Public Access and confirm access is denied.
  6. Create a second IAM user with AdministratorAccess attached. Use aws iam list-attached-user-policies --user-name AdminUser to enumerate permissions.
Deliverable: A documented IAM user creation with least-privilege policy, plus a public-to-private S3 bucket transition with screenshots of policy changes and access verification.
# Create IAM user with read-only access aws iam create-user --user-name security-reader aws iam attach-user-policy --user-name security-reader \ --policy-arn arn:aws:iam::aws:policy/IAMReadOnlyAccess aws iam create-access-key --user-name security-reader > keys.json # Create S3 bucket and upload test file aws s3 mb s3://sec-lab-$(whoami)-001 echo "cloud security lab" > test.txt aws s3 cp test.txt s3://sec-lab-$(whoami)-001/ # Make bucket public (then revert) aws s3api put-bucket-policy --bucket sec-lab-$(whoami)-001 \ --policy '{"Version":"2012-10-17","Statement":[{"Effect":"Allow","Principal":"*","Action":"s3:GetObject","Resource":"arn:aws:s3:::sec-lab-$(whoami)-001/*"}]}' # Block public access (the fix) aws s3api put-public-access-block --bucket sec-lab-$(whoami)-001 \ --public-access-block-configuration BlockPublicAcls=true,IgnorePublicAcls=true,BlockPublicPolicy=true,RestrictPublicBuckets=true
INTERVIEW READY
Q: "What is the shared responsibility model in cloud security?"
A: "The shared responsibility model splits security between the cloud provider and the customer. The provider secures the infrastructure — physical data centers, hypervisors, network fabric. The customer secures everything within their account — IAM policies, data encryption, network ACLs, OS patching, application code. Most cloud breaches happen on the customer side: misconfigured S3 buckets, over-privileged IAM roles, exposed credentials, and unpatched workloads."
Q: "How would you detect and remediate a public S3 bucket?"
A: "Detection: run aws s3api get-bucket-policy --bucket name to check bucket policies, and aws s3api get-public-access-block to check Block Public Access settings. I would also use AWS Trusted Advisor and tools like CloudSploit for automated scanning. Remediation: enable S3 Block Public Access at the account level, apply least-privilege bucket policies, and use IAM roles instead of bucket policies where possible."
Q: "What is IMDSv2 and why does it matter?"
A: "IMDSv2 is AWS's Instance Metadata Service version 2. Unlike v1, which allows any process on the EC2 instance to access the metadata endpoint with a simple GET to 169.254.169.254, IMDSv2 requires a PUT request to start a session that returns a token required in subsequent GET requests. This prevents the SSRF-to-metadata attack used in the Capital One breach."
AI TOOLS
Prisma Cloud / CloudSploit
Open-source cloud security scanner that audits AWS, Azure, and GCP for misconfigurations. Run it against a test account to find public buckets and over-privileged roles.
TruffleHog secrets scanner
Scans GitHub repos for accidentally committed cloud credentials, API keys, and tokens. Supports high-entropy detection for AWS, Azure, GCP keys.
CloudGoat vulnerable-by-design
Intentionally vulnerable AWS environments from Rhino Security Labs. Deploy with Terraform and practice exploiting and fixing common cloud misconfigurations.
AWS IAM policy simulator
Official AWS tool to test IAM policy effects before applying. Enter a principal, action, and resource to see whether access is allowed or denied.
