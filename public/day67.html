Digital Forensics Basics

Forensics is the scientific method applied to incident investigation. Today you create forensic images, recover deleted files, and build a timeline of attacker activity.

THEORY

The Forensic Methodology: Digital forensics follows the same scientific principles as physical forensics — preserve the scene, collect evidence systematically, maintain chain of custody, analyze without altering originals. The core principle: never work on the original evidence. Always create a bit-for-bit forensic image (using tools like dd or FTK Imager) and analyze the copy. This preserves the integrity of evidence for potential legal proceedings.

Key forensic disciplines:

Disk Forensics
Imaging hard drives (dd, FTK Imager), analyzing partition tables, file systems (NTFS, ext4, APFS), recovering deleted files, carving unallocated space.
Memory Forensics
Capturing RAM (LiME, DumpIt, FTK Imager), analyzing processes, network connections, registry keys in memory — artifacts not written to disk.
Network Forensics
Capturing and analyzing pcap files (tcpdump, Wireshark), reconstructing TCP streams, identifying C2 patterns, extracting exfiltrated data.
File Carving
Extracting files from raw disk images by looking for file headers (magic bytes). Tools: foremost, scalpel, photorec — recover JPEGs, PDFs, archives from unallocated space.

Chain of Custody: The legal foundation of digital forensics. Every time evidence is collected, transferred, accessed, or analyzed, it must be documented. The chain of custody form records: who collected the evidence, when, where, what tools were used, the hash values (MD5/SHA256) of the evidence at acquisition, and every subsequent person who accessed it. A broken chain of custody makes evidence inadmissible in court.

Why this matters: Forensic evidence is what separates a security incident report from admissible evidence in court. Every SOC analyst should know how to acquire evidence without altering it, because you never know which incident will end up in litigation. The dd command with the right flags is the foundation.
// attacker vs defender mindset
ATTACKER
  • Use timestomping (e.g., Set-MacAttribute or touch -t) to modify file creation/modification timestamps, complicating timeline reconstruction by placing the malware before the compromise window.
  • Overwrite deleted files with zeros (sdelete, wipe) before they can be carved from unallocated space — file carving only works if the data hasn't been overwritten.
  • Run entirely from memory (fileless malware) to leave minimal disk artifacts — PowerShell payloads that never touch the disk are invisible to traditional disk forensics.
  • Encrypt exfiltrated data before sending it — even if network forensics captures the traffic, the contents are unreadable without the encryption key.
DEFENDER
  • Capture memory before powering off a suspect system — RAM contains running processes, open network connections, and decrypted data that disappears on shutdown. Use LiME or DumpIt.
  • Always use a hardware write-blocker when imaging drives — a software write-blocker is not admissible in court. The write-blocker ensures zero bytes are written to the evidence drive.
  • Document the hash chain: original device hash → image hash → analysis copy hash. Any mismatch breaks the chain and the evidence is compromised.
  • Collect volatile data first (order of volatility): registers → cache → RAM → process table → network connections → disk. Get the most volatile data before it changes.

Digital forensics revealed attackers had access to Sony's systems for over a month before data was wiped
FBI forensic investigators recovered critical evidence from Sony's compromised systems after the "Guardians of Peace" group wiped 3,000+ workstations and 800 servers. Disk forensics on surviving systems revealed the attackers had maintained access since September 2014 — a full month before the November 24 wipe event. File system timestamps showed data exfiltration through encrypted FTP channels. Memory forensics on servers that hadn't been fully wiped revealed Beacon malware communicating with C2 infrastructure traced to IPs in South Korea and Singapore. The forensic timeline was crucial in attributing the attack to North Korean state-sponsored actors (Lazarus Group), and the chain-of-custody documentation was used in the FBI's criminal complaint.
HANDS-ON LAB

Goal: create a forensic image, recover deleted files, and build an activity timeline using command-line forensics tools.

  1. Create a practice target: dd if=/dev/urandom of=/tmp/test_disk.img bs=1M count=100 — a 100MB dummy disk image. Then format it: mkfs.ext4 /tmp/test_disk.img.
  2. Mount the image, create some files, then delete them: sudo mount -o loop /tmp/test_disk.img /mnt/forensic; echo "secret data" > /mnt/forensic/passwords.txt; rm /mnt/forensic/passwords.txt.
  3. Create a forensic image of the loop device (not the file): identify the loop device with losetup -a, then sudo dd if=/dev/loop0 of=/tmp/evidence.dd bs=4M status=progress.
  4. Verify hash integrity: sha256sum /tmp/test_disk.img /tmp/evidence.dd — they should match.
  5. Recover deleted files using foremost: sudo foremost -i /tmp/evidence.dd -o /tmp/foremost_output. Check the recovered files in /tmp/foremost_output.
  6. Build a timeline: use fls (from Sleuth Kit) to list files and their metadata: fls -r -m / -o 0 /tmp/evidence.dd | mactime -b -d > /tmp/timeline.csv. Examine the MAC times (modified, accessed, changed).
Deliverable: a screenshot showing the recovered deleted file from foremost output and the timeline CSV showing the file creation and deletion events. This proves you can acquire forensic images and recover deleted evidence.

INTERVIEW READY
Q: "What is the order of volatility and why does it matter?"
A: "The order of volatility is the sequence in which digital evidence should be collected based on how quickly it changes. From most volatile to least: CPU registers and cache, routing tables, ARP cache, process table, kernel statistics, RAM, temporary files, disk. You collect the most volatile data first because it disappears when the power is cut — RAM contents are lost on shutdown, network connections are lost on disconnect. This is why you capture memory before powering off a system during incident response."
Q: "What is a forensic image and how is it different from a backup?"
A: "A forensic image is a bit-for-bit copy of the entire storage device, including the operating system, deleted files, unallocated space, and file system metadata. A backup only copies active files. The forensic image preserves every byte, including data in unallocated space that may contain deleted files. Forensic images use tools like dd or FTK Imager, are verified with SHA256 hashes, and are acquired through a write-blocker to ensure the original isn't modified."
Q: "How would you recover deleted files from an NTFS drive?"
A: "First, create a forensic image of the drive using a write-blocker — never work on the original drive. Then use file carving tools like foremost or scalpel to scan the image for file headers (magic bytes). NTFS Forensics: the MFT (Master File Table) keeps records even after deletion until overwritten. Tools like The Sleuth Kit's fls can list recently deleted files by analyzing the MFT. For deeper recovery, carve the unallocated space looking for known file signatures (JPEG: FF D8 FF, PDF: 25 50 44 46). The key principle: time is critical — the more the drive is used after deletion, the more likely the data is overwritten."
