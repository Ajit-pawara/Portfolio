Pentesting Methodology Recon to Exploit to Report

Professional penetration testing follows a structured process. Today you learn the full lifecycle.

THEORY

The five phases: Phase 1: Reconnaissance (passive + active info gathering). Phase 2: Scanning and enumeration (service discovery, vulnerability ID). Phase 3: Exploitation (gaining access). Phase 4: Post-exploitation (privilege escalation, lateral movement, persistence). Phase 5: Reporting (findings, risk ratings, remediation). Skipping any phase reduces the test's value. Most common mistake: jumping to exploitation without recon.

Phase 1 -- Recon: whois target.com; dig target.com AXFR; subfinder -d target.com Phase 2 -- Scan: nmap -sV -sC -p- target; nikto -h http://target Phase 3 -- Exploit: searchsploit apache 2.4.49; msfconsole Phase 4 -- Post: whoami; id; sudo -l; find / -perm -4000 2>/dev/null Phase 5 -- Report: Executive Summary | Findings (severity, proof, remediation) | Appendix

Rules of engagement: Every pentest starts with a signed ROE defining scope, rules, and deliverables. Never test without written authorization. The CFAA and IT Act do not distinguish between ethical and malicious hacking without consent.

Key insight: The report is the only deliverable the client pays for. New pentesters spend 80% on exploitation, 20% on reporting. Professionals reverse that ratio.
// attacker vs defender mindset
ATTACKER
  • Skip recon at your own risk -- you'll exploit a honeypot or waste hours on dead services.
  • Focus on the critical path: internet-facing service > web shell > privesc > domain admin.
  • Document every command with screenshots. A finding without proof is a guess.
  • Always clean up: remove backdoors, restore configs, delete accounts.
DEFENDER
  • Understand the methodology to detect it. Recon = DNS queries + scans; exploitation = unusual HTTP.
  • Use pentest reports as prioritization guides -- fix Critical and High first.
  • Run your own internal red team exercises. Know your blind spots before attackers do.
  • Demand proof of concept from pentesters. No exploit proof = more investigation needed.

2017 -- HACKERONE UBER BUG -- SSRF TO AWS IAM KEYS
Researcher followed structured recon-to-exploit-to-report methodology on Uber's bug bounty program.
A researcher performed passive recon (subdomain enumeration, Google dorking), found an internal API vulnerable to SSRF, exploited it to reach AWS metadata, retrieved IAM credentials, and enumerated all accessible S3 buckets containing production data. They reported through HackerOne earning a $10,000 bounty. The report followed standard structure: vulnerability description, proof of concept (curl + screenshots), impact, remediation. This became a case study in professional bug bounty methodology.
HANDS-ON LAB

Goal: Create a pentest plan for a fictional target: webapp.example.com (scope 203.0.11...

  1. Create a pentest plan for a fictional target: webapp.example.com (scope 203.0.113.50-55, no DoS, 9PM-6AM IST).
  2. Phase 1: whois example.com, dig example.com ANY, subfinder -d example.com.
  3. Phase 2: nmap -sV -sC -T4 10.10.10.1. Identify 3+ open ports and services.
  4. Phase 3: For each service, run searchsploit . Document attack paths.
  5. Phase 4: On any Linux box, run id; uname -a; sudo -l; ss -tlnp; find / -perm -4000.
  6. Phase 5: Write 2-3 fictional findings with format: Title, Severity (CVSS), Description, PoC, Remediation.
Deliverable: Complete pentest report template covering all 5 phases with tool outputs and CVSS-ranked findings.

INTERVIEW READY
Q: "Walk me through the 5 phases of a pentest."
A: "Phase 1 Recon (passive + active), Phase 2 Scanning (version detection, vuln ID), Phase 3 Exploitation (gaining access), Phase 4 Post-Exploitation (privesc, lateral, persistence), Phase 5 Reporting (findings with severity, PoC, remediation). Iterative -- I return to recon after discovering new info during post-exploitation."
Q: "Vulnerability assessment vs pentest?"
A: "VA is automated scanning producing potential findings -- broad but shallow with false positives. Pentest is manual exploitation proving vulnerabilities exist -- narrow but deep. VAs run monthly, pentests annually or after major changes."
Q: "What should a pentest report contain?"
A: "1) Executive Summary for management. 2) Methodology. 3) Findings with CVSS severity, description, PoC screenshots, remediation. 4) Risk rating summary. 5) Appendix: raw outputs, command history."
