CSRF & Session Attacks

Modern web apps use cookies and tokens to remember who you are. Today you learn how attackers forge requests and steal sessions — and how to prevent both.

THEORY

Cross-Site Request Forgery (CSRF): CSRF tricks an authenticated user into performing an unintended action. The attacker crafts a malicious page that — when visited by a victim who is already logged into a target site — automatically submits a request that the target site interprets as legitimate. The key enabler: browsers automatically attach cookies to every request sent to a domain, regardless of where the request originated.

Classic CSRF attack flow:

# Victim is logged into bank.com (session cookie in browser) # Attacker crafts a page with this image tag: <img src="https://bank.com/transfer?amount=5000&to=attacker-account" width="0" height="0" /> # Browser loads the "image" → GET request with cookies attached # Bank processes the transfer — it has no way to know the request # was forged because the session cookie is valid.

Session hijacking: An attacker steals a valid session identifier (session cookie, JWT, access token) and uses it to impersonate the victim. Methods include: XSS to read document.cookie, network sniffing over unencrypted connections, session fixation (setting the victim's session to a known value), or predictable session token generation.

SameSite cookies — the modern CSRF defense:

# Set-Cookie with SameSite attribute: Set-Cookie: session=abc123; HttpOnly; Secure; SameSite=Lax # Lax (default in modern browsers): cookie sent only for top-level # navigations (GET requests from clicking a link). # Strict: cookie never sent on cross-site requests. # None: cookie sent on all cross-site requests (requires Secure flag).
Anti-CSRF tokens: The server embeds a unique, unpredictable token in each form (typically in a hidden field). When the form is submitted, the server validates the token matches what was issued. Since an attacker's page cannot read the victim's token (same-origin policy), forged requests fail. This is the traditional defense; SameSite cookies are the modern evolution.
// attacker vs defender mindset
⚔ Attacker
  • Craft a CSRF PoC HTML file — an auto-submitting form with hidden inputs that POSTs to the target site. If there's no CSRF token, the request is accepted.
  • Check for missing SameSite cookie attributes — cookies without SameSite default to Lax in Chrome but some browsers still allow None behavior.
  • Steal session cookies via XSS — document.cookie returns session tokens if HttpOnly flag is absent. Use a fetch to exfiltrate.
  • Test for session fixation — see if the server accepts a session ID you provide in the URL or cookie before login.
🛡 Defender
  • Implement anti-CSRF tokens on every state-changing endpoint (POST, PUT, DELETE). Use framework-native CSRF protection (Rails, Django, Spring Security).
  • Set SameSite=Strict or Lax on all session cookies. This prevents most CSRF attacks at the browser level without application changes.
  • Set HttpOnly and Secure flags on all cookies containing session tokens — prevents JavaScript access and transmission over HTTP.
  • Use Origin and Referer header validation on the server side — verify the request originated from your own domain.
// real named incident
2013 — FACEBOOK CSRF TOKEN FLAW — UNATTENDED POSTS ON MILLIONS OF TIMELINES
CSRF allowed attackers to post on victim timelines via malicious image tags
A security researcher discovered that Facebook's "Like" button and status update endpoints lacked CSRF protection for certain image-loaded contexts. An attacker could embed a crafted image tag on any website that, when loaded by a logged-in Facebook user, would POST a status update on their behalf. The vulnerability was amplified by Facebook's widely used social plugins — millions of sites embedded Facebook widgets, giving attackers countless delivery vectors. Facebook patched by adding CSRF tokens to all state-changing endpoints and implementing OAuth 2.0 state parameters for their APIs.

Goal: craft a CSRF proof-of-concept, test SameSite cookie behavior, and simulate a session hijacking attack.

  1. Set up Juice Shop — docker run -d -p 3000:3000 bkimminich/juice-shop. Log in and capture the session cookie from DevTools → Application → Cookies.
  2. Create a local HTML file with an auto-submitting form targeting the Juice Shop change-password or feedback endpoint. Open it in the same browser session.
  3. Check the SameSite attribute on the Juice Shop session cookie — note whether it's Lax, Strict, or None.
  4. Simulate session hijacking: extract the session cookie via DevTools, then use curl to make an authenticated request from another terminal: curl -b "token=STOLEN_COOKIE" http://localhost:3000/profile.
  5. Add SameSite=Strict to the server's cookie configuration and repeat the CSRF test. Observe the attack fails.

INTERVIEW READY
Q: "What is CSRF and how do you prevent it?"
A: "CSRF forces an authenticated user to execute unintended actions by crafting a page that sends a cross-origin request with the user's cookies automatically attached. Prevention: anti-CSRF tokens embedded in every form (validated server-side), SameSite=Strict cookie attribute, Origin/Referer header validation, and requiring re-authentication for sensitive actions like password changes."
Q: "How does SameSite cookie attribute prevent CSRF?"
A: "SameSite tells the browser when to send cookies in cross-site requests. SameSite=Strict never sends the cookie cross-site — a forged request from another site has no session token, so the server rejects it. SameSite=Lax sends cookies on top-level navigations via GET but not on POST from other origins. Both are effective CSRF defenses because the attacker's page can't trigger authenticated state-changing requests without the victim's session cookie."
Q: "What's the difference between session hijacking and session fixation?"
A: "Session hijacking is stealing an already-established session token. Session fixation is forcing the victim's session ID to a value the attacker knows before login. In a fixation attack, the attacker sends the victim a link with a specific session ID, the victim logs in with that session ID, and the attacker now has a valid authenticated session. Prevention: regenerate the session ID after successful login."

ENTREPRENEUR ANGLE

Session management flaws are among the highest-impact findings in a pentest report. CSRF alone can lead to account takeover, data modification, and privilege escalation. As a consultant, every web application assessment includes a full session management review — cookie configuration, token generation randomness, logout behavior, concurrent session handling.

Productized opportunity: "Session Security Audit" — a $1,000-2,000 engagement where you audit a client's authentication and session management implementation. You check cookie flags, CSRF protection, session regeneration, token entropy, and session expiry. Deliver a report with proof-of-concept attacks and remediation code. Most startups and SaaS platforms neglect session security until they're breached — you're offering the audit before the incident.
