DAY 44 OF 90 — PHASE 3: WEB & NETWORK EXPLOITATION

Practicing SQLi & XSS on DVWA

DVWA is the most popular intentionally vulnerable web application. Today you apply everything from Phase 2 — SQL injection, XSS, CSRF, and file upload — against real targets.

THEORY

Damn Vulnerable Web Application (DVWA): DVWA is a PHP/MySQL web application that contains a graduated set of vulnerabilities at three security levels — Low, Medium, and High. Each level demonstrates a progressively harder exploitation scenario with real-world defenses simulated. Low level has no protections at all. Medium level introduces basic sanitization. High level implements more robust (but still flawed) controls. The platform lets you practice SQLi, XSS (stored, reflected, DOM), CSRF, file upload, command injection, and more in a legal sandbox.

SQLi across DVWA levels:

// Low — no input handling at all $query = "SELECT first_name, last_name FROM users WHERE user_id = '$_GET['id']'"; $result = mysqli_query($conn, $query); // Payload: ' UNION SELECT user,password FROM users -- // Medium — mysqli_real_escape_string (bypassable with numeric injection) $id = mysqli_real_escape_string($conn, $_GET['id']); $query = "SELECT first_name, last_name FROM users WHERE user_id = $id"; // Integer field — no quotes needed: 1 UNION SELECT user,password FROM users // High — separate query page, limits results $query = "SELECT first_name, last_name FROM users WHERE user_id = '$id' LIMIT 1"; // Use # instead of -- to comment out LIMIT: 1' UNION SELECT user,password FROM users#

XSS across DVWA levels: Reflected XSS at Low level simply echoes the name parameter without sanitization. Medium level uses str_replace to swap <script> to empty string — bypass with <scr<script>ipt>. High level uses preg_replace with a regex that's case sensitive — bypass with uppercase or img tag vectors. Each level teaches a specific filter bypass technique that maps directly to real-world WAF evasion.

Why DVWA matters: Over 500 academic papers reference DVWA as the standard platform for controlled vulnerability testing. It's the closest you can get to a real penetration test without needing authorization. The three-tier difficulty system teaches you to adapt your attacks as defenses increase — exactly what happens in real engagements.
// attacker vs defender mindset
⚔ Attacker
  • Start at Low security to confirm the vulnerability exists, then escalate to Medium and High — this teaches you exactly which defense blocks which payload and how to bypass it.
  • Use Burp Suite's Repeater to send modified requests without relying on the browser form — this lets you bypass client-side validation and see the raw server response.
  • For file upload at Medium level, the server checks MIME type client-side via JavaScript. Disable JS or intercept with Burp and change the Content-Type to image/jpeg.
  • For CSRF at Low level, craft a self-contained HTML page with a hidden form that auto-submits to DVWA's password change endpoint — no user interaction needed.
🛡 Defender
  • Apply the same security level escalation in your thinking — Low security = no input validation (fix: prepared statements + CSP), Medium = string-level filters (fix: context-aware encoding), High = regex-based blocking (fix: allowlists over blocklists).
  • DVWA's Medium-level XSS defense uses str_replace to remove "script" — but this fails because the filter runs only once. Implement recursive filtering or better, use a proper HTML sanitizer library.
  • File upload vulnerabilities are prevented by validating file content (magic bytes), not by checking Content-Type or extension. Use php.ini disable_functions to limit what uploaded PHP files can do.
  • The fact that DVWA exists and is necessary shows that defensive coding is not intuitive — you must actively learn each vulnerability class and its prevention.

HANDS-ON LAB

Goal: set up DVWA locally and exploit SQLi, XSS, CSRF, and file upload vulnerabilities across all three security levels.

  1. Deploy DVWA: docker run -d -p 80:80 vulnerables/web-dvwa. Navigate to http://localhost — default creds: admin/password.
  2. Set security to LOW in DVWA Security tab. Go to SQL Injection page. Enter ' UNION SELECT user,password FROM users# in the User ID field.
  3. Switch to Reflected XSS. Enter <script>alert('XSS')</script> — observe the popup. Now enter <img src=x onerror=alert('XSS')> and note it works even when script tags are blocked.
  4. Set security to MEDIUM. Try the same SQLi payload. It fails. Change to 1 UNION SELECT user,password FROM users — no quotes needed for integer fields.
  5. For Medium XSS, enter <scr<script>ipt>alert('XSS')</script> — the inner tag survives the str_replace filter. Test the file upload module by intercepting the request in Burp and changing Content-Type.
  6. Set security to HIGH. Use Burp Repeater to send SQLi payloads with # (URL-encoded as %23) to bypass the LIMIT clause. Document which payloads work at each level.
Deliverable: A comparison table documenting which payloads succeeded at Low, Medium, and High security for SQLi and XSS. Include bypass techniques at each level and the defense mechanisms encountered.

INTERVIEW READY
Q: "What is DVWA and how would you use it in a penetration testing context?"
A: "DVWA — Damn Vulnerable Web Application — is a deliberately insecure PHP/MySQL web app that contains SQL injection, XSS, CSRF, file upload, and other vulnerabilities. It has three security levels: Low (no protection), Medium (basic filters like str_replace and mysqli_real_escape_string), and High (regex-based and token-based protections). I use it to practice exploitation techniques at each level and to study how defenses change attack syntax. It's also useful for demonstrating specific vulnerabilities in training scenarios."
Q: "How would you bypass MEDIUM security SQL injection in DVWA?"
A: "Medium security uses mysqli_real_escape_string, which escapes quote characters and prevents string-based injection. However, the application still concatenates user input directly into the query string. For integer-based fields like the User ID input, no quotes are needed — I can inject UNION SELECT directly. For string fields, I look for second-order injection or alternative injection points that aren't filtered. The key insight is that escaping is not parameterization — the data is still being interpreted as SQL code."
Q: "Explain the difference between how LOW, MEDIUM, and HIGH security levels handle SQL injection and what real-world equivalent each represents."
A: "Low security has no protection — direct string concatenation. This represents legacy PHP applications written before security awareness. Medium uses mysqli_real_escape_string — equivalent to applications that think escaping is sufficient. HIGH uses a separate form submission with LIMIT and token-based CSRF protection — this represents modern applications with basic security controls. None use parameterized queries, which is the actual industry-standard fix. This progression is useful for learning but doesn't represent truly secure coding."
