Endpoint Detection & Response — EDR Basics

EDR is the difference between knowing something is wrong and knowing exactly what happened, how it got in, and what it's doing right now. While IDS (Day 62) watches network traffic, EDR watches the endpoint — every process, file, registry change, and network connection.

THEORY

What is EDR? Endpoint Detection & Response (EDR) is a security system that continuously monitors endpoint devices (servers, workstations, laptops) for malicious activity. Unlike traditional antivirus (signature-based), EDR is behavior-based — it looks for what processes do, not just what files exist.

Behavioral Detection vs Signature-Based Detection:

❌ Signature-Based (Traditional AV)
Matches file hashes or byte sequences against a known-bad database.
Zero-day malware: invisible until signature is published.
Polymorphic malware: same behavior, different hash — bypasses easily.
Fileless attacks: no file to hash — invisible entirely.
✅ Behavioral (EDR)
Models normal behavior — flags deviations without needing a known-bad list.
Detects zero-days by observing anomalous actions (e.g., powershell.exe making outbound connections).
Builds process ancestry trees — winword.exe -> cmd.exe -> powershell.exe is always suspicious.
Fileless execution: monitors process injection, WMI persistence, and registry run keys.

EDR telemetry sources — what gets collected:

Process Events
Process creation (PID, parent PID, command line, user token). Every process has a parent — the ancestry chain is the most powerful signal in EDR. winword.exe (PID 1234) -> cmd.exe (PID 5678) is a red flag.
File System Events
File create, modify, delete, rename. EDR tracks which process wrote which file. A DLL written to System32 by explorer.exe (not a legitimate installer) triggers an alert.
Registry Events
Registry key create/modify/delete. Persistence mechanisms (Run keys, services, scheduled tasks) are registry operations — EDR flags reg.exe add HKLM\...\Run /v Backdoor.
Network Events
Outbound connections, DNS queries, TLS handshakes. The EDR sensor logs process.exe -> IP:port — even if the traffic is encrypted, the beacon pattern is visible at the telemetry layer.

Fileless attack detection — the crown jewel of EDR: Fileless malware never writes a file to disk. It executes entirely in memory — often through PowerShell, WMI, or macro code. Signature AV sees nothing because there's no file to scan. EDR detects fileless attacks by monitoring the behavior chain: a Microsoft Office process spawning a script interpreter (cmd.exe, powershell.exe) that makes an outbound network connection is automatically flagged as suspicious, regardless of whether a file was ever written.

Key insight: EDR is not a replacement for antivirus — it's a fundamentally different approach. AV answers "is this file known-bad?" EDR answers "is this process behaving suspiciously?" A modern defense uses both: AV for known threats, EDR for everything else.
// attacker vs defender mindset
⚔ Attacker
  • Use living-off-the-land binaries (LOLBins) — powershell.exe, wmic.exe, mshta.exe, cscript.exe — to execute code without dropping a custom binary.
  • Perform process injection: allocate memory in a trusted process (e.g., svchost.exe) and execute shellcode inside it — EDR sensors see the write but it blends with normal process activity.
  • Disable or tamper with EDR sensors if you gain administrator privileges: sc stop CrowdStrike Falcon or unload kernel drivers.
  • Use reflective DLL loading — load a DLL from memory without registering it in the PEB (Process Environment Block), bypassing user-mode API hooks.
🛡 Defender
  • Enable command-line auditing — the single most important EDR data source. Always capture the full command line of every process creation event. Attackers' intentions are visible in their arguments.
  • Build process tree baselines: svchost.exe should never parent powershell.exe. Any deviation from the baseline is immediately investigated.
  • Deploy EDR sensors in blocking mode (not just monitoring) for signed processes — block unsigned PowerShell execution, block remote WMI calls from non-admin workstations.
  • Correlate EDR alerts with network IDS alerts (Day 62) — a Suricata alert for C2 beaconing + an EDR process tree showing a suspicious PowerShell parent chain = confirmed compromise.
// real named incident
2021 — HAFNIUM EXCHANGE ATTACKS — 30,000+ US ORGANIZATIONS COMPROMISED
CrowdStrike Falcon EDR detected anomalous w3wp.exe process behavior — the first sign of Hafnium exploitation
In March 2021, a Chinese state-sponsored group (Hafnium) exploited four zero-day vulnerabilities in Microsoft Exchange Server (CVE-2021-26855, CVE-2021-26857, CVE-2021-26858, CVE-2021-27065). The first indicators were not signatures — they were process tree anomalies detected by CrowdStrike Falcon EDR sensors. The EDR telemetry showed w3wp.exe (the IIS worker process for Exchange) spawning cmd.exe and powershell.exe — behavior that never occurs in normal Exchange operation. The EDR process trees revealed the full attack chain: web shell upload through the SSRF vulnerability, followed by cmd.exe execution to install additional persistence, then outbound PowerShell connections for data exfiltration. CrowdStrike published their findings within 24 hours, but over 30,000 US organizations were compromised before patches could be deployed. The EDR telemetry was critical for incident response — it told responders exactly which systems were compromised, what the attackers executed, and what data was accessed.

INTERVIEW READY
Q: "How does EDR differ from traditional antivirus?"
A: "Traditional antivirus is signature-based — it matches file hashes or byte sequences against a database of known malware. It can't detect zero-days or fileless attacks. EDR is behavior-based — it collects telemetry from every process, file, registry, and network event, then uses analytics to detect anomalous behavior. EDR answers 'what is this process doing?' instead of 'is this file known-bad?' The key difference: AV detects known files, EDR detects suspicious actions. They complement each other."
Q: "What is a process tree and why is it important in EDR?"
A: "A process tree shows the ancestry chain of every running process — which process spawned which child processes. In normal operation, explorer.exe spawns chrome.exe, winword.exe stays alone. In an attack, winword.exe spawning cmd.exe which spawns powershell.exe is deeply suspicious. Process trees are the most reliable signal in EDR because they reveal the attacker's execution flow regardless of what specific malware is used. During the Hafnium exchange attacks, the w3wp.exe -> cmd.exe -> powershell.exe chain was the definitive indicator of compromise."
Q: "How does EDR detect fileless malware?"
A: "Fileless malware executes entirely in memory — it never writes a file to disk, so signature-based AV can't see it. EDR detects fileless attacks by monitoring the behavior chain: a trusted process (like a macro in Word) spawning a script interpreter (PowerShell, WMI) that makes an outbound network connection. Even if the payload is never written to disk, the behavioral sequence is anomalous. EDR also monitors for process injection (VirtualAllocEx + WriteProcessMemory + CreateRemoteThread), reflective DLL loading, and WMI event subscription persistence — all fileless techniques that leave no on-disk artifact but generate detectable telemetry."
