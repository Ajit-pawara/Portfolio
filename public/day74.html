DevSecOps & Secure SDLC Intro

Security can't be bolted on at the end. DevSecOps integrates security testing into every stage of software development. Today you shift left.

THEORY

What is DevSecOps? DevSecOps is the practice of integrating security controls and testing into every phase of the DevOps lifecycle — from planning and coding to building, testing, releasing, and monitoring. The core principle is "shift left" — moving security earlier in the development process where fixes are cheaper, faster, and less disruptive. A vulnerability found in production costs 100x more to fix than one caught during design.

The Secure SDLC phases:

1. Plan
Threat modeling (STRIDE), security requirements, risk acceptance criteria. Tools: OWASP Threat Dragon.
2. Code
SAST scanning in IDE (Semgrep, SonarLint), pre-commit hooks for secrets (truffleHog, git-secrets), peer review.
3. Build
SCA for dependency vulns (OWASP Dependency-Check, Snyk), SAST in CI pipeline, container scanning (Trivy).
4. Test
DAST (OWASP ZAP), dynamic analysis, fuzz testing, API security testing. Pre-prod environment.
5. Deploy
Infrastructure as Code scanning (Checkov), configuration drift detection, signed artifacts.
6. Monitor
Runtime protection, SIEM alerts, vulnerability management, incident response playbooks.

SAST vs DAST vs SCA:

SAST — Static Application Security Testing (white-box, source code scanning) Tools: Semgrep, SonarQube, Fortify, Checkmarx Finds: SQLi, XSS, hardcoded secrets, insecure deserialization When: During coding (IDE + CI pipeline) DAST — Dynamic Application Security Testing (black-box, running app) Tools: OWASP ZAP, Burp Suite, Arachni Finds: Runtime vulnerabilities, auth bypass, config issues When: After deployment to test/staging SCA — Software Composition Analysis (open-source dependency scanning) Tools: OWASP Dependency-Check, Snyk, Dependabot Finds: Known CVEs in third-party libraries, license violations When: During build (CI pipeline)

Sample Semgrep rule for detecting debug endpoints:

# semgrep-rule.yaml — detect exposed debug routes rules: - id: debug-endpoint-exposed pattern-either: - pattern: | @app.route("/debug", ...) - pattern: | @app.route("/admin", ...) message: "Debug or admin endpoint exposed in production" severity: ERROR languages: [python]
Shift left principle: Finding and fixing a vulnerability during the coding phase costs ~$80. The same vulnerability found in production costs ~$8,000. DevSecOps is about catching issues where they're cheapest to fix, not where they're easiest to notice.
// attacker vs defender mindset
⚔ Attacker
  • Scan for exposed CI/CD pipelines — misconfigured Jenkins, GitLab CI, or GitHub Actions runners can give you code access, secret exposure, or RCE through build scripts.
  • Look for hardcoded secrets in public repos — pre-commit hooks catch them, but many teams don't use them. TruffleHog across an org's GitHub org finds API keys, DB creds, and cloud tokens.
  • Target dependency confusion — if open-source packages aren't validated, you can upload a malicious package with the same name to a public registry and get it pulled into the build.
  • Exploit infrastructure-as-code misconfigurations — public S3 buckets, overly permissive IAM roles, and open security groups in Terraform/CloudFormation templates.
🛡 Defender
  • Implement pre-commit hooks that block secrets, large files, and known vulnerable patterns before they reach the remote repository.
  • Run SAST (Semgrep) and SCA (Dependency-Check) in CI — fail the build on critical or high-severity findings. Gate deployments on scan results.
  • Scan container images with Trivy or Clair before pushing to registry — don't deploy containers with known CVEs even if the code is clean.
  • Use infrastructure-as-code scanning (Checkov, tfsec) to catch misconfigurations before cloud resources are provisioned.

2016 — ETSY DEVSECOPS — CONTINUOUS SECURITY TRANSFORMATION
Etsy automated 23 security checks into their deployment pipeline, reducing discovery-to-patch from weeks to under 24 hours
Etsy's security team realized their manual security review process couldn't keep up with 50+ deployments per day. They embedded security engineers into product teams and automated 23 distinct security checks into their continuous deployment pipeline — including automated XSS scanning, CSRF verification, SQL injection detection, and dependency vulnerability checks. The result: vulnerabilities that previously took weeks to discover and patch were now detected and fixed within a single deployment cycle. Their "blameless postmortem" culture meant developers reported security issues without fear of punishment, creating a continuous improvement feedback loop that reduced mean time to remediation (MTTR) by 95%.
HANDS-ON LAB

Goal: Set up a pre-commit hook for secret scanning, run SAST with Semgrep, and integrate OWASP Dependency-Check into a CI pipeline.

  1. Install pre-commit and truffleHog: pip install pre-commit trufflehog then create .pre-commit-config.yaml with a secrets scanning hook.
  2. Run pre-commit install && pre-commit run --all-files to scan an existing project for accidentally committed secrets.
  3. Install Semgrep: pip install semgrep and run semgrep --config=p/python --error main.py against a sample Python file.
  4. Write a vulnerable Flask app (a login endpoint with SQLi) and run Semgrep with the OWASP Top 10 rule pack: semgrep --config=p/owasp-top-ten login.py.
  5. Install OWASP Dependency-Check: docker run --rm -v $(pwd):/src owasp/dependency-check --scan /src --format HTML --out /src/report.html.
  6. Create a GitHub Actions workflow (.github/workflows/security.yml) that runs Semgrep and Dependency-Check on every push and pull request.
Deliverable: A GitHub repo with working pre-commit hooks, a Semgrep scan output showing at least one finding, and a Dependency-Check HTML report. These demonstrate your DevSecOps pipeline skills in an interview.

INTERVIEW READY
Q: "What is 'shifting left' in DevSecOps?"
A: "Shifting left means moving security testing earlier in the software development lifecycle. Instead of waiting for a security review before production release, you integrate automated security checks into the IDE, pre-commit hooks, and CI pipeline. A vulnerability caught during coding costs ~$80 to fix; the same issue found in production costs ~$8,000. Shifting left is about economics as much as security."
Q: "What's the difference between SAST and DAST?"
A: "SAST is white-box testing — it scans source code statically without running the application. It finds vulnerabilities like SQLi and XSS in code patterns but can produce false positives. DAST is black-box testing — it scans the running application by sending HTTP requests and analyzing responses. It finds runtime issues like auth bypass and config problems but can miss code-level issues. They're complementary — SAST catches things DAST misses and vice versa."
Q: "How would you implement security scanning in a CI/CD pipeline?"
A: "I'd add multiple gates: pre-commit hooks for secrets before code reaches the repo, then in CI — run SAST (Semgrep) for code vulnerabilities, SCA (Dependency-Check) for library CVEs, and container scanning (Trivy) for base image issues. I'd fail the build on critical/high findings but allow medium/low as warnings. For the deploy stage, I'd add DAST (OWASP ZAP) against the staging environment. I'd use Infrastructure-as-Code scanning (Checkov) on Terraform files before provisioning."
