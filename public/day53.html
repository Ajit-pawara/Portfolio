CTF Strategy Web Category

Web challenges in CTFs reward methodical enumeration over brute force. Today you learn to read source code, manipulate cookies, and bypass filters systematically.

THEORY

Why web challenges are the most common CTF category: In any CTF competition, 40-60% of challenges are web-based. They test the same skills as real-world pentesting — reading source code, understanding HTTP mechanics, manipulating parameters, and exploiting logic flaws. Unlike binary exploitation (which requires assembly knowledge) or cryptography (which requires math), web challenges are accessible to anyone who understands how web applications work. The best CTF web players are methodical problem-solvers who read everything on the page before touching a tool.

Four critical web CTF techniques:

// 1. PHP Type Juggling — loose comparison exploits if ($_GET['pass'] == "admin123") { echo $flag; } // Input: pass=true → true == "admin123"? No, but: // Input: pass=0e123 → 0e123 == "0e456"? PHP treats "0e..." as scientific notation = 0 // If hash starts with "0e" followed by digits, it equals "0e" + otherdigits == 0 == 0 // 2. JWT Manipulation — "none" algorithm attack // Original token: header: {"alg":"HS256"}, payload: {"user":"guest"} // Modified token: header: {"alg":"none"}, payload: {"user":"admin"} // Server accepts unsigned tokens if it doesn't validate the "none" algorithm // 3. SSTI (Server-Side Template Injection) // Jinja2 (Flask) — test with: {{7*7}} // If you see "49" in the response, SSTI is confirmed // Payload: {{config.__class__.__init__.__globals__['os'].popen('cat flag.txt').read()}} // 4. GraphQL Introspection // Most GraphQL endpoints accept introspection queries // Query: {__schema{types{name fields{name}}}} // If the endpoint returns the schema, you can find hidden queries and mutations

The web CTF methodology: Start by viewing the page source (Ctrl+U). Check the Network tab for hidden API calls. Look at cookies and local storage. Read JavaScript files — commented-out endpoints are common CTF clues. Try path traversal, parameter pollution, and HTTP method override. If there's a form, test it with empty data, special characters, and long strings. If there's a login, try SQLi, NoSQLi, and type juggling. If there's an API endpoint, fuzz it for hidden parameters. The flag is almost never protected by a sophisticated exploit — it's usually behind a simple manipulation you haven't tried yet.

Key insight: In CTF web challenges, the source code is always the answer. Not the rendered page — the actual HTML source, the JavaScript, the comments, the error messages. CTF creators leave hints in the source that you'd never see just looking at the rendered page. Always view source first.
// attacker vs defender mindset
⚔ Attacker
  • View page source immediately — look for HTML comments, hidden inputs, base64-encoded strings, and JavaScript functions that reveal API endpoints.
  • Use the browser's Developer Tools Network tab — intercept every XHR/fetch request. Modern web apps load data via API calls that aren't visible in the HTML.
  • Test PHP type juggling on login forms: send password[]= (array bypass), password=true, password=0e999999 (magic hash).
  • For JWT challenges, decode the token on jwt.io — modify the payload, change alg to "none", and re-encode without a signature. If it works, you bypassed authentication.
🛡 Defender
  • Use strict comparison (===) in PHP instead of loose (==) — this prevents type juggling where "0e123" == "0e456" because both evaluate to 0 in scientific notation.
  • In JWT libraries, explicitly disable the "none" algorithm and always verify the signature — don't rely on the algorithm in the token header, use a server-side whitelist.
  • For template engines, use sandboxed rendering environments and disable access to dangerous globals like os, subprocess, and eval.
  • Disable GraphQL introspection in production — the playground and schema dump should only be available in development environments.

ANNUAL — DEF CON CTF WEB CHALLENGES — REAL-WORLD VULNERABILITIES
DEF CON CTF web challenges reflect real-world vulnerabilities seen in major platforms
Since 1996, DEF CON's annual CTF has been the world's most prestigious cybersecurity competition, with web challenges that closely mirror real-world vulnerabilities. Past challenges have recreated Facebook Graph API access control flaws, Google App Engine misconfigurations, and Twitter XSS vectors. The web category at DEF CON CTF is known for requiring contestants to reverse-engineer obfuscated JavaScript, exploit server-side template injection in custom frameworks, and chain multiple low-severity bugs into a full compromise. Many DEF CON CTF web challenge authors are now security engineers at FAANG companies, and their challenges reflect the vulnerability classes they encounter in production. Solving CTF web challenges at this level directly translates to finding real-world bugs — several DEF CON CTF web challenge solutions have been adapted into actual bug bounty submissions.
HANDS-ON LAB

Goal: solve 3-5 CTF web challenges covering PHP type juggling, JWT manipulation, SSTI, and GraphQL introspection using free online platforms.

  1. Go to websec.fr (free CTF web challenges). Start with the "Easy" level challenges. Read each challenge's PHP source code — identify the vulnerability class.
  2. For PHP type juggling: find a challenge where the comparison uses == instead of ===. Submit password=0e999999 or password[]= to bypass.
  3. For JWT: go to jwt.io and decode a sample JWT. Modify the payload, change alg to "none", and encode it. Test against a JWT-based challenge on websec.fr or portswigger.net.
  4. For SSTI: navigate to a Jinja2/Flask challenge page. Enter {{7*7}} in a form field — if you see "49" in the response, SSTI is confirmed. Extract secrets with {{config}}.
  5. For GraphQL: use a public test API like countries.trevorblades.com. Send the introspection query {__schema{types{name}}} via the GraphQL playground to see all available types.
  6. Document each challenge: vulnerability class, the clue that revealed it, the payload used, and the flag obtained. Create a reusable "Web CTF Checklist" from your experience.
Deliverable: A "Web CTF Strategy Checklist" — a one-page document listing every technique you used, ordered by the sequence you'd try them on any new challenge. Include example payloads for: source review, cookies, JWT, SSTI, type juggling, and GraphQL.

INTERVIEW READY
Q: "What's your approach when you encounter a web challenge in a CTF?"
A: "I follow a systematic checklist: (1) View page source for comments and hidden fields. (2) Check cookies and localStorage. (3) Intercept all API requests in the Network tab. (4) Read JavaScript files for endpoints. (5) Test forms with special characters, long strings, and empty values. (6) Check for SQLi/
Q: "Explain PHP type juggling and how attackers exploit it."
A: "PHP's loose comparison operator (==) performs type coercion — it converts values to the same type before comparing. This causes many 'magic' behaviors: '0e12345' == '0e67890' because both are interpreted as 0 (scientific notation: 0 × 10^N). An empty array == null. 'abc' == 0 because PHP converts 'abc' to 0. Attackers exploit this by passing inputs like password=0e999999 or password[]= to bypass authentication checks that use == instead of ===."
Q: "What is Server-Side Template Injection and how do you find it?"
A: "SSTI occurs when user input is embedded into a template that's rendered server-side (Jinja2, Twig, Freemarker, Pug). The attacker injects template syntax that gets executed. Detection: enter {{7*7}} in a form field — if the response shows '49', SSTI is confirmed because the server evaluated the expression. After detection, I explore the template's object hierarchy to find ways to execute system commands. In Jinja2, I chain __class__, __mro__, __subclasses__ to reach the os module and execute commands."

ENTREPRENEUR ANGLE

CTF skills translate directly to bug bounty hunting — the independent contractor model of cybersecurity. Many of the techniques you've learned today (JWT manipulation, SSTI, GraphQL introspection) are the exact vulnerabilities that pay $500-5,000 on HackerOne and Bugcrowd. A dedicated bug bounty hunter with good methodology can earn $50,000-150,000/year part-time, with top earners exceeding $500K.

Beyond bug bounty, you can productize your CTF methodology as a "Web Security Auditing Checklist" and sell it to development teams. Many startups don't have security teams and their developers don't know where to start looking for vulnerabilities. A $50 downloadable PDF with your systematic approach — source review → cookie analysis → API fuzzing → authentication testing → input validation → GraphQL audit — solves a real problem for cash-constrained startups that can't afford a $15K pentest.
