DAY 22 OF 90 — PHASE 2: CORE SECURITY CONCEPTS

Web Fundamentals How a Request Actually Works

Every vulnerability you'll exploit in the coming days — SQL injection, XSS, CSRF, SSRF, IDOR — begins with understanding what a normal HTTP request looks like. Today you learn the raw mechanics so the attacks make sense.

THEORY

The request-response cycle: You type a URL, hit Enter, and less than a second later a page loads. In between, your browser assembled a structured message (the HTTP request), sent it across the internet to a server, which parsed it, decided what to send back, and returned an HTTP response. The entire web — every API call, every form submit, every image load — runs on this exchange.

Anatomy of an HTTP request:

GET /dashboard/profile HTTP/1.1 Host: example.com User-Agent: Mozilla/5.0 ... Accept: text/html Cookie: session=abc123; theme=dark Authorization: Bearer eyJhbGciOiJI... ↑ request line (method + path + protocol) ↑ headers (metadata about the request) ↑ blank line separates headers from body [body — present only for POST/PUT/PATCH]

Anatomy of an HTTP response:

HTTP/1.1 200 OK Content-Type: text/html; charset=UTF-8 Set-Cookie: session=xyz789; HttpOnly; Secure Content-Length: 4521 ↑ blank line <!DOCTYPE html> <html><head><title>Dashboard</title>...</body> </html>

The HTTP methods (verbs): Each method communicates intent. Security testing is often about finding endpoints that accept methods they shouldn't.

GET
Retrieve a resource. No body. Should be idempotent and safe.
POST
Create or submit. Has a body. Form submissions, API creates.
PUT
Replace an entire resource. Idempotent.
PATCH
Partial update to a resource.
DELETE
Remove a resource.
OPTIONS
Ask what methods are allowed. Useful for recon.

Status codes every security person knows by heart:

1xx — Informational (mostly irrelevant for testing) 200 — OK (request succeeded) 201 — Created (POST succeeded, resource made) 301/302 — Redirect (check where it redirects to!) 400 — Bad Request (malformed syntax) 401 — Unauthorized (no auth credentials) 403 — Forbidden (credentials present but denied) 404 — Not Found (or deliberately hidden) 405 — Method Not Allowed (wrong HTTP method) 500 — Internal Server Error (classic error-based injection signal) 502/503 — Gateway/Service Unavailable

How HTTPS actually protects the request: Before any HTTP data is exchanged, the client and server complete a TLS handshake. The handshake negotiates a symmetric session key using asymmetric cryptography (RSA or ECDHE). Once established, everything — the URL path, headers, cookies, body — is encrypted. Only the Host header's domain name leaks in plaintext (SNI). This is why HTTP/2 encrypts headers too, and why a WAF sees only encrypted traffic unless it terminates TLS.

Why this matters for security: Every attack you'll study in the coming days is an HTTP-level abuse. SQL injection poisons the body/parameters. XSS injects into the response body. CSRF forges a request. SSRF tricks the server into making a request. Understanding the raw request format is the prerequisite for all of them.
// attacker vs defender mindset
⚔ Attacker
  • Intercept login forms in Burp Suite — change the username, password, or hidden fields before the request reaches the server. No code, no credentials needed.
  • Craft raw HTTP requests manually using curl or Burp Repeater — manipulate headers (Host, Referer, X-Forwarded-For) to bypass access controls and WAF rules.
  • Inspect response headers for information leaks — Server: Apache/2.4.49, X-Powered-By: PHP/7.4, debug=true. Every header tells you what to attack next.
  • Test every endpoint for methods it shouldn't accept — OPTIONS returns allowed methods, and PUT on a static file server means you can write a webshell.
🛡 Defender
  • Remove or harden server headers — Server, X-Powered-By, and X-AspNet-Version leak technology stack details that attackers use to pick their exploits.
  • Restrict HTTP methods at the web server level — Apache's LimitExcept, Nginx's limit_except, and cloud WAF rules block PUT/DELETE on public endpoints.
  • Monitor for unusual request patterns — unexpected User-Agent strings, requests to non-existent paths (recon scanners), and OPTIONS requests to everything.
  • Enforce HTTPS with HSTS — Strict-Transport-Security ensures no request ever travels in plaintext, even if the user types http:// manually.
// real named incident
2018 — MAGEcart BRITISH AIRWAYS — 380,000 PAYMENT CARDS
Attackers injected a JavaScript skimmer into ba.com's payment page by abusing HTTP request flow
The Magecart group found an unsecured staging endpoint on ba.com that proxied requests to the main site. By injecting a 23-line JavaScript payload into that request flow, every payment form submission was silently copied to an attacker-controlled server. The attack was invisible to users because the main page still loaded normally — only the HTTP response body had been modified. 380,000 payment card details were stolen over two weeks. British Airways was fined £20M under GDPR. The root cause wasn't a complex exploit — it was a staging server that shouldn't have been reachable from the public internet.
HANDS-ON LAB

Goal: intercept, inspect, and modify real HTTP traffic using Burp Suite — the foundational skill for every web vulnerability test.

  1. Open Burp Suite Community Edition and configure the proxy listener on 127.0.0.1:8080.
  2. Set your browser to use Burp as its HTTP/HTTPS proxy and install the CA certificate for HTTPS inspection.
  3. Visit any website (e.g. example.com or your college portal). Watch each request appear in the Proxy → HTTP History tab.
  4. Send a request to Repeater, modify the User-Agent header, and observe how the server responds differently.
  5. Identify: request line, headers, body, response status code, response headers, and response body for one request.
  6. Go to tryhackme.com/room/howwebworks — complete the HTTP requests and responses tasks inside their interactive web-based lab (free, no VM needed).
Deliverable: a screenshot or written breakdown of one intercepted HTTP request and response with every part labelled (method, path, status, headers, body). This is the raw material every vulnerability finding is built from.

INTERVIEW READY
Q: "Walk me through what happens when you type https://example.com/login into a browser and press Enter."
A: "DNS resolves the domain to an IP. TCP three-way handshake. TLS handshake to negotiate encryption. Then the browser sends an HTTP GET request to /login with headers including Host, User-Agent, and any stored cookies. The server processes it and returns an HTTP response with a status code, headers (Set-Cookie, Content-Type), and the HTML body. The browser parses the HTML, fetches any linked resources (CSS, JS, images) via additional GET requests, and renders the page."
Q: "What's the difference between 401 Unauthorized and 403 Forbidden?"
A: "401 means you haven't authenticated — send credentials and try again. 403 means you're authenticated but don't have permission to access that resource. 401 is 'who are you?', 403 is 'I know who you are, but you can't do that'."
Q: "What is the difference between GET and POST besides visibility in the URL?"
A: "GET is for retrieving data, should be idempotent and safe — no side effects. POST is for creating or submitting data, can have side effects, and includes a request body. GET requests can be cached, bookmarked, and appear in browser history. POST requests won't. From a security perspective, POST doesn't protect data — it still needs HTTPS."

ENTREPRENEUR ANGLE

HTTP fundamentals are the bedrock of every security consulting service you'd ever offer. Web app pentests, API security reviews, WAF tuning — every single engagement involves reading raw HTTP. A consulting firm that can't explain the difference between a 401 and a 403 to a client won't be trusted with anything deeper. Building this foundation now means you're not learning basic HTTP while a client is paying you $150/hr for a pentest.

Also worth noting: the simplest productized service you can offer as a freelancer is an "HTTP Security Headers Audit" — checking for missing Content-Security-Policy, Strict-Transport-Security, X-Frame-Options, etc. It's a 30-minute job you can charge $150-300 for because most small businesses don't even know these exist.
