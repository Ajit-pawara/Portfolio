SQL Injection Deep Dive

SQL injection is the oldest and most destructive web vulnerability. It has caused billions in damages, yet it's entirely preventable. Today you write actual injection payloads, understand exactly why parameterized queries stop them, and test against a real database.

THEORY

What is SQL Injection? SQL injection (SQLi) occurs when an application builds a SQL query by concatenating user-supplied input directly into the query string. An attacker can "break out" of the intended input context and execute arbitrary SQL commands — reading, modifying, or deleting data the application never intended to expose.

Anatomy of the vulnerability:

❌ Vulnerable (String Concatenation)
username = request.body["user"] query = "SELECT * FROM users WHERE username = '" + username + "' AND password = '" + password + "'" db.execute(query)
✅ Safe (Parameterized Query)
username = request.body["user"] query = "SELECT * FROM users" + " WHERE username = ? AND password = ?" db.execute(query, [username, password])

What happens when you input: ' OR '1'='1 into the vulnerable version?

SELECT * FROM users WHERE username = '' OR '1'='1' AND password = '' OR '1'='1' ↑ The '1'='1' is always true, so the WHERE clause evaluates to TRUE for every row. All users returned.

The five types of SQL injection:

UNION-Based
Uses the UNION SQL operator to append the attacker's query results to the original query's result set. Most straightforward — you see the data immediately in the app response.
Error-Based
Forces the database to generate error messages that reveal schema information (table names, column names). Useful when UNION is blocked or results aren't displayed.
Boolean-Based Blind
No data returned in the response. You ask yes/no questions by injecting conditions that change the page behavior (true = page loads, false = error/empty). Extract data one character at a time.
Time-Based Blind
When even true/false differences are invisible. You use database sleep functions (SLEEP(5), WAITFOR DELAY '0:0:5') to infer truth from response delays. Slow but reliable.
Out-of-Band (OOB)
Exfiltrates data through a separate channel — typically DNS lookups or HTTP requests to a server you control. Used when the database can make network requests (MySQL LOAD_FILE, MSSQL xp_cmdshell).

Every SQLi variant shares the same root cause: user input treated as executable code instead of data. The fix is always the same: parameterized queries (prepared statements). Not input validation, not escaping — parameterization.

Key insight: Input validation (blocking quotes, keywords, special chars) is fragile — attackers find bypasses. Parameterized queries separate SQL code from data at the protocol level, making injection structurally impossible. This is not "better" — it's a fundamentally different, sound approach.
// attacker vs defender mindset
⚔ Attacker
  • Inject a single quote ' into every input field — the first sign of SQLi is a database error or unexpected behavior when a quote breaks the query syntax.
  • Use UNION-based injection to extract other tables: ' UNION SELECT username, password FROM admins --. The -- comments out the rest of the original query.
  • When you get no visible output, switch to blind techniques: ' OR (SELECT SUBSTRING(password,1,1) FROM users WHERE id=1) = 'a — test one character at a time.
  • Use sqlmap for automated detection and extraction once you've confirmed a single injectable parameter. Manual discovery, automated exploitation.
🛡 Defender
  • Use parameterized queries / prepared statements exclusively — never concatenate user input into SQL strings. This is the single fix that prevents 100% of SQLi.
  • Apply the principle of least privilege to database accounts — the web app's DB user should only have access to what it needs (no DROP TABLE, no xp_cmdshell).
  • Implement a Web Application Firewall (WAF) as defense-in-depth — ModSecurity with the OWASP CRS blocks common SQLi patterns even if a developer makes a mistake.
  • Log and monitor for SQL errors — a sudden spike in 500 errors or database timeout alerts is often the first sign of an automated SQLi scan.
// real named incident
2008 — HEARTLAND PAYMENT SYSTEMS — 130 MILLION CARDS
Albert Gonzalez exploited SQL injection in Heartland's web-facing payment forms to access the backend database
Albert Gonzalez injected malformed SQL queries through Heartland's web payment forms. The injection gave him access to the backend SQL server, where he installed sniffers that captured card data as it moved through the processing pipeline. Over 130 million credit and debit card numbers were stolen — at the time, the largest data breach in history. Heartland processed 100 million transactions per month across 250,000 businesses. The breach cost Heartland over $140 million in settlements. The root cause: unsanitized input in a web form that fed directly into a SQL query string.
HANDS-ON LAB

Goal: discover, confirm, and exploit a SQL injection vulnerability using both manual and automated techniques against a deliberately vulnerable app.

  1. Start the OWASP Juice Shop (Docker: docker run -d -p 3000:3000 bkimminich/juice-shop). Navigate to the login page.
  2. Enter ' OR 1=1 -- as the email field with any password. If the login succeeds, SQLi is confirmed.
  3. Use UNION injection to extract data: ' UNION SELECT id,email,password FROM Users --. Adjust the column count until the query succeeds.
  4. Switch to the search bar — enter '; SELECT * FROM sqlite_master-- to enumerate the database schema. Note: Juice Shop uses SQLite.
  5. Run sqlmap -u "http://localhost:3000/rest/user/login" --data='{"email":"admin","password":"test"}' --dbs against a confirmed injectable endpoint.
  6. Document each finding: injection point, payload used, data extracted, and the column count method.

INTERVIEW READY
Q: "What is SQL injection and how does it work?"
A: "SQL injection is a web vulnerability where user input is concatenated directly into a SQL query string. The attacker can inject SQL syntax — like a single quote to close the string, then add OR conditions or UNION SELECT statements — to execute arbitrary queries. The root cause is always the same: the application fails to separate SQL code from user data. Parameterized queries fix it because they send the SQL structure and the data as separate channels."
Q: "How do you find SQL injection vulnerabilities?"
A: "I start by testing every user input — URL parameters, form fields, headers, cookies — with a single quote '. A database error or behavioral change indicates injection. Then I determine the injection type: UNION if I see results, error-based if errors are shown, blind if neither. I use ORDER BY or UNION SELECT null to find the column count. For blind SQLi, I use SUBSTRING and binary comparison to extract data character by character. I also run sqlmap for automation after manual confirmation."
Q: "How do you prevent SQL injection?"
A: "Parameterized queries (prepared statements) are the definitive fix — they separate SQL code from data at the protocol level, making injection structurally impossible. Input validation and escaping are secondary defenses that can be bypassed. Additional layers: least privilege for database accounts, WAF rules for defense-in-depth, and regular dependency scanning for ORM vulnerabilities."
