Intro to TryHackMe / HackTheBox
TryHackMe's Role in Security Training
// platform comparison
TryHackMe
Browser-based, guided rooms with step-by-step tasks. Best for beginners — virtual machines run in-browser. 1M+ users.
HackTheBox
VPN-based, no handholding. You get an IP and a free-form challenge. Harder, more realistic. Active machines rotate weekly.
Room Difficulty
Easy/Medium/Hard/Insane. Start with THM Pre-Security → THM Jr. Pentester → HTB Starting Point → HTB Easy boxes.
Methodology Docs
Every box solved without a walkthrough = documented methodology. Your notes become your playbook.
// commands
nmap -sC -sV -p- <target>
First command on any THM room or HTB box — full scan before anything else.
gobuster dir -u http://target -w /usr/share/wordlists/dirb/common.txt
Directory brute-forcing for web-based rooms — find hidden endpoints.
whatweb http://target
Technology fingerprinting — CMS, framework versions, JavaScript libraries in use.
curl -v http://target
Verbose HTTP request — see headers, cookies, and response body manually.
enum4linux <target>
SMB/Linux enumeration — users, shares, system info via SMB protocol.
smbclient -L //target -N
List SMB shares with null session — anonymous access is a common misconfiguration.
hydra -l admin -P /usr/share/wordlists/rockyou.txt ssh://target
Brute-force SSH login — when you have a username but no password.
// key rule

THM teaches you what to look for. HTB teaches you how to look. Both are necessary. The biggest skill: knowing when to run a walkthrough vs. when to persist. If you haven't touched a tool in 2 hours, read one hint — then continue on your own.

HTB Blue — EternalBlue in Practice
// independent solving methodology
1. Enumeration
nmap -sC -sV -p- target. Every port, every service, every version. Write everything down.
2. Vulnerability Search
Search service versions against public exploits. "SMB 3.1.1 exploit" → MS17-010 EternalBlue.
3. Exploit Matching

4. Privilege Escalation
After initial shell, check privesc vectors. HTB Blue: exploit gives SYSTEM directly — no privesc needed.

Nmap script to check if target is vulnerable to EternalBlue — safe check before exploiting.
use exploit/windows/smb/ms17_010_eternalblue
Metasploit module for EternalBlue — targets SMBv1 buffer overflow in Windows.
set PAYLOAD windows/x64/meterpreter/reverse_tcp
Set 64-bit reverse Meterpreter payload — modern Windows targets are 64-bit.

EternalBlue (MS17-010, Mar 2017): a buffer overflow in SMBv1 leaked by the Shadow Brokers from the NSA's exploit arsenal. Weaponized into WannaCry (May 2017 — 200k+ machines, 150 countries) and NotPetya (Jun 2017 — $10B+ damages). CTF Strategy — Web Category
DEF CON CTF Web Challenges
// web CTF categories
PHP Type Juggling
$key == 0 vs $key === 0. Loose comparison: "admin" == 0 is TRUE. md5("240610708") == md5("QNKCDZO") bypasses.
JWT Manipulation
Change alg from RS256 to HS256 with public key as secret. Set alg: none with empty signature. Bypass verification.
SSTI
Server-Side Template Injection. Twig: {{7*7}}, Jinja2: {{config}}, Flask: {{''.__class__.__mro__}}. Template context leaks.
GraphQL Introspection
POST {__schema{types{name,fields{name}}}}. If introspection is on, the entire data schema is readable.
SSRF
Server-Side Request Forgery. Manipulate url parameter to point at internal services: http://127.0.0.1:6379 (Redis) or file:///etc/passwd.
// commands
curl -v -c cookies.txt -b cookies.txt http://target
Preserve session cookies across requests — essential for authenticated web challenges.
sqlmap -u "http://target/page?id=1" --batch
Automated SQL injection testing — batch mode auto-answers prompts for speed.
gobuster dir -w /usr/share/wordlists/dirb/common.txt -x php,txt,html,backup
Directory fuzzing with extension filtering — find backup files and hidden pages.
jwt.io (debugger)
Decode JWTs without server interaction. Copy token, paste, inspect header/payload, modify and re-encode.
{{config}} / {{7*7}} / {{''.__class__}}
SSTI test payloads. {7*7} returns 49 = template injection confirmed. Escalate to RCE from here.
// real incident

DEF CON CTF finals feature web challenges mirroring real-world flaws — past challenges replicated Facebook Graph API flaws and Google App Engine misconfigurations. The same SSTI bugs that win CTF flags have been found in Uber, Twitter, and Shopify production systems via bug bounty programs.

CTF Strategy — Cryptography & Forensics
Crypto CTF in Real Forensics
// crypto challenge types
Classical Ciphers
Caesar (rot13), Vigenère, Atbash. Frequency analysis cracks these in seconds with Python.
XOR Analysis
Single-byte XOR: brute-force 0-255 and check English frequency. Repeating-key XOR: crib dragging and Hamming distance.
AES Decryption
Given key + IV: openssl enc -aes-256-cbc -d -in enc -out dec -K hexkey -iv hexiv. Missing one piece = dead end.
Steganography
Strings, binwalk, zsteg, steghide, exiftool. Data hidden in images, audio, or file metadata — always check LSB.
PCAP Analysis
tshark -r capture.pcap -Y "http.request" -T fields -e http.host. Reconstruct files with wireshark "follow stream".
// commands
python3 -c "import codecs; print(codecs.decode('uryyb', 'rot13'))"
Quick ROT13 decode in Python — class cipher CTF basics.
for i in range(256): print(i, ''.join(chr(ord(c)^i) for c in ciphertext))
Single-byte XOR brute-force — check output for English text to identify the key byte.
openssl enc -aes-256-cbc -d -in flag.enc -out flag.txt -K ... -iv ...
Decrypt AES-256-CBC with hex key and IV — one wrong byte = garbage output.
tshark -r capture.pcap -Y "http" -T fields -e http.request.uri
Extract all HTTP request URIs from a pcap — find exfiltrated data in web requests.
zsteg -a image.png
Automated LSB steganography detection — checks all bit planes and color channels for hidden data.
// real incident

The same XOR analysis and frequency analysis techniques used in CTF crypto challenges are applied by forensics analysts when decrypting ransomware-encrypted files. In 2021, the REvil ransomware decryptor was built using XOR key recovery — the same skill you practice on CTF crypto challenges.

SOC Structure — Tier 1/2/3 Analyst Roles
SOC Tier Model at Microsoft
// tier responsibilities
Tier 1 — Triage
Monitor SIEM dashboards. Ack/Categorize alerts. Escalate confirmed positives to Tier 2. Entry-level — SOC analysts start here.
Tier 2 — Investigation
Deep-dive into escalated incidents. Log analysis, containment, evidence collection. Determines scope and impact.
Tier 3 — Threat Hunting
Proactive search for threats before alerts trigger. Write detection rules. Reverse-engineer malware. Most senior.
Escalation Path
Tier 1: <15 min per alert. Tier 2: hours to days per incident. Tier 3: weeks-long threat hunts and rule development.
// commands

Real-time monitoring of authentication failures — Tier 1 watching for brute-force attacks.

Log Analysis Fundamentals
Uber Breach 2022 — Log Evidence
// critical log sources
auth.log / secure
Authentication events — successful logins (Accepted), failures (Failed password), sudo commands. Primary brute-force detection source.
Apache/Nginx Access Log
Every HTTP request: IP, method, path, status code, user-agent. Detect directory traversal (../), SQLi (SELECT), XSS (<script>).
Windows Event Logs
Security.evtx: 4624 (logon), 4625 (failed), 4688 (process create), 1102 (log cleared). System.evtx: service installs, driver loads.
Firewall Logs
Allowed/Denied connections. Outbound connections to known-bad IPs indicate C2 callbacks. Port scans appear as burst blocks.
// commands
cat /var/log/apache2/access.log | cut -d' ' -f1 | sort | uniq -c | sort -nr
Top IPs by request count — find web scrapers, DoS sources, or a single IP hitting many endpoints.
grep "401\|403" /var/log/apache2/access.log
Filter for unauthorized/forbidden responses — directory traversal attempts return 403/401.
grep "Accepted" /var/log/auth.log | awk '{print $9" "$11}'
Extract successful SSH logins — user and IP address. Detect unauthorized access.
wevtutil qe Security /q:"*[System[EventID=4625]]" /c:10 /f:text
Windows: query last 10 failed logon events (Event ID 4625) from Security log.
tail -f /var/log/syslog | grep -E "error|fail|denied"
Real-time syslog monitoring — filter critical keywords for live incident tracking.
// real incident

Uber Breach (Sep 2022): an attacker social-engineered a contractor through MFA fatigue, accessed Uber's internal systems, and posted proof on HackerOne. Post-breach log analysis revealed evidence in VPN authentication logs — but the attack was detected by EDR, not log review. Lesson: logs tell the story, but detection tools catch it first.

Intro to SIEM — Splunk / ELK Basics
Splunk in the SolarWinds Investigation
// SIEM components
ELK Stack
Elasticsearch (index/store), Logstash (ingest/parse), Kibana (visualize/query). Free, open-source, self-hosted.
Splunk
Enterprise SIEM. SPL (Search Processing Language) queries. Index-based, role-based access. Expensive but industry standard.
Data Ingestion
Forwarders (Splunk Universal Forwarder, Filebeat) ship logs from endpoints to SIEM. Parsing extracts fields from raw logs.
KQL / SPL
Kibana Query Language: source.ip: "10.0.0.0/8" AND event.action: "user_login". Splunk SPL: index=* sourcetype=* Failed password.
// commands
sudo systemctl start elasticsearch && sudo systemctl start kibana
Start the ELK stack locally — Elasticsearch must start before Kibana.
curl -X GET "localhost:9200/_cat/indices?v"
List all Elasticsearch indices — view ingested data sources and document counts.
curl -X POST "localhost:9200/_search" -H 'Content-Type: application/json' -d '{"query":{"match":{"event":"login"}}}'
Elasticsearch JSON query — search all indices for documents matching "login" in the event field.
index=windows EventCode=4625 user=admin | stats count by source_ip
Splunk SPL: count failed logins for admin user grouped by source IP.
event.action: "user_login" AND user.name: "admin" AND source.ip: 10.*
KQL: admin logins from private IP range — filter for normal vs. anomalous login patterns.
// real incident

SolarWinds SUNBURST (Dec 2020): the investigation relied heavily on Splunk queries identifying compromised Orion servers. A single SPL query — search index=* ImagePath=*solws* — helped thousands of organizations determine if they were affected. In a widespread supply chain attack, SIEM queries are the difference between "maybe" and "confirmed."

Writing Detection Queries
Sigma Rules at NATO Cyber Range
// detection rule formats
Sigma
YAML-based, SIEM-agnostic detection format. Write once, convert to Splunk/Elasticsearch/QRadar with sigmac. Open standard.
Splunk SPL
index=* EventCode=4688 ProcessName=*mimikatz*. Proprietary to Splunk. Pipeline syntax: search → filter → transform → output.
KQL (Kibana)
event.code: 4688 AND process.name: mimikatz.exe. Field:value syntax. Used in Elastic Security and Kibana.
Sigma → Platform
sigmac -t splunk rule.yml converts Sigma YAML to Splunk SPL. Write Sigma, deploy anywhere.
// commands
title: Mimikatz Detection
logsource: category: process_creation
detection: selection: EventID: [20, 10]
Sigma rule skeleton — YAML title, log source, detection selection for mimikatz events.
sigmac -t splunk -c tools/config/generic/splunk-windows.yml rule.yml
Convert Sigma YAML to Splunk SPL query using Windows config mapping.
index=windows EventCode=4688 Process_Name=*mimikatz* OR Process_Name=*Invoke-Mimikatz*
Splunk SPL detecting mimikatz execution via process creation event.
event.code: 4688 AND process.name: (mimikatz.exe OR Invoke-Mimikatz.ps1)
KQL equivalent — detects mimikatz process creation in Elastic Security.
// real incident

NATO's Cooperative Cyber Defence Centre of Excellence (CCDCOE) uses Sigma as the standard detection rule format, enabling cross-platform threat detection across heterogeneous SIEM deployments in its member nations. Sigma is the closest thing to a universal detection language — learning it means your rules work on any platform.

Windows Event Logs Deep Dive
NotPetya — Event Log Tampering
// critical event IDs
4624 — Logon Success
Every successful login. Logon type 2 (interactive), 3 (network), 10 (remote interactive/RDP). Track who logged in from where.
4625 — Logon Failure
Failed login attempt. Source IP + username + failure code (0xC000006D = bad password, 0xC0000064 = bad username).
4688 — Process Creation
Every process launched. Command line arguments included. Detect powershell.exe -enc, mimikatz, net user, whoami execution.
4648 — Explicit Credentials
A process ran using alternate credentials. Pass-the-hash and runas operations log here.
1102 — Log Cleared
Security log was cleared. No event is more suspicious than no events — attacker wipe attempt.
4697/7045 — Service Install
A new service was installed. Attackers often install persistence via services (AlwaysInstallElevated, malicious drivers).
// commands
wevtutil qe Security /q:"*[System[(EventID=4624)]]" /c:20 /rd:true /f:text
Query last 20 successful logon events in reverse chronological order.
Get-WinEvent -FilterHashtable @{LogName='Security';ID=4688} | Select-Object -First 10
PowerShell: last 10 process creation events with full command line details.
wevtutil cl Security
Clear the Security event log — executed by attackers (and NotPetya) to destroy forensic evidence.
Get-WinEvent -FilterHashtable @{LogName='Security';ID=4648} | Format-List
Detect explicit credential usage — identify pass-the-hash and secondary logon activity.
// real incident

NotPetya (Jun 2017): the 'mischa' component executed wevtutil cl commands to clear Windows Event Logs across 80% of infected systems, destroying forensic evidence of its execution timeline. Combined with EternalBlue for propagation and Mimikatz for credential theft, NotPetya caused over $10B in damages — the most costly cyberattack in history. The log clearing itself became a detection signal in later attacks.

// interview answer
Q: If you see Event ID 1102 in the Security log, what's your immediate next step?
1102 means someone cleared the Security event log. First: check other logs (System, Application, PowerShell operational) — they may still have evidence. Second: check Event ID 1100 (log service shutdown) before the clear — the service was stopped, then the log was wiped, suggesting an attacker. Third: correlate with 4688 events from before the clear to see what was running. The absence of logs is still data.

// Quick Self-Quiz — Can You Answer These Without Looking?

Cover the answers. Say them out loud. If you can't answer one fluently, revisit that day before continuing.

Q01
What is the difference between TryHackMe and HackTheBox in terms of learning approach?
TryHackMe is guided — browser-based rooms with step-by-step tasks, best for beginners. HackTheBox is free-form — you get an IP, no instructions, must enumerate and solve independently. Both are necessary: THM teaches what to look for, HTB teaches how to look.
Q02
What is EternalBlue, and what major attacks used it?
MS17-010 — a buffer overflow in SMBv1. Used by WannaCry (May 2017, 200k+ machines, 150 countries) and NotPetya (Jun 2017, $10B+ damages). Exploit leaked from NSA. Safe practice on HTB "Blue" machine.
Q03
How does a JWT algorithm confusion attack work?
Change the JWT header from RS256 (asymmetric) to HS256 (symmetric). Some libraries use the public key as the HMAC secret. Since the public key is often exposed, you can forge tokens by signing with the leaked public key.
Q04
You see {{7*7}} rendered as 49 in a web page. What have you found?
Server-Side Template Injection (SSTI). 49 confirms the template engine evaluated your input. From here, escalate to read config ({{config}} in Flask), list class hierarchy ({{''.__class__.__mro__}}), and achieve RCE.
Q05
What are the three SOC tiers, and what does each do?
Tier 1: triage — monitor SIEM, ack/categorize alerts, escalate confirmed positives. Tier 2: investigation — deep log analysis, containment, scope determination. Tier 3: threat hunting — proactive searches, detection rule writing, reverse engineering.
Q06
How would you identify a brute-force attack using auth.log?
grep "Failed password" /var/log/auth.log | awk '{print $11}' | sort | uniq -c | sort -nr. Multiple failed logins from a single IP in quick succession = brute-force. Pair with Accepted entries from the same IP to identify successful breaches.
Q07
What is the purpose of Sigma rules, and why does NATO use them?
Sigma is a YAML-based, SIEM-agnostic detection rule format. Write once in Sigma, convert to Splunk SPL, KQL, QRadar AQL, etc. with sigmac. NATO's CCDCOE uses them because member nations run different SIEMs — Sigma ensures cross-platform threat detection.
Q08
What does Event ID 1102 mean, and what should you do if you find it?
1102 = Security log was cleared. Immediately check other logs (System, Application, PowerShell) for remaining evidence. Check for Event ID 1100 (log service shutdown) before 1102. Correlate with 4688 events from before the clear. The absence of logs is itself a detection signal.
Q09
How did Splunk queries help in the SolarWinds investigation?
Security teams ran the query "index=* ImagePath=*solws*" to identify compromised Orion servers. Since SUNBURST was a supply chain attack affecting thousands of organizations, this single SPL query became the fastest way to determine infection status across different environments with Splunk.
Q10
How did NotPetya destroy forensic evidence, and what was the detection irony?
NotPetya's 'mischa' component ran "wevtutil cl" to clear Windows Security Event Logs across 80% of infected systems. The irony: after NotPetya, security teams began monitoring for mass Event ID 1102 events as a detection signal — so the log clearing itself became a way to detect attacks.
