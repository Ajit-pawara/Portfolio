Intrusion Detection — Snort / Suricata

Intrusion Detection Systems (IDS) and Intrusion Prevention Systems (IPS) are the burglar alarms of the network world. Snort and Suricata are the gold-standard open-source engines — together they power the majority of the world's network security monitoring deployments.

THEORY

What is an IDS/IPS? An Intrusion Detection System monitors network traffic for suspicious patterns and generates alerts. An Intrusion Prevention System does the same but blocks traffic inline. Snort (1998) and Suricata (2010) are rule-based engines that match packet content against signatures — also called rules — to detect attacks.

Snort rule structure — every rule has two parts:

# Rule header + rule options alert tcp $EXTERNAL_NET any -> $HOME_NET 80 (msg:"SQL Injection - UNION SELECT detected"; flow:to_server,established; content:"UNION"; nocase; content:"SELECT"; nocase; distance:0; sid:1000001; rev:1;)

Breaking down the rule:

Rule Header
Action (alert), Protocol (tcp), Source IP/port, Direction (->), Destination IP/port. The rule fires when traffic matches ALL header conditions.
Rule Options
msg — alert text; content — pattern to match in payload; nocase — case-insensitive match; sid — unique rule ID; rev — revision number.
Flow Keyword
flow:to_server,established — only inspect packets in the server-bound direction of an established TCP connection. Reduces false positives from outbound server responses.
PCRE (Perl-Compatible Regex)
pcre:"/(?:UNION\s+ALL\s+SELECT|UNION\s+SELECT)/i" — regex matching for complex patterns that content keywords cannot express.

Detection Mode vs Prevention Mode:

Detection (IDS): Snort/Suricata receives a copy of the traffic (via SPAN port or TAP). It can only alert — the original traffic passes through unaffected. Zero risk of blocking legitimate traffic. Prevention (IPS): The engine sits inline in the traffic path. When a rule matches, it can drop, reject, or sdrop the packet. Must be tuned aggressively to avoid blocking legitimate traffic.

Suricata advantages over Snort: Multi-threaded by default (Snort is single-threaded), native GPU acceleration via CUDA, built-in HTTP/ TLS/ DNS parsers (app-layer inspectors), and support for emerging protocols like MQTT and HTTP/2. Both use nearly identical rule syntax, so rules are portable between them.

Key insight: Rule quality matters more than rule quantity. The Emerging Threats Open rule set ships ~35,000 rules; a tuned production deployment typically enables only 3,000-5,000. Every enabled rule is a source of potential false positives — tune ruthlessly.
// attacker vs defender mindset
⚔ Attacker
  • Evade signature detection by encoding payloads: URL-encode SQLi (%27%20OR%201%3D1), base64-encode shellcode, use UTF-16 or gzip compression in HTTP requests.
  • Fragment packets across multiple TCP segments — a single rule match may span two packets that the IDS fails to reassemble properly.
  • Use encrypted channels (HTTPS, DNS-over-HTTPS, SSH tunnels) to bypass content inspection entirely — the IDS sees only the encrypted wrapper.
  • Slow down automated scans to 1-2 packets per minute to evade rate-based threshold rules that trigger on 10+ connections per second.
🛡 Defender
  • Enable preprocessors in Snort or app-layer parsers in Suricata — they normalize traffic (de-fragment IP, re-assemble TCP, decode HTTP) before rule matching.
  • Write threshold rules to suppress noisy-but-harmless alerts: threshold: type both, track by_src, count 5, seconds 60 — only alert if a source triggers 5 times in 60 seconds.
  • Run rules against a pcap of known-good traffic first — identify and exclude rules that generate false positives in your environment before going live.
  • Use fast_pattern to optimize rule matching — Snort/Suricata uses the longest content match as the initial filter, rejecting non-matching packets early.

Snort powers the Top 10 IDS deployments globally — from NSA to Fortune 50 SOCs
Created by Martin Roesch in 1998, Snort became the de facto standard for network intrusion detection. By 2003 it had over 5 million downloads and a community of 40,000+ rules covering everything from SQL injection to zero-day exploit signatures. Snort rules detected the SQL Slammer worm (2003), the Blaster worm, and Sasser within hours of outbreak. Today, Snort 3 (written in C++) delivers 10x performance over Snort 2. Suricata, created by the OISF in 2010, emerged as the multi-threaded successor and now powers most modern IDS deployments, including AWS Shield and many MSSP platforms. Both engines remain actively developed — Snort by Cisco, Suricata by the OISF community — and both share the same rule language, making skills 100% transferable between them.
HANDS-ON LAB

Goal: install Suricata, write custom detection rules, generate attack traffic, and verify alerts in the unified2 alert log.

  1. Install Suricata: sudo apt install suricata -y. Verify with suricata --build-info. The default rule path is /etc/suricata/rules/.
  2. Create a custom rule file: sudo nano /etc/suricata/rules/local.rules. Add: alert http $HOME_NET any -> $EXTERNAL_NET any (msg:"C2 Beacon — Nslookup to Suspicious Domain"; content:"|0d 0a|Host|3a 20|evil"; nocase; sid:2000001; rev:1;)
  3. Test the rule syntax: suricata -T -c /etc/suricata/suricata.yaml -S /etc/suricata/rules/local.rules. Fix any syntax errors before proceeding.
  4. Run Suricata in IDS mode on an interface: sudo suricata -c /etc/suricata/suricata.yaml -i eth0. Generate attack traffic: visit a test site and trigger an HTTP request.
  5. Check alerts in the fast log: sudo tail -f /var/log/suricata/fast.log. Each alert shows the rule message, source/dest IPs, and timestamp.
  6. Write a threshold rule to suppress excessive alerts from one source: event_filter: gen_id 1, sig_id 2000001, type threshold, track by_src, count 3, seconds 60.
Deliverable: A working Suricata rule that fires on a specific HTTP request, a screenshot of the alert in fast.log, and a threshold filter configured to reduce alert noise. You're now writing production-grade IDS rules.

INTERVIEW READY
Q: "Explain the difference between IDS and IPS with Snort/Suricata."
A: "IDS mode — also called promiscuous mode — receives a copy of the traffic via a SPAN port or network TAP. The engine can only alert; it can't block. IPS mode — also called inline mode — sits directly in the traffic path, and the engine can drop, reject, or pass packets. IPS requires the af-packet inline mode in Suricata or nfqueue in Snort. The same rules work in both modes, but in IPS mode the action changes from alert to drop for malicious signatures."
Q: "How do you tune Snort rules to reduce false positives?"
A: "First, I run the full rule set against a pcap of known-good traffic and identify which rules fire on benign traffic. Those get either disabled or wrapped in a threshold or suppress filter. Second, I use flow:established to avoid inspecting handshake packets. Third, I create custom pass rules for known-safe IPs. Fourth, I use Suricata's app-layer parsers — the HTTP parser decodes the request before rule matching, so URL-encoded attacks are caught without needing dozens of encoding variant rules."
Q: "How do attackers evade signature-based IDS?"
A: "Common techniques: payload encoding (base64, UTF-16, URL encoding), packet fragmentation (IP fragments or TCP segmentation that the IDS doesn't reassemble), encryption (HTTPS/SSH tunnels), and polymorphic shellcode that changes its byte signature every infection. Modern engines counter evasion with preprocessors (frag3, stream5 in Snort) that normalize traffic before rule matching, and with app-layer inspectors that decode the protocol (HTTP, TLS, DNS) before looking for patterns in the decoded content."
AI TOOLS
Snort rule generator
Describe an attack in plain English: "detect directory traversal in HTTP requests to Apache servers" — get a ready-to-use Snort rule with proper content matching and sid numbering.
False positive analyzer
Paste a fast.log alert entry into an LLM and ask it to determine whether the alert indicates a real attack or a false positive based on the IP, port, and payload pattern.
PCRE rule optimizer
Ask an LLM to convert a complex PCRE-based Snort rule into an equivalent multi-content rule — PCRE is CPU-intensive and should be the last resort, not the first.
Rule set auditor
Paste your enabled rule list and ask for a review: which rules are redundant, which have known bypasses, and which should be prioritized in your environment.
