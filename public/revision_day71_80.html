Security Policies & Compliance (ISO 27001, GDPR)
Meta GDPR Fine Euro1.2B (2023)
// core concepts
ISO 27001 ISMS
Information Security Management System framework. 93 controls in 2022 revision across 4 domains. Certification requires external audit every 3 years.
GDPR Data Subject Rights
Right to access, rectification, erasure, data portability, object to processing. 72-hour breach notification. Fines up to 4% of global revenue.
Annex A Controls (2022)
93 controls: Organizational (37), People (8), Physical (14), Technological (34). New: threat intelligence, cloud security, physical security monitoring.
DPA / DPO
Data Processing Agreement defines vendor data handling. Data Protection Officer mandated for large-scale personal data processing under GDPR Art. 37.
// the real incident

Meta Euro1.2B GDPR Fine (May 2023): The Irish Data Protection Commission fined Meta Euro1.2B for violating GDPR data transfer requirements — sending EU user data to the US under Standard Contractual Clauses that the Court of Justice ruled inadequate via the Schrems II decision. This is the largest GDPR fine in history. The fine was not for a data breach but for a compliance failure in cross-border data transfers. Compliance risk can exceed breach risk financially.

// commands / key docs
NIST SP 800-53 Rev. 5
1,000+ security and privacy controls mapped to ISO 27001. Crosswalk for FedRAMP compliance.
GDPR Article 33 — Breach Notification
Mandatory 72-hour notification. Document everything — delay justification must be provable.
ISO 27001:2022 Clause 6.1
Risk treatment: define acceptance criteria, identify risks, evaluate options (mitigate/accept/transfer/avoid), produce Statement of Applicability.
// interview answer
Q: What is the difference between a security policy and a security control?
A policy is a high-level document stating management intent (all data at rest must be encrypted). A control is the specific mechanism that implements it (AES-256 via BitLocker on all laptops). ISO 27001 requires policies AND controls — the policy without a control is just a wish.

Risk Assessment Basics
JPMorgan Chase — $600M+ Annual Budget
// NIST SP 800-30 methodology
Asset Identification
List everything of value: customer data, IP, systems, facilities. Risk assessment starts when you know what you are protecting.
Threat + Vulnerability = Risk
Threat (ransomware group) + Vulnerability (unpatched SMB) = Risk. No vulnerability = no risk regardless of threat.
Inherent vs Residual Risk
Inherent = risk before controls. Residual = risk after controls. Your job is to reduce residual risk to the acceptable level.
Risk Treatment Options
Mitigate (implement control), Accept (management sign-off), Transfer (cyber insurance), Avoid (discontinue activity).
// the real incident

JPMorgan Chase (2014-Present): After the 2014 breach affecting 76M households, JPMorgan increased annual cybersecurity spending to $600M+. Risk assessment frameworks drive budget allocation: highest-risk assets (payment systems, customer PII, SWIFT) receive the most investment. The bank calculates Annualized Loss Expectancy (ALE) for each threat scenario to justify spending. This is risk-based budgeting at enterprise scale.

// commands / risk formulas
ALE = SLE x ARO
Annualized Loss Expectancy = Single Loss Expectancy x Annual Rate of Occurrence. Quantitative risk in dollars.
SLE = AV x EF
Single Loss Expectancy = Asset Value x Exposure Factor. If $1M server has 50% exposure, SLE = $500K.
NIST SP 800-30 Rev. 1
Guide for risk assessments. 9-step process from system characterization to monitoring.
FAIR Model
Factor Analysis of Information Risk — decomposes risk into Loss Event Frequency and Loss Magnitude. More granular than NIST qualitative.
// interview answer
Q: Explain qualitative vs quantitative risk assessment.
Qualitative uses ordinal scales (Low/Medium/High/Critical) based on expert judgment — faster but subjective. Quantitative uses hard numbers (ALE in dollars, probability percentages) — more defensible but requires data. Most orgs do qualitative for quick decisions and quantitative for major investments. Both are valid under NIST 800-30.

Zero Trust Architecture Concept
Google BeyondCorp Implementation
// zero trust pillars
Identity Verification
Every request authenticated, not just perimeter. MFA required for every access. No implicit trust based on network location.
Device Posture Check
Device must meet baseline (OS patched, AV running, disk encrypted). Non-compliant devices blocked regardless of user identity.
Micro-Segmentation
Network segmented into tiny zones. Lateral movement requires re-authentication at each boundary. Unlike flat networks where one breach = total compromise.
Least Privilege Access
Minimum permissions for minimum time (JIT). No standing admin rights. Every elevated access logged and time-bound.
// the real incident

Google BeyondCorp (2009-2017): Google eliminated the corporate VPN for 100K+ employees. No more trusted internal network. Access decisions based on device inventory, user identity, and real-time posture — not IP address. Employees access internal apps from any network (office, home, coffee shop) with identical security controls. 100K+ daily authentications, zero VPN infrastructure to attack. Now the industry standard.

// commands / implementation
NIST SP 800-207
Zero Trust Architecture framework. Defines PDP (Policy Decision Point), PEP (Policy Enforcement Point).
iptables -A FORWARD -i eth1 -o eth2 -j DROP
Micro-segmentation — drop inter-VLAN traffic to prevent lateral movement.
OPA rego policy
Policy-as-code: allow { input.user.role == "admin"; input.device.encrypted == true }
// interview answer
Q: How does Zero Trust differ from traditional perimeter security?
Traditional security trusts everything inside the network (castle-and-moat). Once past the firewall, you are trusted. Zero Trust trusts nothing — every access request, even from inside the building, must authenticate, be authorized, and pass device posture checks. The network is always considered hostile. Google does not trust its own office Wi-Fi.

DevSecOps & Secure SDLC Intro
Etsy — Continuous Security Pipeline
// core concepts
Shift Left
Find security issues earlier. A vuln found in production costs 100x more than one caught in IDE. Test at commit, not at deploy.
SAST vs DAST
SAST analyzes source code without running (finds SQLi, XSS, hardcoded secrets). DAST tests the running app (finds runtime issues like misconfigured headers).
SCA (Software Composition Analysis)
Scans open-source dependencies for known CVEs. OWASP Dependency-Check, Snyk, Dependabot. Modern apps are 80%+ open source.
Pre-Commit Hooks
Scan for secrets before code reaches repo. gitleaks, truffleHog, detect-secrets — catch credentials before they hit git history.
// the real incident

Etsy DevSecOps Program: Etsy automated 23 security checks into their CI/CD pipeline — SAST, SCA, secret scanning, container scanning, IaC validation. Result: vulnerability discovery-to-patch time dropped from weeks to under 24 hours. A developer commits code with a vulnerable library, the pipeline blocks the build, the developer fixes it within hours, and safe code deploys. Security becomes part of velocity, not a bottleneck.

// commands
semgrep --config=auto src/
Run SAST scan — auto-detect language and apply community rules.
dependency-check --scan target/*.war --format HTML
OWASP Dependency-Check SCA scan — finds CVEs in dependencies.
gitleaks detect --source . --verbose
Scan git repo for hardcoded secrets in commit history.
trivy image --severity HIGH,CRITICAL myapp:latest
Scan container image for high and critical CVEs.
// interview answer
Q: Your DAST finds reflected XSS in staging but SAST missed it. Why?
SAST analyzes source code without execution — it can miss vulns introduced at runtime through framework templating, dynamic includes, or server-side rendering. DAST tests the running app and catches issues SAST cannot see. Both are necessary; they find different bug classes. Mature DevSecOps runs SAST on commit, DAST on staging deploy, and SCA on every build.

// the 90-day roadmap mindset

Generalists fill entry-level roles. Specialists fill senior roles at 2x+ compensation. The key is picking one path and committing 90 days of focused learning. Pivoting after 90 days is fine. Flipping every month is career suicide. For SOC: SIEM queries + Windows logs. For pentesting: AD exploitation + web testing. For cloud: AWS/Azure certs + Terraform.

// interview answer
Q: I am interested in both red and blue team. Which should I pick?
Start in a SOC (blue team). SOC experience teaches you how attacks are detected and what defenders look for — this directly makes you a better pentester. Most top pentesters spent 2-3 years in SOC first. Purple team roles exist but are rare and usually senior. Path: SOC (2 years) then specialize based on what energizes you.

Security+ — Domains 1 & 2 Deep Review
Security+ PBQ Case Study Analysis
// domain 1 — attacks, threats & vulnerabilities
Phishing Variants
Spear (targeted), whaling (C-suite), vishing (voice), smishing (SMS). Spear phishing = 65% of initial access in breaches.
Malware Types
Virus (attaches to files), worm (self-propagating, WannaCry), trojan (Dridex), ransomware, rootkit (kernel level), spyware.
Application Attacks
SQLi (injection via input), XSS (stored/reflected/DOM), CSRF (forged requests), buffer overflow (memory overwrite — code execution).
Crypto Attacks
Birthday (hash collision), rainbow tables (pre-computed chains), downgrade (force weaker protocol), collision.
// domain 2 — architecture & design
Defense in Depth
Layered: perimeter firewall → network segmentation → endpoint protection → app controls → data encryption. If one layer fails, next catches it.
Secure Network Design
DMZ for public servers, VLAN segmentation, NAT for IP hiding, HSM for cryptographic key storage.
Virtualization Security
Type 1 (bare metal) vs Type 2 (hosted) hypervisor. VM escape, snapshot security, vMotion encryption.
Cloud Security Models
IaaS (you secure OS+), PaaS (you secure app+), SaaS (you secure usage). Shared responsibility model.
// PBQ simulation

PBQ Example: A company suffered a phishing attack. Configure the firewall to: (1) block outbound to C2 IP 203.0.113.5, (2) allow HTTPS, block HTTP outbound, (3) deny inbound from all countries except US. Rule ordering: deny specific before allow general. C2 block first, then country block, then HTTPS allow. iptables: iptables -A OUTPUT -d 203.0.113.5 -j DROP

Real PBQ Scenario — IR Walkthrough
// domain 3 — implementation
PKI & Certificates
CA issues certs. CRL/OCSP check revocation. Wildcard certs (*.example.com). Certificate pinning for high-security apps.
Secure Protocols
SSH (22, replaces Telnet), TLS 1.3 (443, replaces SSL), SFTP, IPsec (VPN), DNSSEC, SNMPv3 (encrypted).
Identity & Access Management
SSO (SAML/OAuth), MFA (know/have/are), PAM (JIT admin), Kerberos (AD auth), RADIUS (network auth).
Host Security Controls
SELinux (MAC), Windows Defender (AV+EDR), host firewall, CIS benchmarks for secure baselines.
// domain 4 — operations & IR
NIST SP 800-61 IR Process
Preparation → Detection → Containment → Eradication → Recovery → Lessons Learned. Know this order for PBQs.
IR PBQ Walkthrough
Compromised web server: 1) Disable account, 2) Collect memory, 3) Preserve logs, 4) Scan for lateral movement, 5) Patch vuln, 6) Restore from backup.
Forensic Procedures
Order of volatility: register → cache → RAM → process table → disk. Hash verification, chain of custody, write blocker.
Data Sovereignty
Data stays in jurisdiction. Classification: Public, Internal, Confidential, Restricted. Handling procedures per level.
// interview answer
Q: A PBQ gives you six response steps for a compromised server. How do you order them?
Step 1: Disable compromised account (immediate containment). Step 2: Capture memory dump (preserve volatile evidence). Step 3: Preserve logs (copy to external media). Step 4: Scan other servers for lateral movement (scope assessment). Step 5: Patch the exploited vulnerability. Step 6: Restore from clean backup. Always preserve evidence before taking remediation actions.

// interview answer
Q: What is the difference between RTO and RPO, and why do both matter?
RTO (Recovery Time Objective) = how long until systems are back online. RPO (Recovery Point Objective) = how much data loss is acceptable (in time). Example: RTO of 4 hours means restored within 4 hours. RPO of 1 hour means losing up to 1 hour of data is OK. RTO drives infrastructure cost (hot site = expensive). RPO drives backup frequency (hourly backups = more storage).

// Quick Self-Quiz — Can You Answer These Without Looking?

Cover the answers. Say them out loud. If you cannot answer one fluently, revisit that day before continuing.

Q01
Why was Meta fined Euro1.2B under GDPR, and what does it teach about compliance risk?
Meta violated GDPR data transfer requirements (sending EU data to US under invalidated mechanisms per Schrems II). The lesson: compliance failure fines can exceed breach-related fines. Compliance risk is financial risk.
Q02
What is the difference between inherent risk and residual risk?
Inherent risk is the risk level before any controls are applied. Residual risk is what remains after controls are implemented. The goal of security is to reduce residual risk to the organization's acceptable level. If residual risk remains above the threshold, additional controls or risk acceptance are needed.
Q03
Google BeyondCorp eliminated VPNs entirely. What replaced them for access control?
Access decisions based on device identity (inventory database), user identity (SSO/MFA), and real-time device posture (OS patch level, AV status, disk encryption). No network location-based trust. Every request authenticated regardless of origin.
Q04
What is the difference between SAST and DAST? When would you use each?
SAST (Static) analyzes source code without execution — finds issues early in IDE/commit. DAST (Dynamic) tests the running application — finds runtime issues. SAST on every commit, DAST on staging deploy. They find complementary bug classes.
Q05
How did Etsy reduce vulnerability discovery-to-patch time from weeks to 24 hours?
Automated 23 security checks into their CI/CD pipeline: SAST, SCA secret scanning, container scanning, IaC validation. Vulnerable builds are blocked automatically. Developers fix before deploy. Security is integrated into velocity, not a manual gate.
Q06
What are the five CompTIA Security+ domains and their percentage weights?
D1 Attacks/Threats 24%, D2 Architecture/Design 21%, D3 Implementation 25%, D4 Operations/IR 16%, D5 Governance/Risk/Compliance 14%. Total 100%. Passing score 750/900.
Q07
A Security+ PBQ asks you to order incident response steps. What is the correct sequence?
1) Disable account (containment). 2) Capture memory (volatile evidence). 3) Preserve logs. 4) Scan for lateral movement. 5) Patch vulnerability. 6) Restore from clean backup. Always preserve evidence before remediation.
Q08
What is RTO vs RPO? Give an example.
RTO = Recovery Time Objective (how long to restore — 4 hours). RPO = Recovery Point Objective (acceptable data loss in time — 1 hour). RTO drives DR infrastructure cost. RPO drives backup frequency.
Q09
What four cybersecurity career paths were compared and what were their average salaries?
SOC Analyst ($65-95K), Penetration Tester ($150K), Cloud Security Architect ($180K), AppSec Engineer ($165K). Specialization significantly impacts earning potential.
Q10
What was JPMorgan Chase's 2014 breach impact and how did their response use risk assessment?
76M households affected. Response: $600M+ annual cybersecurity budget driven by risk assessment frameworks. Highest-risk assets (payment systems, PII, SWIFT) get proportional investment. Quantitative risk (ALE/SLE/ARO) calculations justify every dollar.
