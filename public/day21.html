DAY 21 OF 90 — PHASE 2: CORE SECURITY CONCEPTS

OWASP Top 10 & Web App Security Fundamentals

The single most-referenced list in appsec interviews — and the foundation every bug bounty hunter, pentester, and secure developer builds on. Today you stop reciting it like a poem and start using it as a mental model.

THEORY

What it is: A standard awareness document ranking the most critical web application security risks, based on real-world data from hundreds of organizations and thousands of applications. Maintained by the OWASP Foundation, updated periodically — 2025 is the current edition, following 2021 and 2017.

Why it matters: It's the shared vocabulary between developers, pentesters, and security teams. When an interviewer asks "walk me through a SQL injection" or "how would you secure a login form," they're testing whether you know this list cold.

The 10 categories (2021/2025 lineage):

01Broken Access Control — users acting outside intended permissions (e.g. changing a URL parameter to view someone else's account).
02Cryptographic Failures — weak or missing encryption exposing sensitive data (ties to Day 19's crypto lesson).
03Injection — untrusted input executed as code (SQL injection, command injection, XSS).
04Insecure Design — flaws baked into the architecture itself, not just the code (new emphasis since 2021).
05Security Misconfiguration — default credentials, open cloud buckets, verbose error messages leaking stack traces.
06Vulnerable & Outdated Components — libraries/frameworks with known CVEs — why dependency scanning exists.
07Identification & Authentication Failures — weak password policies, session fixation, missing MFA (builds on Day 20).
08Software & Data Integrity Failures — trusting unsigned updates/plugins — exactly what caused SolarWinds.
09Security Logging & Monitoring Failures — breaches going undetected for months because no one was watching the logs.
10Server-Side Request Forgery (SSRF) — tricking a server into making requests to internal systems it shouldn't reach.

Mental model — three buckets, not ten flashcards:

Bad Input Handling
Injection — trusting data that came from outside the app.
Bad Access Decisions
Access Control + Authentication — who you are vs. what you're allowed to touch.
Bad Hygiene
Misconfig, outdated components, missing logging — the boring stuff that still breaches you.
The interview tell: Every real breach maps cleanly to one of these three buckets. Naming the bucket before the category number is what separates "memorized the list" from "understands the model."
// real named incident
2017 — EQUIFAX BREACH — ~147 MILLION PEOPLE
A known, unpatched Apache Struts CVE — Vulnerable and Outdated Components, textbook
Equifax was running Apache Struts with a publicly disclosed remote code execution vulnerability (CVE-2017-5638). A patch existed for months before the breach. Attackers exploited the unpatched framework to gain access and exfiltrate names, SSNs, birth dates, addresses, and driver's license numbers for roughly 147 million people. The root cause maps directly to today's "Vulnerable and Outdated Components" category — the fix wasn't a new feature, it was applying a patch that already existed. This is why dependency scanning (Snyk, Dependabot, OWASP Dependency-Check) exists as a standard practice today.
HANDS-ON LAB

Goal: identify and exploit a Broken Access Control flaw and a basic Injection flaw on a deliberately vulnerable app.

  1. Spin up OWASP Juice Shop locally with Docker — a deliberately vulnerable app built to practice exactly this list.
  2. Test Broken Access Control: log in as a normal user, note your basket/order ID in the URL, then try incrementing it. Viewing another user's basket confirms an IDOR.
  3. Test basic Injection on the login form using a classic SQL injection payload in the email field.
  4. Document your finding like a real pentest report — target, severity, proof-of-concept, one-line remediation.
Deliverable: a short written finding (target, severity, PoC, remediation) from today's lab — this is the artifact that stands out in campus placement interviews, because most classmates can recite the list but few can show a documented finding.
# Step 1 — spin up OWASP Juice Shop (vulnerable-by-design app) docker pull bkimminich/juice-shop docker run -d -p 3000:3000 bkimminich/juice-shop # Step 2 — Broken Access Control test # log in, note your basket ID in the URL: /rest/basket/2 # try /rest/basket/3, /rest/basket/4 ... # viewing another user's basket = Broken Access Control confirmed # Step 3 — basic Injection test on the login form Email: ' OR 1=1-- Password: anything # logging in without valid credentials = SQL Injection confirmed # Step 4 — document like a real pentest report echo "Finding: IDOR on /rest/basket/{id} — severity: High" >> findings.txt
INTERVIEW READY
Q: "What's the difference between authentication failures and access control failures?"
A: "Authentication failures happen at the 'who are you' step — weak passwords, no MFA, session hijacking. Access control failures happen after login, at the 'what are you allowed to do' step — IDOR, privilege escalation. One is about proving identity, the other is about enforcing boundaries once identity is known."
Q: "How would you test for SQL Injection manually before running a tool?"
A: "Insert a single quote into an input field and watch for a database error message — that's the first signal the input isn't sanitized, before I'd reach for sqlmap or Burp."
Q: "What's an IDOR?"
A: "Insecure Direct Object Reference — when an app exposes an internal ID, like an order number, directly in a URL or parameter, without checking whether the requesting user actually owns that object."

ENTREPRENEUR ANGLE

The OWASP Top 10 is the backbone of every appsec consulting engagement. When you eventually run your own security consulting firm, this list becomes your default audit checklist for client web apps — it's vendor-neutral, free, and universally recognized, so clients trust an assessment structured around it without you having to justify the framework.

It's also the basis for a productized service: an "OWASP Top 10 Quick Audit" — a fixed-scope, fixed-price engagement (3–5 days) that's easy to sell to small businesses who can't afford a full pentest but need to tell their own customers "we got assessed against OWASP standards."

If you can do this cold, you've actually learned it — that's the 10% "teaching" slice of the 70/20/10 rule.
