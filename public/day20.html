DAY 20 OF 90 — PHASE 2: CORE SECURITY CONCEPTS

Password Security & Hashing Attacks

The rockyou.txt wordlist sitting in /usr/share/wordlists on your Kali machine right now came from a 2009 breach of a company called RockYou. They stored 32 million passwords in plaintext. Today you use that wordlist to crack hashes — and understand the full attack chain from breach to cracked credential.

THEORY

Attack types — how passwords are actually cracked:

Dictionary Attack
Hash every word in a wordlist (rockyou.txt = 14M passwords) and compare against the target hash. Fast, covers the most common passwords. Most breaches fall to this first.
Brute Force
Try every possible character combination up to a given length. Guaranteed to find the password eventually — but exponentially slow for long/complex passwords.
Rainbow Table Attack
Pre-computed hash → plaintext lookup tables. Instant cracking for unsalted hashes — no computation needed at attack time, just a lookup. Defeated entirely by salting.
Rule-Based Attack
Apply transformation rules to wordlist entries — capitalize first letter, append numbers, substitute letters with symbols. "password" → "P@ssw0rd1!" — covers how real people choose passwords.
Mask Attack
Define a pattern (e.g., 8 chars, starts uppercase, ends 2 digits) and only try combinations matching that pattern. Faster than pure brute force when you know the policy.
Credential Stuffing
Use username:password pairs from one breach to try other services. Works because 65%+ of users reuse passwords. Not hash cracking — uses already-cracked credentials.

Why algorithm choice determines cracking speed:

MD5 — ~60 billion/sec
On a modern GPU, Hashcat tries 60B MD5 hashes per second. rockyou.txt (14M entries) finishes in milliseconds. MD5 is effectively instant to crack.
SHA-256 — ~10 billion/sec
Still very fast. Without a salt, common passwords fall in seconds. LinkedIn used unsalted SHA-1 in 2012 — cracked within days of the breach.
bcrypt (cost 12) — ~25k/sec
~2.4 million times slower than MD5 per attempt. rockyou.txt would take ~9 minutes instead of milliseconds. Significantly harder to crack at scale.
Argon2id — memory-hard
Not just slow but requires large memory per attempt — GPUs can't parallelize effectively. The current best practice for password storage. Defeats GPU-based cracking at scale.
Hashcat modes you must know:
-m 0 = MD5 | -m 100 = SHA-1 | -m 1400 = SHA-256 | -m 3200 = bcrypt | -m 1800 = sha512crypt (Linux /etc/shadow)
-a 0 = dictionary | -a 1 = combination | -a 3 = brute force (mask) | -a 6 = wordlist+mask hybrid
John the Ripper vs Hashcat: John auto-detects hash types and is great for quick attempts and formats Hashcat handles poorly. Hashcat is GPU-accelerated and vastly faster for large-scale dictionary/rule attacks. In practice: use John to identify an unknown hash type, use Hashcat to crack at speed.
// real named incident
2009 — ROCKYOU BREACH — 32 MILLION PASSWORDS IN PLAINTEXT
The breach that created the wordlist you're using today — stored in plaintext, no hashing at all
RockYou was a social app platform popular in 2009 for Facebook and MySpace widgets. Attackers discovered an SQL injection vulnerability in their site and dumped the entire user database — 32,603,388 accounts with passwords stored in complete, unencrypted plaintext. Not weak hashing. Not unsalted hashing. No hashing at all. The dumped password list was cleaned, sorted, and released publicly — becoming the "rockyou.txt" wordlist that ships with every Kali Linux installation at /usr/share/wordlists/rockyou.txt. Every Hashcat tutorial, every TryHackMe hash-cracking room, every CTF password challenge uses this list as the primary dictionary. When you run hashcat -a 0 hash.txt rockyou.txt today, you are using a dataset produced by the privacy failure of 32 million real people in 2009. The top 10 passwords in the list: 123456, 12345, 123456789, password, iloveyou, princess, 1234567, rockyou, 12345678, abc123 — in that order. This is why password policies exist.
// attacker vs defender mindset
⚔ Attacker
  • Dictionary attack first — 90% of real-world hashes from leaked databases fall to rockyou.txt in under a minute for MD5/SHA-1.
  • Rule attacks next — most people append numbers or substitute letters. Rules like best64.rule catch this pattern without brute-forcing the full space.
  • Hash identification is step zero — use hash-identifier or hashid to determine the algorithm before picking the Hashcat mode (-m).
🛡 Defender
  • Use bcrypt or Argon2 — the speed difference against an attacker is the point. 25,000/sec vs 60B/sec is the margin between "practical to crack at scale" and "not worth the resources."
  • Enforce password length over complexity — "correcthorsebatterystaple" (4 random words) is stronger than "P@ssw0rd!" and easier to remember. Length matters more than symbols.
  • Check new passwords against known breach databases (HaveIBeenPwned API) — if a user tries to set "password123", you know it's in rockyou.txt and should reject it.

# Step 0: identify an unknown hash type echo "48bb6e862e54f2a795ffc4e541caed4d" | hash-identifier Possible Hashs: [+] MD5 # MD5 = 32 hex chars, SHA-1 = 40, SHA-256 = 64, bcrypt starts with $2y$ # Dictionary attack against MD5 hash (-m 0) echo "48bb6e862e54f2a795ffc4e541caed4d" > hash.txt hashcat -m 0 -a 0 hash.txt /usr/share/wordlists/rockyou.txt 48bb6e862e54f2a795ffc4e541caed4d:easy # cracked in milliseconds — MD5 + rockyou.txt is essentially instant # Rule-based attack — handles "Password1!" style transformations hashcat -m 0 -a 0 hash.txt rockyou.txt -r /usr/share/hashcat/rules/best64.rule # Brute force mask — 6 chars, lowercase only (?l) hashcat -m 0 -a 3 hash.txt ?l?l?l?l?l?l # bcrypt hash — notice how much slower this is (cost factor = intentional) echo '$2y$12$Dwt1BZj6pcyc3Dy1FWZ5ieeUznr71EeNkJkUlypTsgbX1H68wsRom' > bcrypt.txt hashcat -m 3200 -a 0 bcrypt.txt rockyou.txt Progress: 1024/14344391 (0.01%) Speed: 24.7 kH/s ETA: 9 mins # 24,700/sec vs 60B/sec for MD5 — same GPU, same wordlist, 2.4M× slower # Real target: crack /etc/shadow from Metasploitable using John unshadow /etc/passwd /etc/shadow > combined.txt john --wordlist=/usr/share/wordlists/rockyou.txt combined.txt msfadmin (msfadmin) # Linux /etc/shadow uses sha512crypt — john handles it automatically
INTERVIEW READY
Q: "Walk me through how you'd crack a password hash recovered from a compromised database."
A: "First I'd identify the algorithm — hash length and format narrow it down: 32 hex chars is likely MD5, 40 is SHA-1, bcrypt starts with $2y$. Then I'd choose the right Hashcat mode (-m 0 for MD5, -m 3200 for bcrypt) and start with a dictionary attack against rockyou.txt — 90% of MD5/SHA-1 hashes from real breach dumps crack on the first pass because most people's passwords are in that list, which itself came from the 2009 RockYou breach of 32 million plaintext passwords. If dictionary fails, I'd apply rule-based transformations (best64.rule handles most 'P@ssw0rd1' patterns) then move to mask attacks. Against bcrypt, all of this is viable but orders of magnitude slower — 25,000 attempts/sec vs 60 billion for MD5. That's exactly why bcrypt is the correct choice for password storage."
AI TOOLS
CrackStation / hashes.com
Free online rainbow tables. Paste an MD5 or SHA-1 hash and get the plaintext instantly if it's in the database. Great for CTF hash challenges — try this before firing up Hashcat.
ChatGPT + Hashcat
Ask "what is the Hashcat -m mode for NTLM hashes?" or "write me a Hashcat command for bcrypt with rules" — faster than reading the man page during a time-pressured CTF.
HaveIBeenPwned API
haveibeenpwned.com/API — check if a password hash appears in known breach databases using k-anonymity (sends only first 5 chars of the SHA-1). Use in your own tools to block common passwords.
hashcat.net example hashes
hashcat.net/wiki/doku.php?id=example_hashes — official reference for every hash format and its corresponding -m mode. Bookmark this — you'll use it every CTF.

You crack a sample of their hashed passwords (with authorization) to demonstrate how many would fall to a dictionary attack within minutes, then present a report showing the percentage of weak passwords and the cost of switching to bcrypt/Argon2. This is concrete, visual, undeniable evidence that lands with non-technical executives. TEACH IT BACK

Explain to a non-technical person: "A rainbow table attack is like having a pre-made answer key to a test — instead of solving every problem fresh, you just look up the answer. Salting is like giving every student a different version of the test, so one answer key can't work for all of them. The rockyou.txt wordlist that hackers use today is basically a leaked answer key from 2009 — when a company called RockYou had 32 million users' passwords stolen in plaintext. Their carelessness became every attacker's gift. bcrypt makes the test so slow to grade that even with the answer key, checking each answer takes too long to do at scale."
