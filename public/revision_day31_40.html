Malware Types: Virus, Worm, Trojan, Ransomware
NotPetya (2017)
// four core types
Virus
Attaches to a file. Needs user action (open infected .exe) to spread. Self-replicates within a host.
Worm
Self-propagates across networks without user action. NotPetya used EternalBlue to spread worm-style — no clicks needed.
Trojan
Disguised as legitimate software. Doesn't self-replicate — relies on user downloading and running it.
Ransomware
Encrypts files and demands payment. NotPetya was a wiper disguised as ransomware — irreversible destruction, not decryptable.
// real incident

NotPetya (2017): A Russian military cyberweapon disguised as ransomware. It spread via EternalBlue (worm component) across Maersk, Merck, and FedEx. Master Boot Record overwritten — $10B+ in damages. It wasn't actually ransomware: even if you paid, there was no key to decrypt. The destruction was the objective.

// interview answer
Q: What is the difference between a virus and a worm? Give a real example of each.
A virus needs user action to spread (opening a file). A worm propagates automatically over the network. ILOVEYOU (2000) was a virus — users had to open the attachment. NotPetya (2017) had worm capabilities via EternalBlue — scanned and infected adjacent systems without user interaction.

Social Engineering & Phishing
Google/Facebook Wire Fraud (2013–2015)
// human attack vectors
Phishing
Fraudulent email disguised as legitimate. Creates urgency ("your account is locked") to bypass rational thinking.
Spear Phishing
Targeted. Attacker researches the victim and crafts a personally relevant message. Far higher success rate.
Pretexting
Creating a fabricated scenario (fake support call) to extract information. No email needed.
Tailgating / Baiting
Follow someone through a secured door (tailgating) or leave infected USB drives labeled "confidential" in parking lot (baiting).
// real incident

Google/Facebook Wire Fraud (2013–2015): Lithuanian hacker Evaldas Rimasauskas sent fake phishing invoices impersonating Quanta Computer, a legitimate Asian manufacturer. Google and Facebook employees — paid millions of dollars in salaries — wired over $100M to fraudulent bank accounts. No malware, no exploit — just a convincing email and invoice.

grep -i "received-spf\|dkim\|dmarc" email.txt
Check email authentication headers. Missing SPF/DKIM = likely spoofed.

Wireless Security Basics
KRACK WPA2 Attack (2017)
// wireless encryption standards
WPA2 (802.11i)
4-way handshake for auth. CCMP/AES encryption. KRACK broke the handshake itself — protocol-level, not brute-force.
WPA3 (2018)
Simultaneous Authentication of Equals (SAE). Protects against offline dictionary attacks even if password is weak.
WPA2-Enterprise
Uses RADIUS server. Each user gets unique credentials. Still vulnerable to rogue AP attacks if users don't validate certificates.
// capturing handshakes
airodump-ng wlan0
List all nearby Wi-Fi networks — BSSID, channel, encryption, signal strength.
airodump-ng -c 6 --bssid AA:BB:CC:DD:EE:FF -w capture wlan0
Capture packets on specific channel/BSSID to file. Use aireplay-ng to force deauth and capture handshake.
aireplay-ng -0 2 -a AA:BB:CC:DD:EE:FF wlan0
Deauthentication attack — kicks client off AP, forces reconnection and re-transmission of 4-way handshake.
// real incident

KRACK (CVE-2017-13077, 2017): Vulnerable in the WPA2 protocol itself — not a specific implementation. The 4-way handshake could be manipulated to reuse nonces, allowing decryption of traffic without knowing the Wi-Fi password. Every WPA2 device was affected. The fix required OS/device firmware updates — not something users could easily do.

Cloud Security Basics: AWS/Azure Shared Model
Capital One Cloud Breach (2019)
// shared responsibility model
AWS: Security OF the cloud
AWS handles physical data centers, hardware, network infrastructure. You never touch their switches.
AWS: Security IN the cloud
YOUR responsibility. IAM policies, S3 bucket permissions, security groups, encryption at rest, OS patching on EC2.
Most breaches happen here
Capital One, Verizon, Accenture — all misconfigured the "IN the cloud" part. Overly permissive IAM + public-facing resources.
// IAM least privilege
aws iam list-attached-role-policies --role-name MyRole
List all policies attached to an IAM role — review for overly permissive actions like s3:*.
aws s3api get-bucket-acl --bucket target-bucket
Check S3 bucket ACLs. Look for URI=http://acs.amazonaws.com/groups/global/AllUsers — means public.
aws sts assume-role --role-arn arn:aws:iam::123456:role/CrossAccountRole --role-session-name test
Assume a role — see what permissions you inherit. Abuse over-privileged cross-account roles.
// real incident tie-in

The Capital One breach (covered Day 28) is the textbook cloud security failure: SSRF to metadata service → IAM credentials with S3 read access → 100M+ records. Both violations are STRICTLY "IN the cloud" — AWS infrastructure functioned exactly as designed. The customer's configuration was the vulnerability.

Pentesting Methodology: Recon → Exploit → Report
HackerOne Uber Bug Bounty (2017)
// the five phases
1. Reconnaissance
Gather intel without touching systems. WHOIS, DNS, Shodan, Google dorking, job postings. 70% of effort.
2. Scanning
Active probing — Nmap, directory brute-force, vulnerability scanners. Turn intel into a target list.
3. Exploitation
Gain access using discovered vulnerabilities. SQLi, XSS, weak credentials, known CVEs.
4. Post-Exploitation
Lateral movement, privilege escalation, persistence, data exfiltration. Pivot to next target.
5. Reporting
Document findings, severity, reproduction steps, remediation. Without this, you're not a professional pentester.
// real incident

Uber Bug Bounty 2017: A researcher followed structured recon → found an SSRF on an internal API → extracted AWS IAM keys → owned the production environment. The report won a $10k bounty. The methodology mattered more than the tool: phase-by-phase reconnaissance identified the SSRF that a broad scan would have missed.

// interview answer
Q: Walk me through your pentesting methodology from start to finish.
Five phases: Recon (passive OSINT, Shodan, DNS enumeration), Scanning (Nmap port scans, directory brute-force, vulnerability scans), Exploitation (targeted CVE hunting, credential testing), Post-Exploitation (lateral movement, privilege escalation), Reporting (severity ratings, reproduction steps, remediation). Every phase must be documented — if it's not in the report, it didn't happen.

Passive Reconnaissance & OSINT
Shodan — Exposed ICS Devices
// passive recon sources
WHOIS
Domain registration data — registrant name, email, address, phone. Often reveals real names and org structure.
DNS Enumeration
dig ANY, subdomain brute-force (subfinder), zone transfers if misconfigured. Maps the entire digital footprint.
Shodan
Search engine for internet-connected devices. Find exposed RDP, SSH, webcams, ICS/SCADA — without scanning.
Google Dorking
Advanced search operators. site:target.com filetype:pdf, intitle:"index of" /backup — finds exposed files.
// commands
whois target.com
Query domain registration — registrar, dates, nameservers, registrant contact.
dig ANY target.com
Return all DNS record types — A, AAAA, MX, NS, TXT, SOA in one shot.
subfinder -d target.com
Passive subdomain enumeration using multiple sources (Crtsh, VirusTotal, etc.)
shodan search "port:22 country:IN"
Search Shodan — find SSH servers in India. Replace filters to find any exposed service.
// real incident

Shodan ICS Discovery: Researchers used Shodan to find thousands of industrial control systems accessible from the internet — water treatment plants running HTTP interfaces without auth, power grid controllers on default credentials, building management systems with public dashboards. These weren't discovered by scanning each target — Shodan had already indexed them. Passive recon is often more revealing than active probing.

Active Recon — Nmap Basics
Mirai Botnet Nmap-Style Scanning (2016)
// scan types
TCP SYN (-sS)
Half-open scan. Sends SYN, reads SYN-ACK (open) or RST (closed). Never completes handshake. Stealthy — not logged by some apps.
TCP Connect (-sT)
Completes full 3-way handshake. More reliable, less stealthy — every connection is logged by the target.
UDP Scan (-sU)
Slow — UDP doesn't respond with port closed. Timeout-based detection. Essential for DNS (53), SNMP (161).
// port states
Open
Application is listening. SYN-ACK received. The target surface.
Closed
Port accessible but no app listening. RST received. Not exploitable but host is reachable.
Filtered
Firewall blocks probe. No response or ICMP unreachable. May need stealth timing to bypass.
// commands
nmap -sS -p 22,80,443,3389 target.com
Stealth SYN scan on common ports. Fast and effective for initial assessment.
nmap -sT -p- target.com
Full TCP connect scan on all 65535 ports. Slow but comprehensive — every open port found.
nmap -sU -p 53,161 target.com
UDP scan — checks DNS (53) and SNMP (161). Often forgotten, often exposed.
// real incident tie-in

Mirai 2016: The Mirai botnet scanned the entire IPv4 internet on port 23 (Telnet) using simple SYN probes — exactly what nmap -sS does. When it found a telnet-enabled IoT device with default credentials, it logged in and installed DDoS malware. No advanced exploit — just a SYN scan and admin:admin. Nmap-style scanning at scale.

Nmap Advanced — Scripts & Evasion
NSE EternalBlue Detection (2017)
// NSE script categories
vuln
Check for specific CVEs. nmap --script vuln — scans every known vuln script against open ports.
exploit
Run actual exploits against service. Dangerous — only use with explicit permission.
auth
Test authentication — default credentials, anonymous FTP, LDAP null binds. Finds weak auth.
discovery
Enumerate services — SMB shares, NFS exports, database schemas. Post-exploration mapping.
// evasion techniques
Decoy Scan (-D)
Spoof random source IPs alongside your real one. Target sees 20 scans from 20 IPs — harder to block.
Fragment Scan (-f)
Split TCP header across multiple packets. Some simple firewalls can't reassemble and pass the fragments.
Timing (-T0 to -T5)
T0 = paranoid (5 min between probes). T5 = insane (fast). Lower timing evades rate-based IDS.
// commands
nmap --script vuln -sV target.com
Run all vulnerability detection NSE scripts with version detection. One command vulnerability assessment.
nmap --script smb-vuln-ms17-010 -p 445 target.com
Check if target is vulnerable to EternalBlue — the WannaCry/NotPetya entry point.
nmap -sS -D RND:10 -f -T2 target.com
Stealth scan: 10 decoy IPs + fragment packets + slow timing. Evades most default IDS rules.
// real incident

NSE EternalBlue Detection: After WannaCry exploded in May 2017, defenders worldwide used nmap --script smb-vuln-ms17-010 to rapidly scan their networks. What would have taken days of manual checking became a single Nmap command. This script turned Nmap from a port scanner into a vulnerability assessment tool — the same scan Mirai used to find targets could also find vulnerabilities.

Vulnerability Scanning with Nessus/OpenVAS
Nessus in Target Breach Forensics (2013)
// automated vs manual
Automated (OpenVAS)
Scans 50k+ CVEs. Fast, broad coverage. High false positives. Misses business logic flaws.
Manual (Nmap + Burp)
Targeted, creative, finds logic flaws and chained attacks. Slow but thorough.
Best Practice
Run automated scanner first for baseline → manually verify every finding → then hunt for what scanners miss.
// openvas basics
gvm-cli --gmp-username admin --gmp-password pass socket --socketpath /var/run/gvmd.sock --xml "<create_target><name>Scan_Target</name><hosts>10.0.0.1</hosts></create_target>"
Create a scan target in OpenVAS (Greenbone) via GMP API.
openvas-nvt-sync
Update all Network Vulnerability Tests — get latest CVE coverage before scanning.
gvm-cli --gmp-username admin --gmp-password pass socket --socketpath /var/run/gvmd.sock --xml "<start_task><task_id>TASK-ID</task_id></start_task>"
Start a scan task. Results appear in Greenbone Security Assistant web UI.
// real incident

Nessus in Target Forensics: After the Target breach (2013), forensic investigators deployed Nessus to scan Target's POS environment. They found 50+ unpatched vulnerabilities including the HVAC vendor's systems that were the initial entry point. An automated scan before the breach would have caught all of them. The scans existed — the scanning policy didn't.

// Quick Self-Quiz — Can You Answer These Without Looking?

Cover the answers. Say them out loud. If you can't answer one fluently, revisit that day before continuing.

Q01
What made NotPetya different from typical ransomware? Why couldn't victims get their data back even if they paid?
NotPetya was a wiper disguised as ransomware. The MBR was overwritten and files were irreversibly encrypted — no decryption key existed. Payment was pointless because recovery was never intended.
Q02
How did a single phishing email convince Google and Facebook to wire over $100M to a Lithuanian hacker?
The emails impersonated Quanta Computer, a real supplier. The invoices looked legitimate. Employees in finance approved payments totaling $100M+ without verifying the bank account change via a separate channel — the classic BEC (Business Email Compromise) technique.
Q03
What is the KRACK attack and why was it especially dangerous compared to other Wi-Fi vulnerabilities?
KRACK broke the WPA2 protocol itself — not a specific implementation. It manipulated the 4-way handshake to reuse nonces, allowing traffic decryption without knowing the password. Every WPA2 device was affected, and the fix required OS/firmware updates.
Q04
In the AWS shared responsibility model, who was at fault for the Capital One breach — AWS or Capital One?
Capital One. AWS provided the tools correctly (metadata service, IAM, S3). Capital One misconfigured their WAF (allowing SSRF) and IAM role (over-privileged read access to S3). Security IN the cloud is always the customer's responsibility.
Q05
What are the five phases of a professional penetration test? Which phase takes 70% of the effort?
Reconnaissance (70% of effort), Scanning, Exploitation, Post-Exploitation, Reporting. Most beginners rush to exploitation — professionals spend the most time on recon because that's where the target reveals its weaknesses.
Q06
How does Shodan find exposed industrial control systems without actively scanning the internet?
Shodan continuously crawls the internet and indexes banners from every service it connects to. ICS devices like water treatment controllers often run HTTP interfaces without authentication — Shodan finds and indexes them during routine internet-wide probing, making them searchable by anyone.
Q07
What is the difference between a TCP SYN scan (-sS) and a TCP Connect scan (-sT)? When would you use each?
SYN scan is half-open (sends SYN, reads SYN-ACK, sends RST) — never completes handshake, stealthier, faster. Connect scan completes full TCP handshake — logged by target, more reliable. Use SYN for speed/stealth, Connect when you need accuracy through firewalls.
Q08
What Nmap command would you run to check if a target is vulnerable to EternalBlue? What's the script category?
nmap --script smb-vuln-ms17-010 -p 445 target.com. The category is "vuln" — includes all detected CVE checks. This is the script that defenders ran worldwide after WannaCry.
Q09
How did the Mirai botnet use Nmap-style scanning, and what were they looking for?
Mirai scanned the entire IPv4 internet on port 23 (Telnet) using TCP SYN probes. When it found a Telnet-enabled IoT device with default credentials (admin:admin), it logged in and installed DDoS malware. Simple scan + default creds = millions of bots.
Q10
What was the key lesson from the Target breach forensic scan with Nessus?
Post-breach Nessus scans found 50+ unpatched vulnerabilities on Target's systems — including the HVAC vendor entry vector. The lesson: vulnerability scanning tools like Nessus don't work unless you actually run them regularly with a policy that requires remediation before attackers exploit what scanners find.
