Privilege Escalation Linux

Getting a low-privilege shell is step one. Becoming root is step two. Today you master Linux privesc techniques: SUID, cron jobs, kernel exploits, and misconfigurations.

THEORY

Privilege escalation defined: Once you've gained initial access to a Linux system (via a web shell, reverse shell, or SSH), you're typically running as a low-privilege user like www-data or nobody. Privilege escalation is the art of becoming root — or another user with higher privileges. The core principle: every Linux system has misconfigurations that grant unintended access. Your job is to find them faster than the defender patched them.

Five most common Linux privesc vectors:

# 1. SUID binaries — find all files with the SUID bit set find / -perm -4000 -type f 2>/dev/null # 2. Cron jobs — look for writable scripts in cron directories ls -la /etc/cron* /etc/periodic/* cat /etc/crontab # 3. Sudo misconfiguration — list commands your user can run as root sudo -l # 4. Kernel exploits — identify kernel version and match to public exploits uname -a # 5. World-writable directories & files — find what you can modify find / -writable -type f 2>/dev/null | grep -v proc

The SUID exploitation pattern: When a binary has the SUID bit set (permission rwsr-xr-x), it runs with the file owner's privileges — often root. If that binary executes user-controlled input (like nmap --interactive, find -exec, or vim), you can use it to spawn a root shell. The classic example is nmap with SUID: running nmap --interactive then !sh drops you into a root shell because nmap executes as root.

Key insight: Automated enumeration tools like linpeas.sh and linux-exploit-suggester.sh are force multipliers. Run them within seconds of gaining access. Most CTF boxes and real servers have at least one privesc path that a scanner finds immediately.
// attacker vs defender mindset
⚔ Attacker
  • Land and immediately run linpeas.sh — upload it to the target via Python HTTP server + wget/curl, execute, and analyze the color-coded output. Red/Yellow lines are your priority targets.
  • Check sudo -l first — if you can run ANY binary as sudo without a password, GTFO Bins (gtfobins.github.io) tells you the escape sequence for every known binary.
  • Look for writable scripts in cron directories — if /etc/cron.hourly/ contains a world-writable script, write a reverse shell payload and wait for execution.
  • Always check /etc/passwd writability — if world-writable, you can create a new root user with openssl passwd -1 and append to the file.
🛡 Defender
  • Remove SUID bits from all binaries that don't absolutely need them — chmod -s /usr/bin/nmap /usr/bin/vim /usr/bin/find. Only SUID what must be SUID.
  • Restrict sudo access with explicit commands and require password — username ALL=(root) /usr/bin/systemctl restart nginx — no wildcards, no NOPASSWD.
  • Make cron script directories non-writable by non-root — chmod 755 /etc/cron.d/ and ensure all cron scripts are owned by root with no group/other write.
  • Use apparmor/selinux to confine vulnerable services — even if www-data is compromised, the damage is contained to the service boundaries.
// real named incident
2022 — DIRTY PIPE (CVE-2022-0847) — LINUX KERNEL
Vulnerability in Linux's pipe mechanism allowed unprivileged users to overwrite read-only files including SUID binaries
Discovered by researcher Max Kellermann, CVE-2022-0847 (nicknamed "Dirty Pipe") affected Linux kernel versions 5.8 through 5.16.11. The flaw existed in the pipe system call — a buffer initialization bug meant that uninitialized memory could be written into a pipe buffer, then spliced into any file the user could read, including SUID binaries and system files. An unprivileged user could overwrite /usr/bin/passwd or /etc/passwd with arbitrary content, gaining root access. A proof-of-concept was published that achieved root in under 5 seconds. Over 1.5 billion Android devices shipped with vulnerable kernels. The fix was a single line change in the kernel source.
HANDS-ON LAB

Goal: gain initial low-priv access to a target Linux machine then escalate to root using at least three different techniques.

  1. Deploy a vulnerable Linux VM: docker run -d vulhub/linux-privesc or use a TryHackMe room like "Linux PrivEsc Arena" (free). SSH in with provided credentials.
  2. Run sudo -l. If you find a binary like (root) /usr/bin/vim, run sudo vim -c ':!/bin/bash' — this drops a root shell using Vim's escape feature.
  3. Run find / -perm -4000 -type f 2>/dev/null to find SUID binaries. Check each one on GTFOBins. For /usr/bin/base64, read /etc/shadow: base64 /etc/shadow | base64 -d.
  4. Check writable cron scripts: ls -la /etc/cron.d/ /etc/cron.hourly/. If a script is world-writable, append echo "root:newhash:0:0:root:/root:/bin/bash" >> /etc/passwd after generating a hash with openssl passwd -1.
  5. Run uname -a and feed the kernel version into linux-exploit-suggester.sh. If a kernel exploit exists (like CVE-2021-3493 for overlayfs), compile and run it.
  6. For each successful escalation, document the technique, the command used, and what configuration error allowed it. This is your personal privesc reference guide.
Deliverable: A documented privesc chain with at least three unique routes from low-priv to root. For each route: the enumeration command that found it, the exploitation command, and the defense that would have prevented it.

INTERVIEW READY
Q: "Walk me through your Linux privilege escalation methodology from the moment you get a low-priv shell."
A: "First, I stabilize the shell (python -c 'import pty;pty.spawn("/bin/bash")'). Then I run automated enumeration — I transfer linpeas.sh or run linux-exploit-suggester.sh. In parallel, I manually check: sudo -l, find SUID binaries, /etc/crontab and cron directories, writable files, kernel version, and /etc/passwd permissions. Each finding gets cross-referenced with GTFOBins. I prioritize sudo misconfigurations and SUID binaries because they're the fastest exploits. If nothing obvious shows, I analyze the cron jobs and look for any script I can modify."
Q: "How would you exploit a SUID binary to escalate privileges?"
A: "First find all SUID binaries with 'find / -perm -4000 -type f 2>/dev/null'. For each one, check on GTFOBins if there's an escape technique. For example, if /usr/bin/find has SUID, running 'find . -exec /bin/bash -p \;' executes bash with the file's permissions (root). Similarly, 'nmap --interactive' followed by '!sh' works for nmap, and 'vim -c ':!/bin/bash'' works for vim. The key is that the binary runs as the file owner (root), so any command execution within the binary inherits root privileges."
Q: "What's the difference between horizontal and vertical privilege escalation?"
A: "Horizontal escalation is accessing another user's resources at the same privilege level — for example, from the www-data user to the john user. Vertical escalation (privilege escalation) is moving from a lower privilege to a higher one — like from www-data to root. Both are important, but vertical escalation is typically the end goal. In practice, you might do horizontal first to access credentials or keys that enable vertical escalation."
