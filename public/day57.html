Log Analysis Fundamentals

Logs are the ground truth of every security incident. Attackers can clear browser history, delete files, and even wipe databases — but logs on a properly configured system tell the story that can't be erased. Today you parse syslog, Apache access/error logs, auth.log, and Windows Event Logs to identify Indicators of Compromise (IOCs) and reconstruct attacker movements.

THEORY

What is Log Analysis? Log analysis is the systematic examination of system, application, and security logs to detect anomalous activity, reconstruct security incidents, and identify indicators of compromise (IOCs). Every action on a system — a login attempt, a web request, a process start — is recorded somewhere. The discipline of log analysis is knowing where to look and what to look for.

The five foundational log sources in security:

Syslog / rsyslog
Linux's universal logging daemon. Collects kernel messages, system events, cron jobs, and application logs via /var/log/syslog (Debian) or /var/log/messages (RHEL). The starting point for any Linux incident response.
Apache access / error logs
/var/log/apache2/access.log records every HTTP request — IP, method, URI, status, user-agent. error.log captures server-side failures. Together they reveal scanning, brute-force attempts, and exploited endpoints.
auth.log
/var/log/auth.log captures all authentication-related events: SSH logins (success/failure), sudo usage, PAM errors, and useradd/del operations. The single most important file during a Linux intrusion investigation.
Windows Event Logs
Security.evtx (logins, privilege use, policy changes), System.evtx (service starts, driver loads), Application.evtx (app crashes, errors). Event ID 4625 = failed login, 4624 = success, 4688 = process creation.
Firewall & proxy logs
Network-level logs from iptables, pf, or enterprise firewalls. Outbound connections to known-bad IPs, port scan patterns, and DNS queries to algorithmically generated domains (DGA) are definitive indicators of compromise.

A real auth.log — SSH brute-force in progress:

Mar 15 02:14:23 web-srv-01 sshd[12345]: Failed password for root from 103.235.46.12 port 52134 ssh2
Mar 15 02:14:25 web-srv-01 sshd[12347]: Failed password for admin from 103.235.46.12 port 52136 ssh2
Mar 15 02:14:27 web-srv-01 sshd[12349]: Failed password for oracle from 103.235.46.12 port 52138 ssh2
Mar 15 02:14:29 web-srv-01 sshd[12351]: Accepted password for deploy from 103.235.46.12 port 52140 ssh2

Pattern: Three failed passwords for nonexistent users, then a successful login for "deploy" — a user that does exist. The attacker used a dictionary of common usernames and hit a valid account. The time between attempts: 2 seconds. No human types that fast. This is automated.

Key insight: Never read logs line by line like a book. Your adversary generates millions of lines. Use grep, awk, cut, and timestamp aggregation to identify statistical outliers. A spike from 100 auth failures/day to 10,000 in an hour is not a user with a bad password — it's an active attack.
// analyst vs adversary mindset
⚔ Attacker
  • Modify or delete logs after gaining root — sed -i '/103.235.46.12/d' /var/log/auth.log removes evidence of your IP.
  • Use SSH to a compromised jump box as a pivot — the auth.log shows the jump box, not your real IP. Defenders find the jump box, not you.
  • Live-off-the-land: abuse PowerShell, WMI, and SSH keys installed by the victim instead of dropping custom binaries that generate log entries.
  • Time-stomping: use touch -t to backdate file timestamps so timeline analysis places your actions before the initial compromise.
🛡 Defender
  • Ship logs off-box in real-time (rsyslog remote, syslog-ng, or a SIEM) — an attacker who compromises the box cannot delete what is already in your centralized log store.
  • Alert on log deletion: any process truncating or deleting /var/log/auth.log is a high-priority event. Write a file integrity monitor rule for it.
  • Correlate across log sources: an Apache 403 followed 30 seconds later by a successful auth.log entry from the same IP is a credential-stuffing pattern, not coincidental.
  • Use timestamp normalization — convert all logs to UTC so that an attacker working at 03:00 UTC from a country in daylight time doesn't confuse your timeline.
// real named incident
2022 — UBER BREACH — TEENAGER USES SOCIAL ENGINEERING + COMPROMISED VENDOR
An 18-year-old attacker compromised Uber's internal systems via a contractor's compromised HackerOne account — detected in VPN authentication logs
The attacker first obtained an employee's corporate password from a previous data leak or malware on the employee's personal device. When the employee denied the MFA push notification, the attacker contacted the employee on WhatsApp, impersonated Uber IT, and convinced them to approve the MFA notification. Once inside the VPN, the attacker scanned Uber's internal network and discovered a PowerShell script containing hardcoded credentials for Uber's HackerOne admin account — a privilege escalation service account named H1-admin. The attacker used those credentials to access Uber's bug bounty dashboard, read submitted vulnerability reports, and triage them before Uber could. The breach was discovered when Uber's security team analyzed the VPN authentication logs — specifically, a login from an IP address that did not match the employee's known VPN session profile. The attacker's identity was eventually traced through a Telegram channel where they bragged about the breach. Uber paid the attacker as a security researcher through HackerOne's bug bounty program. The root cause was not technical — it was a failure in MFA fatigue awareness and credential separation between corporate and HackerOne access.
HANDS-ON LAB

Goal: analyze a set of synthetic logs (syslog, Apache access/error, auth.log, and Windows Event Log exports) to identify a simulated breach — trace the attacker from initial probe to data exfiltration.

  1. Download the sample log bundle from the resources section below. Extract the archive to ~/logs/lab1/.
  2. Open auth.log — count failed SSH attempts by IP with: grep "Failed password" auth.log | awk '{print $11}' | sort | uniq -c | sort -nr. Identify the top attacker IP.
  3. Open apache_access.log — find SQLi attempts: grep -iE "union|select|drop|--|'|\"|/etc/passwd" access.log. Cross-reference IPs with auth.log.
  4. Open syslog — look for cron jobs that should not exist: grep "CRON" syslog | grep -v "root". Was a reverse shell scheduled?
  5. Parse the Windows Security.evtx export (XML format): grep "EventID>4624" Security.xml | grep -v "NT AUTHORITY" | wc -l — count non-system logins.
  6. Build a timeline: order every event (auth success, web exploit, cron install, data access) by timestamp. Write a 3-paragraph incident report.
Deliverable: A timeline of the attack with the attacker's IP, the initial compromise vector (brute-force or SQLi), the privilege escalation method, and the exfiltration mechanism. Attach all grep/awk commands used to find each piece of evidence.

INTERVIEW READY
Q: "What is the first thing you look for when investigating a Linux server compromise?"
A: "I start with /var/log/auth.log — specifically, Accepted password entries clustered around unusual times, coming from IPs outside the organization's known ranges. A single successful login from an unrecognized IP at 3 AM is the highest-signal IOC. Then I cross-reference that IP with Apache access logs and last output to build a timeline of the attacker's actions on the box."
Q: "How do you distinguish a brute-force attack from a credential-stuffing attack in logs?"
A: "Brute-force: few usernames, thousands of passwords, quick sequential attempts from one IP. You'll see Failed password for root repeated 500 times with different passwords. Credential stuffing: many usernames, one or two passwords each, often via HTTP POST to a login endpoint. In the Apache access log you see 1,000 different username=xyz&password=Spring2024! requests from the same IP in under a minute. The distinction matters because credential stuffing implies the attacker already has a leaked credential list — you need to force password resets for all users, not just block the IP."
Q: "An attacker deleted the auth.log. How do you proceed?"
A: "If logs are shipped off-box in real-time, I have a central copy. If not, I check journalctl — systemd journal may still have the data. I also check /var/log/wtmp and /var/log/btmp which record login/logout events in binary and cannot be easily deleted. I examine the bash history for all users — ~/.bash_history and /root/.bash_history. And I look at file timestamps with stat and debugfs — the deletion of auth.log itself is an artifact that tells me when the attacker escalated privileges."
AI TOOLS
Log summarizer
Feed 10,000 lines of auth.log into an LLM with the prompt: "Summarize all unique IPs, their attempt counts, and whether any succeeded." The LLM extracts the attacker narrative in seconds.
Pattern suggestion engine
Ask: "Here is my auth.log grep output — what patterns suggest a directed attack vs random scanning?" The AI distinguishes spray attacks from targeted intrusion based on timing and username selection.
Splunk SPL translator
Describe what you want in plain English: "Show me all failed logins in the last 24 hours grouped by source IP" — the AI generates the exact SPL query for Splunk, QRadar, or ELK.
Log anomaly explainer
Copy a suspicious log line and ask: "What does this mean in the context of a web shell attack?" The AI explains the process, the user context, and what the parent process tells you about the attacker's entry point.

ENTREPRENEUR ANGLE

Most small and medium businesses have no log retention or analysis capability. They keep auth.log until disk fills up, then rotate it away forever. The product: "Log Audit & Retention Audit" — you spend a day setting up rsyslog remote forwarding, install Logwatch or Wazuh, and write a 10-page report on every IOC found in their existing logs.

Price point: $1,500-3,000 per engagement, recurring quarterly. You sell the compliance angle: PCI DSS Requirement 10 mandates logging and log review. Most SMBs fail this on their first PCI assessment — you help them pass. The Wazuh SIEM community edition is free and covers file integrity monitoring, log collection, and alerting. Your value is configuration and interpretation, not software licensing.

TEACH IT BACK

Teach someone who has never analyzed logs: "There are five main types of logs on a Linux system. I'll show you the two most important — auth.log tells you who logged in, and Apache access.log tells you what they requested. If I find the same IP in both files, I know the attacker used the web app as their entry point and SSH for lateral movement." Draw a timeline on paper showing the attack chain: probe → SQLi → shell → cron persistence → data exfil.
