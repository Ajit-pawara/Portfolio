Network Traffic Analysis — Wireshark

Network traffic analysis is the art of reading packets like a surgeon reads X-rays. Wireshark is the industry-standard protocol analyzer — used by SOC analysts, incident responders, and threat hunters to reconstruct attacks, detect C2 beaconing, and extract artifacts from raw packet captures.

THEORY

What is Network Traffic Analysis? Network traffic analysis captures and inspects packets traversing a network to identify malicious activity, policy violations, and performance issues. Every request, response, DNS lookup, and TLS handshake leaves a forensic record in the packet stream.

Wireshark's core capabilities:

Packet Capture (Live)
Capture traffic in real-time from any network interface. Filter during capture with BPF syntax: host 10.0.0.1 and port 80.
Display Filters
Post-capture filtering without modifying the capture file. Expression-based: http.request.method == "POST", tcp.port eq 443.
Protocol Hierarchy
Statistics → Protocol Hierarchy shows the breakdown of every protocol in the capture. A single DNS-over-HTTPS outlier can reveal C2 traffic.
Follow Streams
Reconstruct TCP, UDP, TLS, or HTTP streams into a readable conversation. Right-click → Follow → TCP Stream shows the full client-server dialogue.

BPF (Berkeley Packet Filter) syntax for live capture:

# Capture only HTTP traffic from a specific host tcp port 80 and host 192.168.1.100 # Exclude broadcast and multicast not broadcast and not multicast # Capture DNS queries only udp port 53

Display filter basics: Wireshark display filters use a field operator value syntax. Examples: ip.src == 10.0.0.5, tcp.analysis.flags (show all TCP anomalies), http.response.code >= 400 (all server errors).

Key distinction: BPF filters are applied during capture and are irreversible. Display filters are applied after capture and are non-destructive. Always capture full packets (no slicing) when doing security analysis — you can't reconstruct what you didn't capture.
// analyst vs adversary mindset
⚔ Adversary
  • Encrypt C2 traffic with HTTPS or custom protocols — Wireshark sees the TLS handshake but the payload is opaque without the key.
  • Use DNS tunneling to exfiltrate data: encode stolen files as subdomain queries (base64.stolen-data.evil.com).
  • Beacon on irregular intervals with jitter (e.g., 47-53 seconds) to evade frequency-based detection rules.
  • Fragment packets or use non-standard TCP window sizes to evade signature-based NIDS rules that expect canonical packet structure.
🛡 Analyst
  • Start with http.request or tls.handshake.type == 1 — identify every HTTP request and TLS Client Hello to map all external communications.
  • Use tcp.analysis.retransmission and tcp.analysis.ack_lost_segment to find hosts experiencing packet loss — often indicates scanning or DoS.
  • Enable TLS key logging (SSLKEYLOGFILE) for decryption in controlled lab environments: set the environment variable before launching the app.
  • Build a baseline of normal traffic patterns first — anomalies only stand out against a known good baseline.
// real named incident
2016 — BANGLADESH BANK HEIST — $81 MILLION (DPRK LAZARUS GROUP)
Wireshark analysis of SWIFT traffic inside Bangladesh Bank reconstructed the Lazarus Group's fraudulent transfer requests
The Lazarus Group (DPRK) compromised Bangladesh Bank's SWIFT network and issued fraudulent transfer requests totaling $951 million. Wireshark was used to capture and reconstruct the SWIFT MT103 message traffic between Bangladesh Bank and the Federal Reserve Bank of New York. Analysts followed the TCP streams to see the exact SWIFT messages, identified the attacker's IPs, and traced the flow of funds through accounts in Sri Lanka and the Philippines. The attackers had installed a custom malware that monitored the bank's SWIFT printer and deleted the printed confirmation records — but the Wireshark pcap files preserved the full conversation. Five of the 35 fraudulent transfers succeeded ($101M), of which $81M was never recovered. This remains the largest cyber-heist in history.
HANDS-ON LAB

Goal: capture, analyze, and reconstruct network traffic using Wireshark. Identify HTTP credentials, extract transferred files, and detect a simulated C2 beacon.

  1. Launch Wireshark and capture on the active interface. Apply BPF tcp port 80 or tcp port 443 to reduce noise. Save the capture as lab61.pcapng.
  2. Visit http://testphp.vulnweb.com/login.php (public Acunetix test site) and enter fake creds. Stop capture. Use display filter http.request.method == "POST" to find the login request.
  3. Right-click the POST packet → Follow → HTTP Stream. The submitted username and password are visible in cleartext in the stream body.
  4. Use Statistics → Protocol Hierarchy to see the protocol breakdown. Note the percentage of TCP, HTTP, DNS, and TLS traffic.
  5. Export an HTTP object: File → Export Objects → HTTP → select a file and save it. This extracts files transferred over HTTP from the pcap.
  6. Set up a simple reverse shell C2 simulation: run nc -lvnp 4444 in a terminal, connect from another terminal, and capture the traffic. Use display filter tcp.port == 4444 and follow the TCP stream.
Deliverable: A Wireshark pcap file with at least one HTTP stream showing cleartext credentials, a protocol hierarchy screenshot, and a TCP stream reconstruction of a netcat conversation. You can now read network traffic like a SOC analyst.
# Launch Wireshark from CLI (or open GUI) wireshark # Capture with tshark (CLI equivalent of Wireshark) tshark -i eth0 -w lab61.pcapng -f "tcp port 80 or tcp port 443" > Press Ctrl+C to stop capture # Read capture and filter for HTTP POST requests tshark -r lab61.pcapng -Y "http.request.method == POST" -T fields \ -e http.host -e http.request.uri -e http.file_data # Follow TCP stream (stream index 0 shown below) tshark -r lab61.pcapng -z follow,tcp,ascii,0 # Reconstructs the full TCP conversation as ASCII # Protocol hierarchy stats tshark -r lab61.pcapng -z io,phs # Extract all HTTP objects tshark -r lab61.pcapng --export-objects "http,/tmp/extracted/" > Files (images, scripts, documents) saved to /tmp/extracted/
INTERVIEW READY
Q: "How would you use Wireshark to investigate a potential data exfiltration?"
A: "I'd start by examining the Protocol Hierarchy Statistics to identify unusual protocols — DNS with high volume, ICMP with large payloads, or non-standard ports. Then I'd filter for suspicious outbound connections: ip.dst != 10.0.0.0/8 and ip.dst != 192.168.0.0/16 and tcp.dstport not in {80 443 53}. I'd look for DNS TXT queries with base64-encoded subdomains, follow TCP streams to see if data is being sent to unknown hosts, and check TLS SNI fields for domains that don't match the organization's normal traffic patterns."
Q: "What's the difference between a BPF capture filter and a Wireshark display filter?"
A: "BPF filters are applied during capture at the kernel level — they drop non-matching packets before they reach Wireshark. This is useful for reducing capture file size on busy networks but means you permanently lose non-matching traffic. Display filters are applied after capture on the full pcap file — they're non-destructive and can be changed anytime. For security analysis, I always capture everything and filter during analysis, unless disk space or performance forces BPF pre-filtering."
Q: "How do you analyze encrypted TLS traffic in Wireshark?"
A: "Without the private key or session keys, Wireshark can only show TLS handshake metadata — cipher suites, certificate details, SNI — but not the encrypted payload. In controlled environments, I set the SSLKEYLOGFILE environment variable before launching the browser or application. Wireshark decrypts the traffic when I point it to that log file via Edit → Preferences → TLS → (Pre)-Master-Secret log filename. In incident response, I'd look at TLS handshake metadata: unusual cipher suites, self-signed certs, or SNI names that don't match the IP's expected domain."
AI TOOLS
Pcap summarizer
Upload or describe a pcap file to an LLM and ask it to summarize the conversation patterns — endpoints, protocols, suspicious anomalies — based on your tshark field output.
Display filter generator
Describe the traffic you need to find in plain English: "show me all DNS queries that are not going to our internal resolvers" — get a ready-to-paste display filter expression.
Malicious traffic profile
Paste extracted TLS handshake or HTTP header data and ask the LLM to classify the traffic as benign, C2 beaconing, or data exfiltration based on known IoC patterns.
Log to pcap converter
Ask an LLM to write a Python script that converts a proxy log or firewall log into a pcap file using scapy — bridges the gap between log analysis and Wireshark visualization.
