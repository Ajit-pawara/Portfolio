Firewalls, IDS, IPS Explained

Firewalls decide who gets in. IDS watches what happens once they're in. IPS stops it automatically. Today you configure all three.

THEORY

The three layers of network defense: A firewall is a gatekeeper — it permits or denies traffic based on rules (source IP, destination port, protocol). An IDS (Intrusion Detection System) is a watchtower — it monitors traffic, compares it against known attack signatures or behavioral baselines, and raises alerts. An IPS (Intrusion Prevention System) is a guard — it sits inline with the traffic and can block malicious packets in real-time based on the same detection mechanisms.

Firewall types:

# Packet Filter — stateless: checks headers only (IP, port, protocol) # Stateful — tracks connection state (NEW, ESTABLISHED, RELATED) # Application-layer (WAF) — understands HTTP, SQL, XSS patterns # Next-Gen (NGFW) — combines stateful firewall + IPS + app awareness # iptables stateful rule example: iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT iptables -A INPUT -p tcp --dport 22 -m state --state NEW -j ACCEPT iptables -A INPUT -j DROP # default deny — block everything else

Snort IDS/IPS rules:

# A Snort rule detects malicious traffic by pattern matching: alert tcp $EXTERNAL_NET any -> $HTTP_SERVERS 80 (msg:"SQL Injection — UNION SELECT detected"; content:"UNION"; nocase; http_uri; content:"SELECT"; nocase; http_uri; sid:1000001; rev:1;) # IPS mode (inline) uses 'drop' instead of 'alert': drop tcp $EXTERNAL_NET any -> $HTTP_SERVERS 80 (msg:"Blocked SQLi attempt"; content:"' OR '1'='1"; sid:1000002;)
The detection gap: Signature-based IDS (Snort, Suricata) can only detect what they have rules for. Zero-day exploits, encrypted traffic (TLS), and heavily obfuscated payloads bypass signature detection. This is why modern SOCs use behavioral analysis, sandboxing, and EDR alongside traditional IDS/IPS.
// attacker vs defender mindset
⚔ Attacker
  • Scan for open ports with nmap — nmap -sS -p- target.com. The firewall blocks some ports; the ones that respond are potential entry points.
  • Use fragmented packets, slow scans, or decoy addresses to evade IDS — nmap -f -D RND:10 target.com fragments IP packets and spoofs decoy sources.
  • Encrypt payloads in TLS to bypass signature-based inspection — if the firewall can't decrypt the traffic, it can't match rules against the content.
  • Use OPSEC-safe techniques: DNS over HTTPS, C2 over legitimate services (M365, Google Drive), and sleep timers to avoid behavioral detection.
🛡 Defender
  • Implement default-deny firewall policy — only allow specific, documented services. Everything else is logged and dropped.
  • Deploy IPS in inline mode for known attack patterns — SQLi, XSS, shellcode, directory traversal — but ensure traffic decryption (SSL inspection) is in place.
  • Use a tiered defense: perimeter firewall → DMZ → internal firewall → host-based firewall. Breaching one layer doesn't expose everything.
  • Monitor firewall and IDS logs centrally (SIEM). A lone alert is noise; correlated alerts across multiple sensors is a signal.
// real named incident
2021 — PALO ALTO NETWORKS PAN-OS ZERO-DAY — CVE-2021-3064
Critical firewall bypass vulnerability in PAN-OS management interface
Security researchers discovered a buffer overflow vulnerability (CVE-2021-3064) in the PAN-OS management interface that allowed unauthenticated remote attackers to execute arbitrary code with root privileges. The vulnerability existed in the GlobalProtect portal and gateway components — ironically, a security product's own exposed surface was the entry point. PAN-OS firewalls with management interfaces exposed to the internet (violating best practice) were vulnerable. Palo Alto Networks released an emergency patch, and CISA issued an emergency directive for federal agencies to patch or disconnect affected devices. The lesson: even firewalls need firewalls — management interfaces should never be internet-facing.
HANDS-ON LAB

Goal: configure iptables firewall rules, write Snort detection rules, and test IPS blocking.

  1. Check current iptables rules: sudo iptables -L -n -v. Note the default policy (ACCEPT or DROP).
  2. Set a default DROP policy on INPUT: sudo iptables -P INPUT DROP. Then explicitly allow SSH and HTTP.
  3. Install Snort: sudo apt install snort -y. Configure it for your network interface.
  4. Write a custom Snort rule that detects and alerts on ICMP ping (echo request). Save to /etc/snort/rules/local.rules.
  5. Test the rule by pinging the Snort host — ping -c 1 snort-host-ip. Check the Snort alert log at /var/log/snort/alert.
  6. Convert the rule to IPS mode (drop instead of alert). Test that ICMP is blocked after the rule change.
Deliverable: A screenshot of iptables rules showing custom policy, plus the Snort alert log entry showing a detected ping, and the ICMP drop rule.
# Step 1 — Examine current iptables rules sudo iptables -L -n -v # Step 2 — Set default DROP policy and allow essentials sudo iptables -P INPUT DROP sudo iptables -A INPUT -i lo -j ACCEPT sudo iptables -A INPUT -m state --state ESTABLISHED,RELATED -j ACCEPT sudo iptables -A INPUT -p tcp --dport 22 -j ACCEPT sudo iptables -A INPUT -p tcp --dport 80 -j ACCEPT # Step 3 — Install Snort IDS sudo apt install snort -y # During install, specify your network interface and subnet # Step 4 — Write a custom Snort rule (IDS mode) echo 'alert icmp any any -> $HOME_NET any \ (msg:"ICMP ping detected"; sid:1000001; rev:1;)' \ | sudo tee /etc/snort/rules/local.rules sudo snort -A console -q -c /etc/snort/snort.conf -i eth0 & # Step 5 — Test from another host # From another machine: ping -c 1 YOUR_SNORT_HOST_IP # On the Snort host, you'll see the alert in the console. # Step 6 — IPS mode (drop instead of alert) # For inline IPS, run: snort -Q --daq afpacket -c snort.conf -i eth0 # The rule changes from "alert" to "drop": # drop icmp any any -> $HOME_NET any (msg:"ICMP blocked"; sid:1000002;)
INTERVIEW READY
Q: "What's the difference between a stateless and stateful firewall?"
A: "A stateless firewall examines each packet in isolation — it only checks header fields (source IP, dest IP, port, protocol). A stateful firewall tracks the state of connections — it knows whether a packet is part of a new connection, an existing established connection, or related to one (like FTP data channels). Stateful firewalls can automatically allow return traffic for outbound connections, which makes rule management much simpler and more secure."
Q: "How does an IDS differ from an IPS?"
A: "IDS monitors traffic passively and generates alerts when it detects malicious patterns — it doesn't block anything. IPS sits inline and can actively block traffic that matches malicious signatures or behavioral anomalies. IPS introduces risk: false positives can block legitimate traffic, so tuning is critical. Most deployments use IPS in alert-only mode initially, then switch to blocking mode after tuning."
Q: "What are signatures in Snort/Suricata and how do you write one?"
A: "Snort signatures are rules that define malicious traffic patterns. A rule has: action (alert/drop/pass), protocol (tcp/udp/icmp), source/destination IPs and ports, direction, and content match fields. For example: alert tcp any any -> any 80 (content:'/etc/passwd'; msg:'Directory traversal';). I write custom rules for specific CVEs, known attacker tooling (C2 patterns), or application-specific attack vectors."
