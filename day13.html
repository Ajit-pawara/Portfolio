Virtualization — building your home security lab

Every lab you've done on TryHackMe runs in someone else's environment. Today you build your own — a personal isolated network where you can break, attack, defend, and experiment freely with zero risk to anything real.

THEORY

What is virtualization? Virtualization lets you run a complete operating system inside a window on your existing computer — like running a full Linux machine inside Windows, or a vulnerable target inside Kali. The software that makes this possible is called a hypervisor.

Type 1 Hypervisor
Runs directly on hardware (no host OS). VMware ESXi, Hyper-V, Proxmox. Used in enterprise data centres and cloud providers.
Type 2 Hypervisor
Runs on top of your existing OS. VirtualBox, VMware Workstation. What you'll use for your home lab — free and easy to set up.
Snapshot
A saved state of a VM at a point in time. Break the VM during an exploit? Restore to snapshot in 10 seconds. Essential for malware analysis.
NAT vs Host-Only Network
NAT = VM shares host's internet connection. Host-Only = isolated network between VMs only, no internet — ideal for safe attack/defence practice.
Why a home lab matters for placement: TryHackMe gives you a browser-based environment. Your home lab gives you something TryHackMe can't — real customization, real tool installation, and real evidence that you built something yourself. "I set up and ran my own attack/defend lab" in an interview carries significant weight.

// real named incident
2018 — TESLA KUBERNETES CRYPTOJACKING
Attackers exploited an unsecured Kubernetes dashboard — inside a cloud VM environment
RedLock researchers discovered in 2018 that Tesla's cloud infrastructure (running on AWS) had a Kubernetes administrative console exposed to the internet with no password. Attackers accessed it, deployed their own container (a virtualized environment — essentially a lightweight VM), and ran cryptomining software inside it. Tesla's own workloads continued running normally around the malicious container, which is why it went undetected for a period. The attack demonstrates exactly why understanding virtualization and container isolation matters: the attackers didn't break into Tesla's servers — they added their own virtual workload alongside Tesla's legitimate ones. Misconfigured virtualization and container security is one of the fastest-growing categories of real-world cloud breaches.
// attacker vs defender mindset
⚔ Attacker
  • VMs are an attacker's best friend too — they run malware in isolated sandboxes to test it before deploying it against real targets.
  • VM escape vulnerabilities (breaking out of a VM to reach the host) are some of the highest-severity CVEs that exist — e.g., CVE-2018-3646 (L1TF) affecting Intel processors.
  • Cloud environments are essentially rented VMs — misconfigured isolation between tenants is a real attack surface on AWS, Azure, and GCP.
🛡 Defender
  • Snapshots are a security tool — malware analysts snapshot before detonating a sample, then restore to clean state afterward.
  • Network isolation is the most important lab setting — always use Host-Only for attack/target VMs, never Bridged (which exposes the VM to your real network).
  • In enterprise environments: VM segmentation and hypervisor hardening follow the same logic as network segmentation from Day 9 — one compromised VM should not reach others.
HANDS-ON LAB — BUILD YOUR HOME LAB TODAY

This is the most important practical day so far — you're building infrastructure you'll use for the rest of this entire 90-day roadmap. Set aside the full hour for this.

  1. Install VirtualBox — download free from virtualbox.org/wiki/Downloads. Install on your Windows/Mac/Linux host. Enable VT-x/AMD-V in BIOS if prompted (virtualization must be enabled in firmware settings).
  2. Download Kali Linux pre-built VM — go to kali.org/get-kali/#kali-virtual-machines and download the VirtualBox image (.ova file). Import it into VirtualBox via File → Import Appliance. Default credentials: kali / kali.
  3. Download Metasploitable 2 — search "Metasploitable 2 VulnHub" or go to vulnhub.com. Download the .vmdk and import into VirtualBox.
  4. Set both VMs to Host-Only networking — in VirtualBox Settings → Network → Adapter 1 → Host-Only Adapter. This creates an isolated network. Both VMs get IPs on the same subnet, internet is blocked from both.
  5. Boot Kali, find your IP and Metasploitable's IP — run ip addr on Kali, then nmap -sn [your subnet]/24 to discover Metasploitable's IP on the same network.
  6. Run your first scan against a legal target — Nmap against Metasploitable. This is the first real pentest workflow step.
  7. Take a snapshot of both VMs — Machine → Take Snapshot. Name it "Clean install." This is your restore point for when things break.
