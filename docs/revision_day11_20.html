Windows Internals for Security
NotPetya 2017 · Mimikatz
// core components
SAM
Security Account Manager — stores local password hashes at C:\Windows\System32\config\SAM. Extracting it requires SYSTEM hive (boot key).
LSASS
Local Security Authority Subsystem Service — manages authentication in memory. Mimikatz dumps credentials from LSASS process memory.
NTLM vs LM
LM is ancient — splits password into 2 halves of 7 chars each, making it trivial to crack. NTLM uses MD4, stronger but still vulnerable to pass-the-hash.
Registry
HKLM\SAM (hashes), HKLM\SYSTEM (boot key for decryption), HKCU\...\Run (persistence via startup programs).
// event log IDs to memorize
4624
Successful logon — first thing SOC analysts check after a breach.
4625
Failed logon — brute-force detection. Spike in 4625 = attack in progress.
4688
Process creation — tracks every new process. Key for detecting malware execution.
1102
Security log cleared — a classic post-exploitation cleanup step. Major red flag.
// real incident

NotPetya (2017): the world's costliest cyberattack (>$10B). Used EternalBlue (SMB exploit) to enter, then dumped LSASS credentials with Mimikatz, moved laterally via WMI and PsExec. Propagated like a worm, destroyed MBRs so systems couldn't boot even after paying. Lesson: Windows credential management is a single point of failure — LSASS in memory. Tools like Mimikatz prove that authentication secrets should never be in process memory.

// interview answer
Q: How would an attacker extract Windows password hashes post-compromise?
Two approaches: (1) Dump LSASS process memory with Mimikatz.exe sekurlsa::logonpasswords while running as admin/SYSTEM. (2) Save registry hives: reg save HKLM\SAM sam.save and reg save HKLM\SYSTEM system.save, then extract hashes offline with tools like impacket-secretsdump. Method 2 requires both hives because SYSTEM hive contains the boot key used to decrypt SAM.

// interview answer
Q: When would you use Nmap's -sS vs -sT vs -sU?
-sS (SYN stealth): sends SYN, never completes handshake. Faster, less logged. Default as root. -sT (TCP connect): completes full handshake. Only option as non-root. -sU (UDP): slow, unreliable — essential for DNS/SNMP/UDP services but ports often don't respond even when open.

Cryptography I — Hashing vs Encryption
LinkedIn 2012 · SHA-1 Collision 2017
// hashing vs encryption — the fundamental difference
Hashing (One-Way)
Irreversible, deterministic, fixed-size output. echo -n "hello" | sha256sum. Use for: integrity verification, password storage.
Encryption (Two-Way)
Reversible with a key. openssl enc -aes-256-cbc -in plain.txt -out enc.txt. Use for: confidentiality, secure communication.
// hash algorithms — know the progression
MD5 — 128-bit
Broken. Collisions found in 2004. Never use for security.
SHA-1 — 160-bit
Deprecated. SHAttered (2017) proved collision attack. Git still uses it but migrating.
SHA-256 — 256-bit
Current standard. Used in TLS certificates, blockchain, password hashing (with salt).
bcrypt/scrypt/Argon2
Password hashing algorithms. Deliberately slow + memory-hard. Argon2 won the 2015 PHC.
// real incident

LinkedIn Password Breach (2012): 167M password hashes leaked. LinkedIn used unsalted SHA-1 — fast enough for GPUs to compute billions of hashes per second. 90% of hashes cracked within days using rainbow tables. Salting alone would have made cracking 1000x slower per user. Lesson: never hash passwords with a fast algorithm, and never without salt.

// interview answer
Q: Why is bcrypt preferred over SHA-256 for password storage?
SHA-256 is designed to be fast — GPUs compute billions of SHA-256 hashes per second. bcrypt has a configurable cost factor that makes it deliberately slow (e.g., 250ms per hash). An attacker with a GPU cluster that cracks 10B SHA-256/s might only crack 4 bcrypt/s. Defense-in-depth for passwords = slow + salted + memory-hard (Argon2).

Cryptography II — Symmetric vs Asymmetric
Heartbleed CVE-2014-0160
// comparison
Symmetric (AES)
One key encrypts AND decrypts. Fast (hardware-accelerated). Problem: how do you securely share the key? Used for bulk data encryption.
Asymmetric (RSA/ECC)
Public key encrypts, private key decrypts. Slow (100-1000x slower than AES). Solves key exchange. Used for handshakes and signatures.
TLS Handshake
ClientHello → ServerHello + Certificate → Key Exchange (RSA or ECDHE) → Symmetric session key established → Encrypted channel with AES.
Perfect Forward Secrecy
Ephemeral Diffie-Hellman (DHE/ECDHE) generates a unique session key each time. Compromising the private key later can't decrypt past sessions. RSA key exchange does NOT provide PFS.
// when to use each
Symmetric (AES-256)
Encrypting files, database columns, disk encryption (BitLocker, LUKS). Fast, bulk data.
Asymmetric (RSA-4096 / ECC-256)
TLS handshakes, SSH key pairs, code signing, email encryption (PGP). Slow, key exchange.
Both Together (TLS)
TLS uses RSA/ECDHE to exchange an AES key, then AES encrypts the session. Security (asymmetric) + performance (symmetric).
// real incident

Heartbleed (CVE-2014-0160): a buffer over-read in OpenSSL's Heartbeat extension leaked up to 64KB of server memory per request. Attackers repeatedly read memory until they extracted private RSA keys. 500K+ servers affected. No patch for 2 years before discovery. Lesson: your private keys are only as safe as the software that manages them — and OpenSSL is used everywhere.

// interview answer
Q: Why does TLS use both asymmetric and symmetric cryptography?
Asymmetric (RSA/ECDHE) is slow but solves the key exchange problem — it securely negotiates a shared secret. Symmetric (AES) is fast (hardware-accelerated) for bulk encryption. TLS uses asymmetric during the handshake to exchange a symmetric session key, then AES encrypts all data. This hybrid = security + performance.

Cryptography III — SSL/TLS, Certificates & PKI
DigiNotar CA Breach 2011
// x.509 certificate anatomy
Subject
Who the certificate belongs to — domain name (CN/SAN). Browser checks this matches the URL.
Public Key
The asymmetric key — anyone can encrypt data that only the private key holder can decrypt.
Issuer
The CA that signed this certificate. Browser checks if this CA is in its trusted root store.
Validity
Not Before / Not After. Expired certificates are rejected by modern browsers.
Signature
CA's cryptographic signature proving the certificate hasn't been tampered with.
// certificate chain hierarchy
Root CA
Self-signed, pre-installed in browsers (~150 globally). If compromised, ALL certs signed by it are untrusted. DigiNotar 2011.
Intermediate CA
Signed by root CA. Issues actual server/leaf certificates. Limits blast radius — only intermediates affected if breached.
Server/Leaf Cert
The certificate your browser sees. Contains domain name (CN/SAN), public key, and signature chain back to a trusted root.
// self-signed CA in lab
openssl req -x509 -newkey rsa:4096 -keyout ca.key -out ca.crt -days 365 -nodes
Create root CA with 4096-bit RSA key. -nodes means no passphrase.
openssl req -newkey rsa:2048 -keyout server.key -out server.csr -nodes
openssl x509 -req -in server.csr -CA ca.crt -CAkey ca.key -out server.crt -days 365
Generate a server CSR then sign it with your CA. Two commands: create cert request, then sign.
// real incident

DigiNotar CA Breach (2011): attackers compromised a Dutch CA and issued 531 fraudulent certificates — including *.google.com, *.torproject.org, and *.cia.gov. Iranian users were MITM'd for months: their connections to Google were intercepted and decrypted using the fake certificate. Lesson: the entire PKI trust model is only as strong as its weakest CA. DigiNotar was ultimately bankrupted and delisted from all major browsers.

// interview answer
Q: What happens in your browser when you visit an HTTPS site with an expired certificate?
The browser checks the certificate's validity period (Not Before / Not After). If expired, the certificate is rejected and the browser shows a warning page saying the connection is not secure — even though the encryption itself may be perfectly sound. The user must manually bypass the warning to proceed. This is why certificate expiration is one of the most common causes of production outages.

Authentication vs Authorization
Capital One 2019 · IDOR
// the fundamental difference
Authentication (AuthN)
"Who are you?" — verifying identity. Passwords, biometrics, MFA, certificates, SSO.
Authorization (AuthZ)
"What are you allowed to do?" — permissions, roles, ACLs, policies. Usually checked AFTER authentication.
// common auth failures
IDOR
Insecure Direct Object Reference. User changes /profile?id=123 to /profile?id=124 and sees another user's data. Authentication is fine — authorization was never checked.
Session Hijacking
Attacker steals your session cookie (via XSS, network sniffing) and the server treats them as you. AuthN already happened — the cookie is the proof.
Broken Access Control
OWASP #1. Occurs when an app checks "are you logged in?" (AuthN) but doesn't check "are you ALLOWED to do this?" (AuthZ).
// real incident

Capital One Data Breach (2019): a former AWS employee exploited a misconfigured WAF that allowed SSRF to the EC2 metadata service. The WAF's IAM role had access to S3 buckets containing 100M+ credit applications. Authentication was fine (the WAF was properly authenticated to AWS). Authorization was broken (the WAF's role had far more permissions than needed). Lesson: least privilege principle — a WAF should only have permissions to forward traffic, not read customer data.

// interview answer
Q: What's the difference between horizontal and vertical privilege escalation?
Horizontal: accessing another user's data at the same privilege level (user A reads user B's email by changing an ID parameter — IDOR). Vertical: gaining higher privileges than intended (standard user becomes admin by manipulating a role field in a JWT token). Both are authorization failures, but vertical is usually more dangerous because it unlocks administrative functions.

Web Recon — Directory Busting & Technology Fingerprinting
British Airways Magecart 2018
// core recon techniques
Directory Busting
Brute-force hidden paths — admin panels, backups, config files, dev endpoints. Attackers discover 10x more surface area than visible on the homepage.
Technology Fingerprinting
Identify the server, framework, CMS, CDN. Each technology has known CVEs. Apache vs Nginx, PHP vs ASP.NET, WordPress vs custom — different attack paths.
// tools
gobuster dir -u https://target.com -w common.txt -x php,txt,zip
Directory enumeration. Recursive with -r. Extensions filter by file type.
ffuf -u https://target.com/FUZZ -w wordlist.txt
Faster than gobuster. Use -ac for auto-calibration of response size filtering.
whatweb https://target.com
Passive fingerprinting — no requests to the target. Reports server, JS libs, CMS, meta tags.
Wappalyzer (browser ext)
Identifies tech stack passively. JS frameworks, CDN, analytics, CMS version. Runs as you browse.
// common discovery targets
Admin Panels
/admin, /wp-admin, /administrator — if no auth, immediate compromise.
Sensitive Files
/.git, /.env, /backup, /config — source code and credentials exposed.
API Endpoints
/api, /swagger, /graphql — direct access to backend functionality.
Info Leakage
/robots.txt, /sitemap.xml, /server-status, /phpinfo.php — unintentional reconnaissance.
// real incident

British Airways Magecart (2018): attackers discovered an unsecured staging admin path on ba.com using directory busting tools. They injected a 23-line JavaScript skimmer into the payment page that captured 380,000 payment card details over 2 weeks. BA was fined £20M under GDPR. Lesson: hidden dev/staging paths are a common finding in directory busting — Magecart operators automate directory scanning across thousands of domains daily.

// interview answer
Q: What's the first three things you check when you find a /.git directory exposed on a website?
(1) Download .git/config for remote origin URLs and credentials. (2) Use git-dumper or similar to reconstruct the entire repository — source code, database passwords, API keys. (3) Check git log for commit history revealing sensitive data that was pushed then "removed" (still in history). Automated scanners check for /.git first because it's one of the highest-impact findings in directory busting.

// Quick Self-Quiz — Can You Answer These Without Looking?

Cover the answers. Say them out loud. If you can't answer one fluently, revisit that day before continuing.

Q01
What two registry hives do you need to extract Windows password hashes — and why both?
SAM (contains encrypted hashes) + SYSTEM (contains the boot key to decrypt them). Without SYSTEM, the SAM data is useless ciphertext.
Q02
EternalBlue + Mimikatz + WMI — name the attack and explain how each component worked.
NotPetya 2017. EternalBlue exploited SMBv1 for initial entry. Mimikatz dumped LSASS credentials from memory. WMI moved laterally to new machines. Combined: worm-like propagation destroying MBRs across networks.
Q03
When would you use Nmap -sS vs -sT vs -sU? Which is stealthiest?
-sS (SYN stealth) = fastest, least logged, needs root. -sT (TCP connect) = completes handshake, logged, no root needed. -sU (UDP) = slow, unreliable. -sS is stealthiest.
Q04
Why is bcrypt better than SHA-256 for storing passwords?
SHA-256 is nanoseconds-fast — GPUs brute-force billions/sec. bcrypt has a configurable cost factor making each hash take ~250ms. A cluster that cracks 10B SHA-256/s cracks only 4 bcrypt/s. Always use slow, memory-hard algorithms for passwords.
Q05
Why does the TLS handshake use BOTH asymmetric and symmetric cryptography?
Asymmetric (RSA/ECDHE) securely exchanges a symmetric session key. Symmetric (AES) encrypts all subsequent data for performance. Asymmetric alone is too slow for bulk data. Symmetric alone has no way to securely share the key.
Q06
What was the real vulnerability in the Capital One breach — authentication or authorization? Why?
Authorization. The WAF was properly authenticated to AWS (AuthN was fine). But the WAF's IAM role had read access to S3 buckets containing 100M+ customer records — far more permission than needed (AuthZ was broken).
Q07
The DigiNotar breach broke which element of the PKI trust model?
The root CA trust. Attackers compromised the CA itself and issued valid-looking certificates for any domain. Browsers trust ~150 root CAs — any one of them can sign a certificate for google.com. One weak CA breaks security for every domain.
Q08
What did Heartbleed actually leak from server memory — and what was the worst-case impact?
Up to 64KB of arbitrary server memory per request, including private RSA keys. Worst case: attacker decrypts previously captured TLS traffic (if no PFS) and can impersonate the server going forward.
Q09
How did the British Airways Magecart attackers find the entry point they exploited?
Directory busting — they found an unsecured staging admin path on ba.com using automated scanning tools like gobuster/ffuf. The hidden path had no authentication and allowed JavaScript file injection.
Q10
How did Lazarus Group use Wireshark in the Bangladesh Bank heist?
They installed Wireshark on an internal machine and captured SWIFT network traffic for weeks. By analyzing the unencrypted protocol format and authentication flow, they learned exactly how to craft fraudulent transfer requests that the system accepted.
