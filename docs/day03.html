Linux Fundamentals I — filesystem & permissions

Every pentest tool, every SOC server, every CTF box runs on Linux. Today you stop "knowing some commands" and start understanding the system the way attackers and defenders both have to.

THEORY

Linux uses one unified filesystem tree starting at /. A handful of directories matter more than the rest, security-wise:

/ (root) ├── /etc/ → config files. /etc/passwd & /etc/shadow live here ├── /home/ → user dirs — attackers look for SSH keys, .bash_history here ├── /var/log/ → system logs — where defenders spend a huge amount of time ├── /tmp/ → world-writable — attackers love dropping payloads here ├── /root/ → superuser's home — getting here = full compromise └── /bin /usr/bin → where executable programs live

Permissions are the actual access-control mechanism. ls -l shows strings like rwxr-xr-- — not decoration, the literal lock and key.

r
read (4)
w
write (2)
x
execute (1)
owner
user
group
group
other
everyone else

rwxr-xr-- = owner rwx, group r-x, others r-- → numerically 754. Memorize the number system, not just the letters — chmod 754 file shows up constantly in labs.

// real, named incident — not hypothetical
2021 — PERMISSION/SUID MISCONFIGURATION
PwnKit (CVE-2021-4034)
Qualys researchers found a memory-corruption flaw in Polkit's pkexec — a SUID-root binary present on nearly every major Linux distro by default. It had been exploitable for over 12 years before discovery, letting any local user escalate straight to root with a few lines of code. It's the cleanest real-world proof that SUID + permission misconfigurations aren't theoretical — they sat unnoticed in production systems worldwide for over a decade.
// attacker mindset vs. defender mindset
⚔ How an attacker thinks
  • "Once I have any shell, what SUID binaries or writable configs can I abuse to become root?"
  • World-writable files + root-owned execution = a near-guaranteed path to full compromise.
  • find / -perm -4000 is one of the first commands run after landing on a box.
🛡 How a defender thinks
  • "What SUID binaries exist on this system, and do they actually need that bit?"
  • Audit permissions regularly — PwnKit existed for 12+ years because nobody was looking.
  • Never use chmod 777 as a quick fix — it's the most common self-inflicted wound in real incident reports.

INTERVIEW READY
Q: "Explain Linux permissions and why misconfigurations matter."
A: "Every file has three permission sets — owner, group, others — built from read(4)/write(2)/execute(1). Misconfigured permissions are a real, proven attack vector: PwnKit, CVE-2021-4034, was a flaw in a default SUID-root binary that sat undiscovered for over 12 years across nearly every major Linux distro, letting any local user escalate to root. Reading a permission string correctly, and auditing SUID bits, is a baseline skill — not optional."
ENTREPRENEUR ANGLE

A "Linux permissions & SUID audit" is one of the fastest freelance services you can offer — most cheap VPS/web hosting runs Linux. Run find / -perm -4000 and find / -writable -type d, document anything risky, hand over a one-page fix list. Reference PwnKit by name in your pitch — "I check for the same class of misconfiguration that caused CVE-2021-4034" instantly sounds more credible than generic "I do security audits." A 30–60 min job, priced ₹3,000–₹8,000 — and it's a direct building block for your future firm's "basic security hygiene" package.

TEACH IT BACK

Explain to a non-technical person: "Every file on a computer has a lock with three keys — one for you, one for your team, one for everyone else. Each key can open the file to read, to rewrite, or to run it. A famous bug called PwnKit happened because one of those locks was left open by mistake on millions of computers for over a decade — and once someone noticed, they could turn themselves into the computer's 'owner' instantly." If that lands, you've actually understood it.
