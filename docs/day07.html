Networking II — TCP/IP & Ports

Yesterday you learned the 7-layer map. Today we slow right down and actually understand the two things that map is built on top of — TCP/IP, and ports — explained in plain language, step by step, so it actually sticks.

THEORY

Let's start with the simplest version of the idea. Every device on the internet needs two things to talk to another device: an address (so the message knows where to go) and a door number at that address (so the message knows which specific program inside the device should receive it). That's really all IP addresses and ports are.

IP address = the address of the building. Every device connected to a network gets one, like 192.168.1.42. It's how data finds the right computer out of millions.

Port = the specific door/apartment number at that address. A single computer might be running a web server, an email server, and an SSH server all at once — ports are how incoming data knows which one of those programs to go to. Ports are simply numbers, 0 to 65535.

Think of it like a big apartment building. The IP address gets your letter to the right building. The port number is the apartment number that gets it to the right door inside that building. Without the apartment number, the mail carrier would just dump every letter in the lobby and nobody would know whose mail is whose — that's basically what happens on a computer without ports.

Now, how does TCP actually work? TCP (Transmission Control Protocol) is the most common way data is reliably sent across a network. Before any real data is sent, the two computers do a quick "handshake" to agree they're both ready — this is called the three-way handshake, and it happens every single time you load a webpage, with zero of it visible to you.

1. SYN →
Your computer says: "Hi, I'd like to start a conversation. Here's a starting number to keep track of our messages."
2. SYN-ACK ←
The server replies: "Got it, I'm ready too. Here's MY starting number, and confirmation I received yours."
3. ACK →
Your computer replies: "Perfect, confirmed. Let's start actually sending data now."

Why does this matter for security? Because that handshake requires the server to set aside memory and wait for step 3 to arrive. If an attacker sends thousands of step-1 messages (SYN) and never sends step 3, the server sits there holding open all those half-finished conversations until it runs out of resources and crashes. This exact technique is called a SYN flood — and it's one of the oldest, simplest denial-of-service attacks that still works today if a server isn't protected against it.

A few ports you'll see constantly — memorize these, they come up in almost every lab and every interview:

21
FTP
File transfer — often unencrypted, a common target
22
SSH
Secure remote login — encrypted terminal access
80
HTTP
Regular, unencrypted web traffic
443
HTTPS
Encrypted web traffic — what your browser shows as the padlock icon
3389
RDP
Windows Remote Desktop — a very common attacker target if exposed to the internet
// real, named incident — not hypothetical
FEBRUARY 2000 — "MAFIABOY" SYN FLOOD ATTACKS
A 15-year-old took down Yahoo, eBay, CNN, and Amazon
Using exactly the SYN flood technique explained above, a Canadian teenager known online as "MafiaBoy" launched distributed denial-of-service attacks that knocked Yahoo (then the web's top search engine), CNN, eBay, Amazon, and Dell offline for hours, causing an estimated $1.2 billion in damages globally. He was caught after bragging about it in IRC chatrooms. The technical lesson hasn't changed in 25 years: the three-way handshake, if left unprotected, can be weaponized by flooding step 1 and never completing step 3.
// attacker mindset vs. defender mindset
⚔ How an attacker thinks
  • "Which ports are actually open and listening on this target?" — this is the very first question in almost every real attack, found using a port scanner.
  • An open port is an opportunity — each one represents a running service, and every running service has its own potential vulnerabilities.
  • Flooding the handshake itself (SYN flood) can take a target offline without ever needing a "real" vulnerability at all.
🛡 How a defender thinks
  • "Does this server actually need this port open to the internet, or can I close/restrict it?" — every unnecessary open port is unnecessary risk.
  • Modern firewalls and servers use "SYN cookies" specifically to survive SYN flood attempts without exhausting memory.
  • Regularly scanning your own systems (the same way an attacker would) is the only reliable way to know what's actually exposed.
HANDS-ON LAB

Today's lab uses Nmap — the most widely used port-scanning tool in the entire security industry — for the very first time, in the gentlest possible way.

  1. Go to tryhackme.com/room/nmap02 ("Nmap Basic Port Scans") and deploy the target machine.
  2. Run a basic TCP connect scan against the target and read the output carefully — you'll see exactly the open/closed/filtered states explained above, for real.
  3. Find which ports are open on the target, and try to guess what each open port is used for before checking the room's hints — this guessing habit is what builds real intuition.
  4. Run the terminal drill below on your own machine to see your own open ports.
Deliverable: a screenshot of your Nmap scan results from the room, with the open ports you found and what service you believe each one runs.
# see which ports are open and listening on YOUR OWN machine first ss -tuln Netid State Local Address:Port tcp LISTEN 0.0.0.0:22 tcp LISTEN 127.0.0.1:631 # port 22 = SSH listening on all interfaces, port 631 = local printing service # now scan a target machine in the THM lab (only do this on systems you're authorized for) nmap -sT [target-ip] PORT STATE SERVICE 22/tcp open ssh 80/tcp open http # -sT performs a full TCP connect scan -- it completes the entire 3-way handshake # for every port, which is why it's the gentlest, most "complete" scan to start with
INTERVIEW READY
Q: "Explain the TCP three-way handshake and why it matters for security."
A: "TCP establishes a reliable connection using three steps — SYN, SYN-ACK, ACK — where both sides confirm they're ready before real data flows. It matters for security because the server has to hold resources open after step one, waiting for step three. If that's exploited — sending thousands of SYN requests and never finishing the handshake — you get a SYN flood, exactly what 'MafiaBoy' used in 2000 to take down Yahoo, eBay, and CNN. Modern defenses like SYN cookies exist specifically because of attacks like that one."
ENTREPRENEUR ANGLE

A basic "open port exposure check" is one of the simplest, fastest things you can offer a small business with their own server or website: scan their public IP, show them exactly which ports are open to the world, and flag anything unnecessary (like RDP port 3389 exposed directly to the internet, a very common real-world mistake). Explaining it in plain terms — "your front door (port 22) is locked properly, but you also have a side window (port 3389) wide open" — makes the risk land immediately for a non-technical business owner, and it's a natural opener into a bigger security engagement.

TEACH IT BACK

Explain to a non-technical person: "Every website you visit, your computer first 'knocks' on the door (SYN), the website says 'come on in, I'm ready' (SYN-ACK), and your computer says 'great, here I come' (ACK) — only then does the actual page load. In 2000, a teenager flooded major websites like Yahoo and eBay with thousands of fake knocks that never finished the conversation, until the websites had no doors left to answer with and crashed. That's still a real attack today, just better defended against now."
