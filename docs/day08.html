Networking III — DNS, HTTP/HTTPS & DHCP

These three protocols run silently behind literally every internet action you take — every website you load, every IP address your device gets assigned. Today you understand what's actually happening, step by step, so you can spot when it's being abused.

THEORY

Part 1: DNS — the internet's phone book.

Every website has an IP address (like 104.26.10.229). Nobody memorizes those. DNS (Domain Name System) is the system that takes a human-readable name like google.com and translates it into the IP address your computer actually needs to connect. Think of it exactly like a phone book: you look up "Google," it gives you the number.

Here's what actually happens when you type tryhackme.com into your browser — the steps you never see:

1
Your computer checks its own local cache first — did I already look this up recently? If yes, skip everything below and go directly to the website.
2
If not cached, it asks your ISP's Recursive DNS Resolver — your internet provider's dedicated "lookup helper."
3
The resolver asks a Root Name Server — "where can I find who manages .com domains?" Root servers don't know the IP, they just point to the next step.
4
The resolver asks the TLD (Top Level Domain) Server for .com — "who manages tryhackme.com specifically?" It points to TryHackMe's own nameserver.
5
Finally, the resolver asks the Authoritative Nameserver — TryHackMe's own server — and gets the actual IP address.
6
That IP is returned to your browser, cached locally for next time, and the connection begins.
Why this matters for security: an attacker who can intercept or forge step 2–5 above can redirect you to a fake website that looks identical to the real one — this is called DNS Spoofing / Cache Poisoning, and it happens in real attacks.

Part 2: HTTP vs HTTPS — why the padlock icon actually matters.

HTTP (HyperText Transfer Protocol) is the language web browsers and servers use to talk to each other — "give me this page," "here it is." HTTPS is the same thing, but with a TLS encryption layer wrapping every message so no one in the middle can read it.

⚠ HTTP (port 80)
  • Every request and response travels in plain text.
  • Anyone on the same network (e.g. public WiFi) can read your passwords, session cookies, everything.
  • No verification that you're actually talking to the real website.
✓ HTTPS (port 443)
  • Everything is encrypted end-to-end with TLS.
  • Even if someone intercepts the traffic, they just see encrypted garbage.
  • The certificate verifies you're talking to the real server, not an impostor.

Part 3: DHCP — how your device gets an IP address automatically.

Every time you connect to a network (WiFi, ethernet) your device needs an IP address. You don't type one in manually — DHCP (Dynamic Host Configuration Protocol) handles this automatically. Your device broadcasts "I just joined, can anyone give me an IP?" A DHCP server on the network responds with an IP address, a subnet mask, a default gateway, and DNS server addresses — your device uses all four to function on the network. Simple, invisible, and essential.

Security angle: an attacker on your local network can run a rogue DHCP server that responds faster than the legitimate one, handing out a "poisoned" DNS address — from that moment, every domain lookup on your machine goes through the attacker's server first. All your "DNS security" is now irrelevant.
// real, named incident — not hypothetical
2010 — DNS CACHE POISONING
Baidu DNS Hijack — China's top search engine redirected
On January 12, 2010, Baidu.com — China's largest search engine with hundreds of millions of users — was hijacked via a DNS attack. Visitors to baidu.com were silently redirected to a page controlled by the "Iranian Cyber Army." The attack exploited DNS infrastructure at the registrar level, redirecting millions of legitimate DNS lookups to the wrong IP address for several hours. Nobody hacked Baidu's servers — they just poisoned the phone book entry.
// attacker mindset vs. defender mindset
⚔ How an attacker thinks
  • "If I control what DNS returns, I control where every user on this network goes — without touching their browser or their device."
  • HTTP (not HTTPS) is an open book on any shared network — a coffee shop, a college campus WiFi, anywhere.
  • A rogue DHCP server on a local network is one of the fastest ways to man-in-the-middle every device silently.
🛡 How a defender thinks
  • "Is DNS traffic on this network going somewhere unexpected?" — DNS monitoring is one of the highest-signal detection sources in a SOC.
  • Force HTTPS everywhere — encrypt the traffic so even a perfect DNS redirect lands on an obvious certificate mismatch warning.
  • Use DNSSEC and trusted resolvers (like 1.1.1.1 or 8.8.8.8) — harder to poison than your ISP's default.

INTERVIEW READY
Q: "What is DNS and how can it be abused?"
A: "DNS translates human-readable domain names into IP addresses through a chain of servers — recursive resolver, root servers, TLD servers, and the authoritative nameserver. It can be abused in a few ways: DNS cache poisoning injects false records so lookups return attacker-controlled IPs — Baidu's entire domain was redirected this way in 2010 through a registrar-level attack. On a local network, a rogue DHCP server can hand out a malicious DNS address to every new device that connects, silently man-in-the-middling all their traffic. DNS monitoring is actually one of the first things a SOC analyst looks at when investigating unusual behavior."

ENTREPRENEUR ANGLE

DNS misconfiguration is one of the most common issues in small business websites — wrong records, no DNSSEC, HTTP instead of HTTPS, email spoofable because of missing SPF/DKIM TXT records. A "DNS & web traffic security check" is a 30-minute audit you can eventually offer any business with a website: check their DNS records, verify HTTPS is enforced everywhere, confirm their DHCP server is the only one on the network, and flag any plain HTTP endpoints. Priced at ₹2,000–₹5,000 as a quick-win audit, it naturally leads into larger web application security engagements.
