Cryptography II — symmetric vs asymmetric encryption

Every HTTPS connection, every SSH login, every encrypted email uses both types of encryption working together. Today you understand exactly how and why — and how the RSA SecurID breach showed what happens when the key material itself gets stolen.

THEORY

The core problem encryption solves: two parties want to communicate privately over a public channel (the internet) where anyone could be listening. The challenge is: how do you agree on a secret key if the channel is already compromised?

Symmetric Encryption — one key
  • Same key encrypts AND decrypts.
  • Very fast — ideal for bulk data encryption.
  • Key sizes: 128–256 bits (AES-256 is current gold standard).
  • Problem: how do you securely share that one key with the other party if the channel is insecure?
  • Examples: AES (current standard), DES (broken, 56-bit), 3DES (deprecated).
  • Used for: encrypting files, disk encryption, VPN tunnels after setup.
Asymmetric Encryption — key pair
  • Two mathematically linked keys: public key (share freely) + private key (never share).
  • What one key encrypts, only the other can decrypt.
  • Slow — NOT used for bulk data.
  • Key sizes: 2048–4096 bits (RSA). Much larger than symmetric because it's doing harder math.
  • Examples: RSA, ECC (Elliptic Curve Cryptography).
  • Solves the key exchange problem — you can share your public key openly with no risk.
How they work together in real life (TLS/HTTPS): asymmetric is too slow for bulk data, but it perfectly solves the key exchange problem. So in practice, every secure connection does both — asymmetric to securely agree on a symmetric key, then symmetric to actually encrypt the data.

How HTTPS actually establishes a secure connection — the TLS handshake:

1
Server sends its public key (inside a TLS certificate) — anyone can receive this, including attackers. That's fine.
2
Your browser generates a random session key (symmetric) and encrypts it using the server's public key. Only the server's private key can decrypt this — so even if an attacker intercepts it, they see encrypted garbage.
3
Server decrypts it with its private key — now both sides have the same symmetric session key, established securely over an insecure channel.
4
All further communication is encrypted with that symmetric session key (AES-256 typically) — fast, efficient, secure.

RSA — why it works: RSA's security is based on the mathematical difficulty of factoring large prime numbers. Multiplying two large primes together is easy (computers do it in milliseconds). Factoring the result back into its two prime factors is computationally infeasible for large enough numbers. Your public key contains the product (N = p × q). Without p and q separately, you can't derive the private key.

AES-256
Current symmetric standard. Used in TLS, disk encryption (BitLocker, VeraCrypt), VPNs. Quantum-resistant at 256-bit. NSA-approved for TOP SECRET data.
RSA-2048/4096
Dominant asymmetric algorithm. Used in TLS certificates, SSH, digital signatures. Will be broken by sufficiently powerful quantum computers — replacement algorithms being standardized now.
ECC (Elliptic Curve)
Modern alternative to RSA. Smaller key size for equivalent security (256-bit ECC ≈ 3072-bit RSA). Faster, used in modern TLS and mobile apps.
Diffie-Hellman
A key exchange protocol — lets two parties agree on a shared secret over a public channel without ever transmitting the secret itself. Underlies modern TLS key exchange.
Quantum threat: sufficiently powerful quantum computers will break RSA and ECC by solving the prime factoring/discrete log problems efficiently. AES-256 is considered quantum-resistant. Post-quantum cryptography algorithms (CRYSTALS-Kyber, CRYSTALS-Dilithium) are being standardized by NIST now — worth knowing as a forward-looking security topic.
// real named incident
2011 — RSA SECURID BREACH — US DEFENSE CONTRACTORS COMPROMISED
Attackers stole the "seeds" behind RSA's asymmetric authentication tokens — then used them to breach Lockheed Martin
In March 2011, RSA Security (the company, not just the algorithm) disclosed that attackers had breached their internal systems and stolen information related to their SecurID two-factor authentication product. SecurID tokens generate time-based one-time passwords using secret "seed" values stored at RSA. When those seed values were stolen, the mathematical relationship between the token and the server was compromised — effectively nullifying the second factor for millions of hardware tokens deployed worldwide across US defense contractors, financial institutions, and government agencies. Two months later in May 2011, Lockheed Martin — one of the world's largest defense contractors — detected an intrusion attempt using the stolen RSA SecurID data. The attack was stopped but highlighted the systemic risk of a cryptographic "root of trust" being compromised: when the underlying key material is stolen, every system that trusts that material is simultaneously at risk. RSA later replaced 40 million tokens at a cost estimated at over $66 million. The lesson: in asymmetric systems, the private key or seed is the entire security — lose that, and all encryption built on top of it collapses.
// attacker vs defender mindset
⚔ Attacker
  • Breaking the math (cracking RSA) is infeasible at modern key sizes — attackers target the keys themselves, not the algorithm. RSA SecurID shows this clearly.
  • Man-in-the-middle attacks on TLS target the certificate validation step — if you can present a fraudulent certificate that the client accepts, you become the "server."
  • Weak Diffie-Hellman parameters (small groups) were exploited in the 2015 Logjam attack, downgrading connections to 512-bit DH which could be factored in hours.
🛡 Defender
  • Protect private keys like the most sensitive credential in your environment — they ARE the security. HSMs (Hardware Security Modules) store keys in tamper-resistant hardware.
  • Certificate pinning and HSTS prevent MitM attacks by making clients reject unexpected certificates even if they're technically valid.
  • Enforce strong cipher suites — disable legacy TLS 1.0/1.1, disable weak DH groups, mandate AES-256-GCM and ECDHE key exchange in TLS configurations.

INTERVIEW READY
Q: "Explain how HTTPS keeps your data secure, step by step."
A: "HTTPS uses both encryption types together. First, the server sends its public key inside a TLS certificate. Your browser generates a random symmetric session key, encrypts it with the server's public key, and sends it. Only the server can decrypt this with its private key — so even if an attacker intercepts it on the wire, they see garbage. Now both sides have the same symmetric key, established securely. All actual data is then encrypted with AES-256 using that key — fast and secure. This hybrid approach is necessary because RSA is mathematically too slow for bulk data, but it perfectly solves the key exchange problem. The RSA SecurID breach in 2011 showed what happens when the private key material itself gets stolen — 40 million tokens were immediately compromised, costing RSA over $66 million to replace."

TEACH IT BACK

Explain to a non-technical person: "Symmetric encryption is like a lockbox where both people have a copy of the same key — fast, but how do you safely hand over that key copy to someone new? Asymmetric encryption solves this with a padlock and key: you give everyone a padlock they can snap shut (the public key), but only you have the key to open it (private key). So when you visit a website securely, it first hands you an open padlock. You put a secret inside, snap the padlock, and send it back — only the website can open it. Inside is the key you'll both use for the rest of the conversation. The 2011 RSA breach is like someone stealing the factory that made all the padlocks — suddenly every padlock they ever made is potentially compromised."
