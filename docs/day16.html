Cryptography I — hashing vs encryption

Adobe stored 153 million passwords using encryption instead of hashing in 2013 — one of the most catastrophic cryptography mistakes in history. Today you learn exactly why that was wrong, and what the difference actually means in practice.

THEORY

The single most important distinction in cryptography for security work:

Hashing — one-way
  • Takes input of any size, produces a fixed-size output (the "hash" or "digest").
  • Cannot be reversed. There is no "unhash" function — ever.
  • Same input always produces the same hash. Different input produces completely different hash.
  • Used for: password storage, file integrity verification, digital signatures.
  • Correct use for passwords: you hash the password at registration, store the hash, then hash again at login and compare — the original password is never stored.
Encryption — two-way
  • Transforms data into unreadable ciphertext using a key.
  • Can be reversed — with the correct key, you can decrypt back to plaintext.
  • Used for: protecting data in transit (TLS), protecting files at rest, secure messaging.
  • Wrong use for passwords: if you encrypt passwords, you have to store the key somewhere — and if an attacker gets the key, every password is instantly decrypted. Adobe did exactly this in 2013.
The rule: passwords should ALWAYS be hashed, never encrypted. Encryption is for data you need to retrieve later. Passwords are verified, not retrieved — you never need the original text again after storing the hash.

Hashing algorithms — which ones matter and which are dead:

MD5
⚠ BROKEN — do not use
128-bit output. Collision attacks known since 2004. Still seen in legacy systems — flag it immediately if you do.
SHA-1
⚠ DEPRECATED
160-bit. Google's SHAttered attack (2017) produced the first real SHA-1 collision. LinkedIn used unsalted SHA-1 in 2012.
SHA-256
✓ CURRENT STANDARD
256-bit output, part of SHA-2 family. Used in TLS certificates, Git commits, Bitcoin. Solid for integrity checks.
bcrypt
✓ BEST FOR PASSWORDS
Intentionally slow — designed for password storage. Has a work factor you can increase as hardware gets faster. Industry standard.
SHA-3
✓ MODERN ALTERNATIVE
Completely different design from SHA-2 — useful if SHA-2 weaknesses are ever found. Government/high-security contexts.
Argon2
✓ MODERN PASSWORDS
Winner of the Password Hashing Competition (2015). Memory-hard — resists GPU cracking. Recommended for new applications.

Salting — the extra step that makes hashing actually secure for passwords:

If two users have the same password, their hashes are identical — an attacker with a precomputed table of hashes (called a rainbow table) can reverse them instantly. A salt is a random value added to each password before hashing, so even identical passwords produce different hashes. bcrypt and Argon2 handle salting automatically — this is part of why they're the correct choice.

Encoding is not encryption and not hashing. Base64 is encoding — it just represents data in a different format. Anyone can decode it instantly with no key. You'll see Base64 constantly in CTFs — it's not a security mechanism, it's a representation format.
// real named incident
2013 — ADOBE SYSTEMS BREACH — 153 MILLION ACCOUNTS
Adobe encrypted passwords instead of hashing them — and lost everyone's passwords instantly when the key was found
Adobe suffered a breach of 153 million user records in 2013. The catastrophic mistake wasn't just that the data was stolen — it was that Adobe had stored all passwords using 3DES symmetric encryption rather than a proper hashing algorithm. This meant: (1) they had a decryption key stored somewhere, (2) when attackers got both the encrypted passwords AND the key, all 153 million passwords were instantly readable. Worse — because they used the same encryption key for everyone, users with the same password had identical encrypted values. Researchers cross-referenced the encrypted values with password hints (also stored in plaintext) and cracked millions of passwords without even needing the key. The correct approach was bcrypt with unique salts per user — a fundamental cryptography decision that would have made the breach far less damaging. The LinkedIn 2012 breach (from Day 2) was the same problem differently — 117M SHA-1 hashes with no salting, instantly cracked via rainbow tables.
// attacker vs defender mindset
⚔ Attacker
  • When you dump a password database, the first question is "how are these stored?" — MD5/SHA-1 unsalted = rainbow table lookup in seconds. bcrypt = likely months per password with GPU cracking.
  • Rainbow tables pre-compute millions of hash→password pairs. Against unsalted MD5/SHA-1, cracking a common password is a dictionary lookup, not brute force.
  • Hashcat is the standard tool — runs on GPU, can attempt billions of hashes per second against weak algorithms. Against bcrypt, the same GPU is limited to thousands per second.
🛡 Defender
  • Use bcrypt or Argon2 for all password storage — never MD5, SHA-1, or encryption.
  • Never store the original password — only the hash. If you ever need to "recover" a password for a user, that means you're doing it wrong (reset it instead).
  • File integrity monitoring: hash critical system files (config, binaries) with SHA-256 and alert if the hash changes — this detects tampering even if the file size and timestamp are spoofed.

INTERVIEW READY
Q: "What's the difference between hashing and encryption, and which should be used for password storage?"
A: "Hashing is one-way — you can't reverse it. Encryption is two-way — you can decrypt with the key. Passwords should always be hashed, never encrypted, because you never need to retrieve the original — you just hash the login attempt and compare. Adobe's 2013 breach is the textbook counter-example: they encrypted 153 million passwords with 3DES. When attackers got the key, every password was immediately readable. The correct choice is bcrypt or Argon2 with unique salts per user — bcrypt is intentionally slow, which means GPU cracking attempts that crack MD5 in seconds take months against bcrypt."

Ask ChatGPT "which Hashcat mode targets bcrypt?" or "write a Hashcat command for SHA-256 with a wordlist" — cuts time spent on man page hunting during CTFs.
CrackStation
Free online rainbow table — paste an MD5 or SHA-1 hash and it searches its database instantly. Shows how fast weak hashes fall to precomputed tables.
Claude / ChatGPT
Paste any hash and ask "what algorithm is this likely?" — hash length and format identify the algorithm (MD5=32 hex chars, SHA-256=64 hex chars, bcrypt starts with $2b$).
have i been pwned (HIBP)
haveibeenpwned.com — checks if an email or password appears in known breached databases. Built on SHA-1 k-anonymity hashing — a real-world privacy-preserving use of hashing.

ENTREPRENEUR ANGLE

A "cryptography audit" of a client's web application — specifically checking how passwords are stored — is one of the fastest, highest-value security checks you can offer. If a client's app stores passwords in MD5 or plaintext (shockingly common in older Indian SMB web apps), that finding alone justifies your engagement fee. Adobe's $1.1M FTC settlement over this exact issue shows regulators take it seriously. Position it as: "I'll check that your app won't be the next Adobe" — and suddenly you're selling risk reduction, not just a technical service.

TEACH IT BACK

Explain to a non-technical person: "Hashing is like putting something through a meat grinder — you can't reverse it. Encryption is like a lockbox — with the right key you can open it and get the original back. Passwords should go through the meat grinder: when you log in, the site grinds your attempt and compares it to the stored grind. Adobe made the mistake of using a lockbox for 153 million passwords in 2013. When attackers found the key to that lockbox, every single password was instantly readable — all 153 million of them."
