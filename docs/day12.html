Windows Internals — Registry, Processes & Event Logs

The world runs on Windows. Most corporate targets are Windows machines. Today you learn where attackers hide, how they persist, and — most importantly — how defenders detect them using the logs Windows generates by default.

THEORY

Part 1: The Windows Registry — the database that controls everything.

The Registry is a hierarchical database that stores all configuration settings for Windows and installed applications. Think of it as the control panel behind the control panel. It has five main "hives" (root keys):

HKEY_LOCAL_MACHINE (HKLM)
Settings for the entire computer — all users, hardware, installed software. The most important hive for security.
HKEY_CURRENT_USER (HKCU)
Settings for the currently logged-in user only. Malware that runs as a standard user hides here.
HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run
THE persistence key — anything listed here runs automatically every time Windows starts. Attacker's favorite location.
HKLM\SYSTEM\CurrentControlSet\Services
Windows services — malware often installs itself as a service here for deeper persistence.
Why this matters: registry persistence is how malware survives a reboot. If you remove malware from disk but don't clean the registry Run key, it re-executes next boot. SolarWinds SUNBURST (today's incident) stored configuration data in the registry to avoid writing suspicious files to disk.

Part 2: Core Windows Processes — know the real ones to spot the fakes.

Just like TeamTNT renamed malware to "kworker" on Linux (Day 4), Windows malware commonly impersonates core system processes by name. Know the real ones:

lsass.exe
Local Security Authority — handles authentication, stores password hashes in memory. Primary target for Mimikatz credential dumping. Should have ONE instance, parent = wininit.exe.
svchost.exe
Service Host — runs Windows services in groups. Multiple instances are normal. Suspicious if running from anywhere except System32, or with no parent services.
explorer.exe
Windows Explorer (desktop/file manager). One instance per logged-in user. Malware injects into this for evasion because it's always trusted.
csrss.exe
Client Server Runtime — critical system process. If killed, Windows crashes. Malware spoofs it to avoid termination. Must run from System32.

Part 3: Windows Event Logs — the defender's source of truth.

Windows automatically logs thousands of events. The key is knowing which Event IDs matter for security. These are the ones every SOC analyst and interview panel expects you to know cold:

4624
Successful logon. Baseline of all activity. Watch for unusual logon types or off-hours.
4625
Failed logon. Multiple in a short window = brute force attempt. Most important alert to set up first.
4688
Process creation. Logs what command was run and by whom. Disabled by default — enabling this is critical for detection.
4648
Explicit credential use (RunAs). Lateral movement and privilege escalation trigger this constantly.
1102
Audit log was cleared. Immediate red flag. Attackers clear logs to hide activity. This event fires when they try.
4720
User account created. Watch for unexpected new accounts — a classic attacker persistence technique.
4697
Service installed. Malware installs as a service for persistence. This event fires when it happens.
4104
PowerShell script block logging. See exactly what PowerShell commands ran. Essential for detecting living-off-the-land attacks.
Important: Event ID 4688 (process creation) and 4104 (PowerShell logging) are disabled by default on most Windows installs. Enabling them via Group Policy is one of the most impactful, zero-cost security improvements any organization can make.
// real named incident
2020 — SOLARWINDS SUNBURST ATTACK
9 months undetected inside US government — Windows registry was the hiding spot
In late 2019, attackers (attributed to Russian SVR intelligence) compromised the build pipeline of SolarWinds Orion software and inserted a backdoor called SUNBURST. When the trojanized update was installed by ~18,000 organizations — including the US Treasury, Departments of Justice, State, Homeland Security, and multiple Fortune 500 companies — SUNBURST used the Windows Registry to store its configuration data and command-and-control server addresses, deliberately avoiding suspicious file writes to disk. It stayed dormant for up to 2 weeks after each installation before activating, and mimicked legitimate SolarWinds API calls to blend into normal traffic. It wasn't discovered until December 2020 — 9 months of undetected access to some of the most sensitive networks on earth. Proper process creation logging (Event ID 4688) and PowerShell logging (4104) in SIEM would have significantly raised the chance of earlier detection.
// attacker vs defender mindset
⚔ Attacker
  • Registry Run keys are the first persistence mechanism to add — one registry value = malware re-runs every reboot forever.
  • Dump lsass.exe memory with Mimikatz to extract plaintext passwords and NTLM hashes from all currently logged-in users.
  • Clear Event Log 1102 after actions — but a good defender knows Event ID 1102 itself fires when logs are cleared, giving them away.
  • Inject into explorer.exe or svchost.exe — trusted processes that are whitelisted by most AV and EDR tools.
🛡 Defender
  • Enable Event ID 4688 and 4104 immediately — they're free and catch the majority of living-off-the-land attacks.
  • Alert on Event ID 1102 — if someone clears logs, that IS the alert. SUNBURST-style attacks avoid this by not clearing logs at all.
  • Baseline your process tree — know what lsass.exe's parent should be, and alert if it spawns child processes (it never should).
  • Monitor registry Run keys for unexpected entries — autorun analysis is a core part of every malware investigation.
HANDS-ON LAB
  1. Go to tryhackme.com/room/windowsfundamentals2x0x — free room, Windows VM included. Covers Registry Editor, Computer Management, Event Viewer, UAC.
  2. Open Registry Editor (run: regedit) and navigate to the Run key: HKLM\SOFTWARE\Microsoft\Windows\CurrentVersion\Run — list every entry. This is your persistence audit.
  3. Open Event Viewer, go to Windows Logs → Security. Filter by Event ID 4624 (successful logons) — read 3 entries and note the logon type and account name.
  4. Run the PowerShell commands below to query Event Logs without the GUI — this is how SOC analysts actually do it at scale.
Deliverable: screenshot of your Registry Run key contents + one PowerShell Get-WinEvent output showing Event ID 4624. Screenshot both — this is your Windows lab proof.
# Run these in PowerShell (Run as Administrator) # 1. Check the persistence Run key — what auto-starts on this machine? Get-ItemProperty "HKLM:\SOFTWARE\Microsoft\Windows\CurrentVersion\Run" SecurityHealth : C:\WINDOWS\system32\SecurityHealthSysHost.exe OneDrive : "C:\Program Files\Microsoft OneDrive\OneDrive.exe" # any unknown entry here = immediate investigation priority # 2. Query last 5 successful logon events (ID 4624) Get-WinEvent -LogName Security | Where-Object {$_.Id -eq 4624} | Select-Object -First 5 TimeCreated : 7/9/2026 10:32:14 Id : 4624 Message : An account was successfully logged on... # 3. Check if the audit log was ever cleared (ID 1102 = attacker red flag) Get-WinEvent -LogName Security | Where-Object {$_.Id -eq 1102} (no output = logs have not been cleared = good sign) # if this returns results — investigate IMMEDIATELY # 4. List running processes and their parent — spot process spoofing Get-WmiObject Win32_Process | Select-Object Name, ProcessId, ParentProcessId | Sort-Object Name lsass.exe | PID: 712 | Parent: 504 (wininit.exe) <- correct svchost.exe| PID: 1024 | Parent: 680 (services.exe) <- correct # if lsass.exe has ANY parent other than wininit.exe — that is malware
INTERVIEW READY
Q: "What Windows Event IDs do you consider most important for security monitoring, and why?"
A: "The ones I'd enable and alert on first: 4625 for failed logons — multiple in a short window flags a brute force immediately. 4688 for process creation — disabled by default but catches most living-off-the-land attacks since everything an attacker does involves creating a process. 4104 for PowerShell script block logging — gives you the actual commands, not just that PowerShell ran. And 1102 for audit log cleared — that one is self-referential: even if attackers clear the logs, the act of clearing generates this ID. The SolarWinds SUNBURST attack went undetected for 9 months in part because many of these logs weren't being collected or correlated across the compromised organizations."
AI TOOLS
Microsoft Sentinel (AI)
Microsoft's cloud SIEM uses AI to correlate Windows Event IDs across thousands of machines — exactly what would have caught SUNBURST faster.
ChatGPT / Claude
Paste a suspicious PowerShell script from Event ID 4104 and ask "what does this do?" Instant malware triage without a sandbox.
Elastic Security (AI)
AI-powered anomaly detection on Windows event logs. Flags unusual process trees and registry modifications automatically.
Autoruns (Sysinternals)
Microsoft's free tool that shows every registry Run key, startup folder, service, and scheduled task in one GUI. Essential for persistence analysis.

ENTREPRENEUR ANGLE

A "Windows security baseline audit" is a natural extension of your growing audit package: check registry Run keys for unauthorized persistence, verify Event Log collection is active, confirm 4688 and 4104 are enabled, and review recent 4625 failed logon spikes. For a small business with 5-20 Windows machines, this takes 1-2 hours and produces a one-page risk report worth ₹5,000–₹15,000. Position it as "what SolarWinds's victims should have had in place" — that framing makes the risk concrete and the value of your service obvious.

TEACH IT BACK

Explain to a non-technical person: "Windows keeps a diary of everything that happens — every login, every program that runs, every file that changes. That diary is called Event Logs. The SolarWinds attack in 2020 hid inside US government networks for 9 months, partly because many organizations weren't reading their diary closely enough. The Windows Registry is like the 'startup settings' drawer — programs write their name there so they restart automatically whenever the computer boots. Malware does the same thing to survive restarts. A security audit checks both the diary and the startup drawer."
