DAY 19 OF 90 — PHASE 2: CORE SECURITY CONCEPTS

Authentication vs Authorization — and why confusing them is catastrophic

Optus Australia in 2022 exposed 11 million customer records through an API that required zero authentication. No login, no token — just call the endpoint and get data. Today you learn the difference between proving who you are and controlling what you're allowed to do — two completely separate problems that developers constantly conflate.

THEORY

Two completely separate questions:

Authentication (AuthN) — WHO are you?
  • The process of proving your identity to a system.
  • Factors: something you know (password), something you have (OTP token), something you are (biometric).
  • Fails: weak passwords, credential stuffing, session hijacking, brute force, phishing.
  • Result of success: a session token or access token that proves "this user is logged in."
  • Real-world: showing your ID at a hotel check-in — proves who you are.
Authorization (AuthZ) — WHAT can you do?
  • The process of determining what an authenticated identity is permitted to access or do.
  • Implemented via: roles (RBAC), permissions, access control lists (ACLs), policies.
  • Fails: IDOR, privilege escalation, missing server-side checks, trusting client-supplied role parameters.
  • Happens AFTER authentication — but must be checked on every single request, not just at login.
  • Real-world: your hotel key card works for your room but not the penthouse — same identity, different permissions.
The critical mistake developers make: they implement authentication (login page) and then assume "if someone is logged in, they're authorized." This is wrong. Authorization must be enforced independently on every resource, every API endpoint, every action — server-side. Every time. The Optus 2022 API had no authentication at all. The TryHackMe Broken Access Control room (from search results) shows a real lab where changing isadmin=false to isadmin=true in a URL grants admin access — because the server trusts the client.

Key concepts in authorization failures:

IDOR (Insecure Direct Object Reference)
Changing an ID parameter in a URL/request to access another user's data. Example: /profile?id=1234 → /profile?id=1235 shows someone else's profile. Pure AuthZ failure — the server doesn't verify you own that ID.
Vertical Privilege Escalation
A regular user accesses admin functionality. Example: changing roleid=1 to roleid=0 in a request to gain admin access. The TryHackMe Broken Access Control room is exactly this scenario.
Horizontal Privilege Escalation
A user accesses another user's data at the same privilege level. IDOR is the most common form. "I can see my orders at /orders/me — what if I change 'me' to another user ID?"
Session Token
After authentication, the server issues a session token (cookie or JWT). This token is how authorization decisions are made on subsequent requests — it represents the authenticated identity.
JWT (JSON Web Token)
A signed token containing claims (user ID, role, permissions). Vulnerable if the "alg: none" attack works (server accepts unsigned tokens) or if the secret key is weak and guessable.
RBAC (Role-Based Access Control)
Users are assigned roles (admin, user, moderator), roles have permissions, and permissions gate actions. Correct implementation requires server-side role checks — never trust client-supplied roles.

MFA — how it strengthens authentication: Multi-factor authentication adds a second (or third) layer to the AuthN process. Even if a password is stolen (phishing, credential stuffing), MFA requires a second factor the attacker doesn't have — typically a TOTP (time-based one-time password) from an authenticator app. SMS-based MFA is weaker — SIM swapping attacks can intercept it — but still significantly better than password-only.

Principle of least privilege: every user, service, and system should have only the minimum permissions needed to perform their function — nothing more. If an API key only needs to read data, it should not have write permissions. If a user only manages invoices, they should not have access to HR records. This is the foundation of correct authorization design.
// real named incident
2022 — OPTUS AUSTRALIA BREACH — 11 MILLION RECORDS — ZERO AUTHENTICATION REQUIRED
An API endpoint with no authentication exposed the personal data of nearly half of Australia's population
In September 2022, Optus — Australia's second-largest telecommunications company — suffered a breach exposing the personal data of approximately 11 million customers, including names, dates of birth, phone numbers, email addresses, home addresses, and identity document numbers (passport, driver's licence). The root cause was an exposed API endpoint that required no authentication whatsoever — no login, no API key, no session token. An attacker simply discovered the API endpoint, began making sequential requests incrementing a customer ID parameter (a classic IDOR pattern), and received full customer records in response. The attacker downloaded 11 million records before being noticed. The breach cost Optus an estimated AUD $140 million in remediation, led to new mandatory data breach notification laws in Australia, and forced Optus to offer free credit monitoring to millions of customers. Subsequent investigation also raised questions about whether the API even needed to be internet-facing at all — least privilege applied to API exposure would have placed it behind a VPN or IP whitelist. A complete failure of both Authentication (none implemented) and Authorization (no check on which records a caller could access even if auth had existed).
// attacker vs defender mindset
⚔ Attacker
  • "Does this API require authentication? Can I call it without a token?" — unauthenticated API endpoints are the fastest, highest-yield bug bounty findings.
  • "If I'm authenticated, can I access resources belonging to other users?" — IDOR testing: change numeric IDs, UUIDs, usernames in every URL and request body.
  • "Does the server actually enforce my role server-side, or does it trust what I send?" — modify roleid, isadmin, usertype parameters in requests and see what happens.
🛡 Defender
  • Enforce AuthN and AuthZ on every single API endpoint — no exceptions. The Optus API was presumably "internal" but was internet-exposed.
  • Never trust client-supplied authorization claims — always verify server-side. If the client sends isadmin=true, the server must check the database, not the parameter.
  • Apply principle of least privilege everywhere — if an API doesn't need to be internet-facing, it shouldn't be. If a role doesn't need write access, it shouldn't have it.

INTERVIEW READY
Q: "What's the difference between authentication and authorization, and can you give a real-world example of each failing?"
A: "Authentication is proving who you are; authorization is controlling what you're allowed to do — and they must both be enforced independently. The Optus 2022 breach is a clean authentication failure: their API required no login, no token — sequential GET requests with incrementing customer IDs returned full personal records for 11 million Australians. A pure IDOR case is the classic authorization failure: the user is authenticated (logged in) but the server doesn't verify on each request whether they own the resource they're accessing. Change /profile?id=1234 to /profile?id=1235 and you're reading someone else's data — because authentication passed at login but authorization wasn't checked per-resource. The fix for both: enforce authentication on every endpoint, then enforce authorization server-side on every resource access — never trust client-supplied role or ID parameters."

ENTREPRENEUR ANGLE

IDOR testing is one of the most impactful, fastest-to-demonstrate findings in a web application security assessment. You can often find an IDOR in a client's app within 30 minutes just by systematically changing ID parameters in every URL and request. Present it with a screenshot showing you accessed another user's private data — the client immediately understands the risk without technical explanation. This single finding can turn a ₹10,000 assessment into a ₹30,000–₹50,000 remediation and re-testing engagement. The Optus breach (AUD $140M in damages from one unauthenticated API) is your pitch: "I check for this class of vulnerability — the one that cost Optus $140 million."

TEACH IT BACK

Explain to a non-technical person: "Authentication is showing your ID at the hotel desk to prove who you are. Authorization is your room key only opening your room, not every room. Optus forgot to ask for ID at the front desk entirely in 2022 — their API had no login requirement, so anyone who walked up to the counter and asked for a room number got full details about any customer they asked about. IDOR is the flip side: you showed your ID (authentication worked), but your key card opens every room — the hotel never checked whether the room you're trying to enter is actually yours."
